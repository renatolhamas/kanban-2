## 1. Diagnóstico

### Problema Raiz

As 28 RLS policies de isolamento de tenant usam `user_metadata`, campo editável pelo usuário:

```sql
-- INSEGURO — user_metadata é editável via supabase.auth.updateUser()
tenant_id = (auth.jwt() -> 'user_metadata' ->> 'tenant_id')::uuid
```

### Dois problemas simultâneos

| # | Problema | Impacto |
|---|----------|---------|
| 1 | `user_metadata` é editável pelo usuário final | Bypass de tenant isolation |
| 2 | `tenant_id` não é inserido em `user_metadata` no registro | RLS ineficaz (claim vazia) |

### Por que ainda não causou incidente

O backend usa `SUPABASE_SERVICE_ROLE_KEY` que bypassa RLS. As queries diretas funcionam, mas sem isolamento real de banco.

### Por que isso precisa ser corrigido AGORA

Story 4.2 implementa **Supabase Realtime** para atualizações em tempo real. O Realtime **depende de RLS** para filtrar eventos por tenant. Sem correção, toda subscrição Realtime retornaria dados de TODOS os tenants.

## 2. Solução Proposta — 3 Componentes

### Visão Geral

```
[Componente 1] register/route.ts
  → ao criar usuário, setar app_metadata.tenant_id via admin API

[Componente 2] Custom Access Token Hook (PostgreSQL Function)
  → ao gerar JWT, injeta tenant_id de public.users em app_metadata
  → garante JWT sempre consistente com o banco

[Componente 3] Migration SQL — atualizar 28 policies
  → substituir user_metadata por app_metadata
  → adicionar (SELECT ...) para performance
```

## 4. Componente 2 — Custom Access Token Hook

**Arquivo:** `supabase/migrations/20260414_create_custom_access_token_hook.sql`

```sql
CREATE OR REPLACE FUNCTION public.custom_access_token_hook(event jsonb)
RETURNS jsonb
LANGUAGE plpgsql
STABLE
SECURITY DEFINER
SET search_path = public, auth
AS $$
DECLARE
  claims jsonb;
  user_tenant_id uuid;
  user_role text;
BEGIN
  claims := event -> 'claims';

  SELECT u.tenant_id, u.role
  INTO user_tenant_id, user_role
  FROM public.users u
  WHERE u.id = (event ->> 'user_id')::uuid;

  IF NOT FOUND THEN
    RETURN event;
  END IF;

  claims := jsonb_set(
    claims,
    '{app_metadata}',
    COALESCE(claims -> 'app_metadata', '{}'::jsonb) ||
      jsonb_build_object(
        'tenant_id', user_tenant_id,
        'role', user_role
      )
  );

  RETURN jsonb_set(event, '{claims}', claims);
END;
$$;

GRANT EXECUTE ON FUNCTION public.custom_access_token_hook TO supabase_auth_admin;
REVOKE EXECUTE ON FUNCTION public.custom_access_token_hook FROM authenticated, anon, public;
```

**Após aplicar migration:**
1. Dashboard → Authentication → Hooks
2. Habilitar Custom Access Token Hook
3. Selecionar: `public.custom_access_token_hook`

## 5. Componente 3 — Migration: Atualizar 28 Policies

**Arquivo:** `supabase/migrations/20260414_fix_rls_use_app_metadata.sql`

Padrão da substituição em todas as 28 policies:

```sql
-- ANTES:
tenant_id = (auth.jwt() -> 'user_metadata' ->> 'tenant_id')::uuid

-- DEPOIS:
tenant_id = ((SELECT auth.jwt()) -> 'app_metadata' ->> 'tenant_id')::uuid
```

**Tabelas afetadas:** `tenants`, `kanbans`, `contacts`, `conversations`, `automatic_messages`, `columns` (via subquery), `messages` (via subquery)

## 6. Script de Backfill

**Arquivo:** `scripts/backfill-app-metadata.ts`

```typescript
import { createClient } from '@supabase/supabase-js';

const supabase = createClient(
  process.env.SUPABASE_URL!,
  process.env.SUPABASE_SERVICE_ROLE_KEY!
);

const { data: users } = await supabase
  .from('users')
  .select('id, tenant_id, role');

for (const user of users ?? []) {
  await supabase.auth.admin.updateUserById(user.id, {
    app_metadata: {
      tenant_id: user.tenant_id,
      role: user.role,
    },
  });
  console.log(`Updated app_metadata for user ${user.id}`);
}
```
