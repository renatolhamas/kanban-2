# F4.plan.002 — Padronização de Variáveis de Ambiente nas Rotas de Auth

**Data:** 2026-04-16  
**Origem:** Diagnóstico de inconsistência durante desenvolvimento da Story 4.1  
**Status:** ✅ CONCLUÍDO

---

## Contexto

As rotas de autenticação usavam nomes inconsistentes para as variáveis de ambiente do Supabase:
- Algumas usavam `SUPABASE_URL`
- Outras usavam `NEXT_PUBLIC_SUPABASE_URL`
- `change-password` fazia fallback entre os dois (`SUPABASE_URL || NEXT_PUBLIC_SUPABASE_URL`)

Isso causava erros em runtime dependendo de qual variável estava definida no `.env.local`.

## Decisão

Padronizar **todas** as rotas para `NEXT_PUBLIC_SUPABASE_URL`, pois:
- Já era o padrão das rotas mais recentes (`forgot-password`, `resend-confirmation`)
- Variáveis `NEXT_PUBLIC_*` funcionam tanto em server-side quanto client-side no Next.js
- `SUPABASE_SERVICE_ROLE_KEY` **permanece sem prefixo** (nunca deve ser exposta ao browser)

## Arquivos Alterados

| Arquivo | Mudança |
|---------|---------|
| `app/api/auth/login/route.ts` | `SUPABASE_URL` → `NEXT_PUBLIC_SUPABASE_URL` |
| `app/api/auth/register/route.ts` | `SUPABASE_URL` → `NEXT_PUBLIC_SUPABASE_URL` |
| `app/api/auth/profile/route.ts` | `SUPABASE_URL` → `NEXT_PUBLIC_SUPABASE_URL` |
| `app/api/auth/change-password/route.ts` | `SUPABASE_URL \|\| NEXT_PUBLIC_SUPABASE_URL` → `NEXT_PUBLIC_SUPABASE_URL` |

## Variáveis no .env.local (padrão final)

```env
NEXT_PUBLIC_SUPABASE_URL=https://...
NEXT_PUBLIC_SUPABASE_ANON_KEY=...
NEXT_PUBLIC_APP_URL=http://localhost:3017
SUPABASE_SERVICE_ROLE_KEY=...
```
