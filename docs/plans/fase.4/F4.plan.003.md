## Issues & Plano de Correção

### 🔴 Fix 1 — Drop View SECURITY DEFINER (`test_user_access`)

**O que é:**  
A view `public.test_user_access` foi criada com a propriedade `SECURITY DEFINER`.
Isso faz com que qualquer consulta à view execute com as permissões do superadmin que
a criou, ignorando completamente as RLS policies do banco.
Qualquer usuário autenticado que consiga consultá-la receberia dados de todos os tenants.

**Estado atual:**
```sql
-- View executa como superadmin, bypassa RLS
CREATE VIEW public.test_user_access AS SELECT ...;
-- SECURITY DEFINER implícito no owner
```

**Estado após correção:**
```sql
-- View não existe mais — era somente para testes
DROP VIEW IF EXISTS public.test_user_access;
```

**Estratégia:**  
Drop da view. É uma view de debug/teste sem uso em produção.  
Migration: `drop_test_user_access_view`

---

### 🟠 Fix 2 — RLS sem Políticas (`failed_registrations`)

**O que é:**  
A tabela `failed_registrations` tem `ROW LEVEL SECURITY = ON` mas nenhuma policy
foi criada. No PostgreSQL, RLS ativo sem policies = acesso negado para todos os
usuários normais. A tabela existe mas está totalmente inacessível via cliente.
Como é uma tabela de uso interno de sistema (auditoria de falhas de registro),
ela só é acessada pelo backend via `service_role` key — que bypassa RLS por definição.

**Estado atual:**
```sql
-- RLS ON, nenhuma policy → acesso negado para todos os usuários autenticados
ALTER TABLE public.failed_registrations ENABLE ROW LEVEL SECURITY;
-- (sem policies)
```

**Estado após correção:**
```sql
-- RLS desabilitado — tabela é interna, acessada apenas por service_role
ALTER TABLE public.failed_registrations DISABLE ROW LEVEL SECURITY;
```

**Estratégia:**  
Desabilitar RLS. A tabela não é consultada por usuários finais.
O acesso é feito exclusivamente via `service_role` (backend Node/API), que ignora RLS.
Migration: `disable_rls_failed_registrations`

### ✅ Fix 3 — JWT `user_metadata` → `app_metadata` (JÁ RESOLVIDO)

**O que é:**  
As RLS policies antigas usavam `user_metadata` para obter o `tenant_id` do JWT.
`user_metadata` é editável pelo próprio usuário via API pública do Supabase Auth,
o que tornava possível um usuário se atribuir a qualquer tenant_id manualmente.
`app_metadata` é imutável pelo usuário — apenas o backend/admin pode escrever.

**Estado atual (resolvido):**  
As migrations `create_custom_access_token_hook` e `fix_rls_use_app_metadata`
(ambas de 2026-04-16) já corrigiram este problema:
- Função `custom_access_token_hook` injeta `tenant_id` e `role` em `app_metadata` do JWT
- Todas as policies usam `(auth.jwt() -> 'app_metadata' ->> 'tenant_id')::uuid`

**Ação restante (não é migration):**  
Verificar se o hook está registrado no Supabase Auth Dashboard:
- Auth → Hooks → "Custom Access Token" → `public.custom_access_token_hook`
- Sem este registro, a função existe mas nunca é chamada

### 🟡 Fix 4 — RLS Performance: `auth.uid()` per-row (`users` table)

**O que é:**  
As 4 policies da tabela `users` usam `auth.uid()` diretamente na cláusula `USING`.
O PostgreSQL avalia essa função **para cada linha** retornada pela query (per-row evaluation).
Com o volume atual é imperceptível, mas ao escalar cria alto CPU e queries lentas.
A solução é envolver em `(SELECT auth.uid())` — o banco avalia uma única vez por query
e reutiliza o resultado (chamado de "init plan optimization").

**Estado atual:**
```sql
-- Per-row evaluation (lento em escala)
USING (auth.uid() = id)
```

**Estado após correção:**
```sql
-- Single evaluation por query (otimizado)
USING ((SELECT auth.uid()) = id)
```

**Estratégia:**  
Drop e recriação das 4 policies da tabela `users`.
Migration: `fix_users_rls_auth_uid_initplan`

```sql
-- Drop
DROP POLICY "Users can read own record" ON public.users;
DROP POLICY "Users can insert own record" ON public.users;
DROP POLICY "Users can update own record" ON public.users;
DROP POLICY "Users can delete own record" ON public.users;

-- Recreate com (SELECT auth.uid())
CREATE POLICY "Users can read own record" ON public.users
  FOR SELECT TO authenticated
  USING ((SELECT auth.uid()) = id);

CREATE POLICY "Users can insert own record" ON public.users
  FOR INSERT TO authenticated
  WITH CHECK ((SELECT auth.uid()) = id);

CREATE POLICY "Users can update own record" ON public.users
  FOR UPDATE TO authenticated
  USING ((SELECT auth.uid()) = id)
  WITH CHECK ((SELECT auth.uid()) = id);

CREATE POLICY "Users can delete own record" ON public.users
  FOR DELETE TO authenticated
  USING ((SELECT auth.uid()) = id);
```
