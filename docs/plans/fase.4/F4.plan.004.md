## 🔍 Problemas Identificados

### Problema 1: Password Protection Desabilitada (AUTH)

**Severidade:** 🔴 ALTO

**O Quê:** Supabase Auth não está verificando senhas contra HaveIBeenPwned.org. Usuários podem usar senhas que foram expostas em data breaches conhecidos.

**Por Quê Importa:**
- Seu projeto é production multi-tenant (B2B SaaS)
- Usuários podem ter contas comprometidas sem saber
- Atacantes podem fazer força bruta com senhas expostas
- Violação = acesso a dados de múltiplos clientes WhatsApp

**Impacto no Projeto:** 🔴 CRÍTICO — Risco de segurança direto

---

### Problema 2: 3 Foreign Keys Sem Índice

**Severidade:** 🟡 MÉDIO

**O Quê:** As colunas abaixo têm Foreign Keys mas NÃO têm índices:
1. `automatic_messages.scheduled_kanban_id` → kanbans
2. `conversations.contact_id` → contacts  
3. `conversations.column_id` → columns

**Por Quê Importa:**
- Sem índice, DELETE em tabela pai (ex: deletar kanban) → full table scan na tabela filha
- `conversations` é sua tabela principal (WhatsApp chats)
- Deletar um contato varre TODAS as conversas → lento
- JOINs nessas colunas também ficam lentos

**Impacto no Projeto:** 🟡 PERFORMANCE — Operações de delete/join degradadas

---

### Problema 3: RLS Disabled em `failed_registrations`

**Severidade:** 🟡 MÉDIO

**O Quê:** Tabela `failed_registrations` tem RLS desabilitado. Qualquer usuário autenticado de qualquer tenant pode ver registros de falha de qualquer outro tenant.

**Por Quê Importa:**
- Dados sensíveis: emails, IDs de tenants, status de falha
- Vazamento entre clientes
- Violação de isolamento multi-tenant

**Status:** ⚠️ CONTROLADO (ver detalhes abaixo)

**Impacto no Projeto:** 🟡 SECURITY — Potencial vazamento de dados

---

### Problema 4: 7 Índices Não-Utilizados

**Severidade:** 🟢 BAIXO

**O Quê:** Os seguintes índices nunca foram usados em queries:
- `failed_registrations_email_idx`
- `failed_registrations_created_at_idx`
- `failed_registrations_resolved_at_idx`
- `idx_tenants_subscription`
- `idx_kanbans_main_tenant`
- `idx_contacts_phone_tenant`
- `idx_conversations_last_message`

**Por Quê Importa:**
- Índices não-usados consomem storage
- Ralentam INSERT/UPDATE (precisa atualizar índice)
- Ruído em metadata do banco

**Impacto no Projeto:** 🟢 HOUSEKEEPING — Cleanup de recursos

### AÇÃO 2: Criar Índices para Foreign Keys Faltantes

**Timeline:** Esta semana (antes do weekend)  
**Agent:** Dara (data-engineer)  
**Risk:** ✅ ZERO — `CONCURRENTLY` não bloqueia

#### Passos

1. **Criar Migração**

   ```bash
   supabase migration new add_missing_fk_indexes
   ```

2. **Editar arquivo de migração**

   ```sql
   -- supabase/migrations/{timestamp}_add_missing_fk_indexes.sql
   
   -- Add missing indexes on foreign keys for performance
   -- These FKs are used in DELETE operations and JOINs
   -- Without indexes: full table scans on dependent tables
   
   CREATE INDEX CONCURRENTLY idx_automatic_messages_scheduled_kanban
     ON public.automatic_messages(scheduled_kanban_id);
   
   CREATE INDEX CONCURRENTLY idx_conversations_contact_id
     ON public.conversations(contact_id);
   
   CREATE INDEX CONCURRENTLY idx_conversations_column_id
     ON public.conversations(column_id);
   ```

3. **Testar em Staging**

   ```bash
   # Pull staging branch
   git checkout -b feature/add-fk-indexes
   
   # Apply migration
   supabase db push
   
   # Verify indexes were created
   SELECT schemaname, tablename, indexname
   FROM pg_indexes
   WHERE indexname LIKE 'idx_%conversations%'
      OR indexname LIKE 'idx_%automatic%';
   ```

4. **Confirmar Performance Melhora**

   ```sql
   -- Antes (sem índice): Seq Scan, full table scan
   EXPLAIN ANALYZE
   DELETE FROM automatic_messages WHERE scheduled_kanban_id = 'some-uuid';
   
   -- Depois (com índice): Index Scan, rápido
   -- (Exemplo, não executar em produção)
   ```

5. **Merge para Main**

   ```bash
   git add supabase/migrations/
   git commit -m "feat: add missing indexes on foreign keys

   - idx_automatic_messages_scheduled_kanban
   - idx_conversations_contact_id
   - idx_conversations_column_id
   
   These improve DELETE and JOIN performance on conversations table."
   
   git push origin feature/add-fk-indexes
   # Create PR, review, merge
   ```

6. **Deploy para Produção**

   ```bash
   # Via CI/CD (GitHub Actions, etc.)
   git merge feature/add-fk-indexes main
   # Automated: supabase db push runs in CI
   ```

### AÇÃO 3: Verificar & Controlar RLS em `failed_registrations`

**Timeline:** Esta semana (antes de Friday deploy)  
**Agent:** Dara ou @dev  
**Risk:** 🟡 MÉDIO — Requer decisão arquitetural

#### Contexto

A tabela `failed_registrations` armazena registros de tentativas FALHADAS de registro (usuários não-autenticados, registro incompleto). 

**Status Atual:** RLS disabled (intencional no design)

**Motivo:** Registros são criados antes de usuário estar autenticado, então RLS não é aplicável.

**Risco:** Se a tabela for exposta via PostgREST, qualquer usuário autenticado vê dados de qualquer tenant.

#### Passos

1. **Verificar Exposição via PostgREST**

   ```bash
   # Test 1: Sem autenticação (deve falhar)
   curl "https://your-project.supabase.co/rest/v1/failed_registrations"
   # Esperado: 401 Unauthorized
   
   # Test 2: Com token válido
   curl "https://your-project.supabase.co/rest/v1/failed_registrations" \
     -H "Authorization: Bearer YOUR_TOKEN"
   # Se retorna dados: PROBLEMA ❌
   # Se retorna 403: OK ✅
   ```

2. **Decisão Baseada em Resultado**

   **Se exposto via PostgREST (recebeu dados em Test 2):**
   
   - Adicionar `tenant_id` à tabela
   - Habilitar RLS
   - Criar política
   
   (Vira nova story de migração)

   **Se NÃO exposto via PostgREST (recebeu 403 em Test 2):**
   
   - Deixar como está (RLS disabled é OK)
   - Documentar decisão
   - Proceder com Ação 4

3. **Documentar Decisão**

   Adicionar ao `docs/db/rls.md`:
   ```markdown
   ## Special Case: failed_registrations
   
   **RLS Status:** DISABLED (intencional)
   
   **Razão:** Registros são criados por usuários não-autenticados
   
   **Segurança:** Controlada por PostgREST — tabela NÃO é exposta
   
   **Acesso:** Service Role key apenas (backend/Edge Functions)
   ```

   SELECT 
     schemaname,
     tablename,
     indexname,
     idx_scan,
     idx_tup_read,
     idx_tup_fetch,
     pg_size_pretty(pg_relation_size(indexrelid)) AS size

   WHERE indexname IN (
     'failed_registrations_email_idx',
     'failed_registrations_created_at_idx',
     'failed_registrations_resolved_at_idx',
     'idx_tenants_subscription',
