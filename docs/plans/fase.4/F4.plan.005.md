# F4.plan.005 — Visibilidade Condicional do UserMenu

**Data:** 2026-04-16
**Status:** ✅ CONCLUÍDO
**Componente afetado:** `UserMenu` (`src/components/layout/UserMenu.tsx`)

---

## Objetivo

O `UserMenu` deve aparecer **somente quando o usuário está autenticado**.
Quando não autenticado, não deve exibir nada.

---

## Histórico de Tentativas (O que NÃO funcionou)

**Tentativa 1:** `localStorage.getItem('auth_token')` no `Header.tsx`
**Por que não deu:** O login nunca escreve em `localStorage`. O token vai para cookie via `Set-Cookie` no servidor. O localStorage estava sempre vazio → `UserMenu` sempre oculto.

**Tentativa 2:** `document.cookie.split('; ').some(c => c.startsWith('token='))`
**Por que não deu:** Dois erros: (1) nome errado — deveria ser `auth_token=`. (2) Mesmo com nome correto não funcionaria: o cookie `auth_token` tem flag `HttpOnly`, que bloqueia qualquer leitura via JavaScript no browser.

**Tentativa 3:** `document.cookie.includes('auth_token=')`
**Por que não deu:** Mesma causa raiz da tentativa 2. Cookie `HttpOnly` é invisível para `document.cookie` independente do nome. `hasToken` sempre retornava `false` → `UserMenu` sempre oculto.

**Tentativa 4:** Endpoint `GET /api/auth/me` + `fetch` no `Header.tsx`
**Por que não deu:** A abordagem era tecnicamente correta, mas atacava o problema errado. O `UserMenu` aparecia não porque a autenticação não estava sendo verificada, mas porque o `Header` é renderizado em **todas as páginas** — inclusive nas páginas públicas como `/login`. Resolver pelo cookie/API era complexidade desnecessária.

**Tentativa 5 (middleware):** Redirecionar usuário autenticado de `/login` para `/`
**Por que não resolveu completamente:** O `matcher` do `middleware.ts` excluía explicitamente a rota `/login` com o padrão `/((?!_next|.*\\..*|login|register).*)` — o middleware nunca rodava nessa rota.

---

## Causa Raiz Real

O problema **não era de autenticação**. Era de **layout**.

`RootLayout.tsx` renderiza o `<Header />` em **todas as páginas** (com o comentário `{/* Header - Always visible */}`), mas só oculta o `<Sidebar />` nas páginas de auth. O `UserMenu` dentro do Header ficava sempre visível porque nunca havia uma verificação de rota.

---

## Solução Final (Simples e Correta)

Em `src/components/layout/Header.tsx`, adicionada lista de páginas de auth e verificação de rota via `usePathname()` — sem cookie, sem fetch, sem API:

```typescript
const AUTH_PAGES = [
  '/login',
  '/register',
  '/forgot-password',
  '/reset-password',
  '/resend-confirmation',
  '/change-password',
];

const isAuthPage = AUTH_PAGES.some((page) => pathname.startsWith(page));

// Renderização:
{!isAuthPage && <UserMenu />}
```

### Arquivos modificados

| Ação | Arquivo | Descrição |
|------|---------|-----------|
| MODIFICADO | `src/components/layout/Header.tsx` | Oculta UserMenu em páginas de auth por rota |
| CRIADO | `app/api/auth/me/route.ts` | Criado durante tentativas — mantido para uso futuro |
| MODIFICADO | `middleware.ts` | Matcher corrigido para incluir `/login` e `/register` |

---

## Lição Aprendida

> Antes de implementar soluções complexas (cookie, fetch, API, middleware), verificar a **estrutura de layout**. O problema era que o Header aparecia em páginas onde não deveria mostrar o menu — não que a autenticação não estava sendo verificada.
