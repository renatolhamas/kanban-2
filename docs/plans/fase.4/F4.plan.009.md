## 2. Diagnóstico Técnico — A Causa Raiz

### 2.1 Como o AuthContext funciona hoje

O estado de autenticação da aplicação é gerenciado por `src/context/AuthContext.tsx`, que é montado uma única vez em `app/layout.tsx` e distribui `isAuthenticated`, `isLoading` e `user` para toda a aplicação via React Context.

A verificação de autenticação acontece assim:

```tsx
// src/context/AuthContext.tsx
const [isAuthenticated, setIsAuthenticated] = useState(false); // começa como false
const [isLoading, setIsLoading]             = useState(true);  // começa como true
const [user, setUser]                       = useState(null);

useEffect(() => {
  async function checkAuth() {
    const response = await fetch('/api/auth/me'); // lê o cookie httpOnly
    const data = await response.json();
    if (data.authenticated) {
      setIsAuthenticated(true);
      setUser({ sub: data.sub });
    } else {
      setIsAuthenticated(false);
      setUser(null);
    }
    // finally: setIsLoading(false)
  }
  checkAuth();
}, []); // ← ARRAY VAZIO: roda apenas UMA vez, no mount inicial
```

**O `useEffect` com `[]` roda somente quando o `AuthProvider` é montado pela primeira vez.**

### 2.2 Como o login funciona hoje

```
1. Usuário preenche o formulário e clica "Entrar"
2. POST /api/auth/login
   → Supabase signInWithPassword
   → Gera JWT customizado (HS256)
   → Seta cookie: auth_token=<jwt>; HttpOnly; SameSite=Lax; Max-Age=3600
   → Retorna { success: true }

3. Login page recebe { success: true }
   → Chama router.push('/profile')   ← NAVEGAÇÃO CLIENT-SIDE

4. Next.js troca a página para /profile SEM recarregar o browser
   → O AuthProvider NÃO é desmontado
   → O AuthProvider NÃO é remontado
   → O useEffect([], checkAuth) NÃO roda novamente

5. /profile renderiza
   → isAuthenticated continua false (valor stale do mount inicial)
   → isLoading já é false (terminou no mount inicial)
   → Sidebar: condição (!isAuthPage && !isLoading && isAuthenticated) → false → NÃO renderiza
   → UserMenu: mesma condição → false → NÃO renderiza
```

### 2.3 Por que o conteúdo da página funciona mas o layout não

As páginas como `/profile` **não consomem o `useAuth()` hook**. Elas fazem fetch direto para `/api/auth/profile` com o cookie httpOnly presente no browser. Por isso, o conteúdo da página carrega corretamente — o cookie está lá, válido. Mas o Sidebar e o UserMenu vivem no layout (`RootLayout` e `Header`) e dependem exclusivamente do `AuthContext`, que está com estado desatualizado.

### 2.4 Por que o F5 "resolve"

Um hard refresh força o browser a recarregar completamente a página. O `AuthProvider` é desmontado e remontado do zero. O `useEffect([], checkAuth)` roda novamente — desta vez com o cookie já presente — e `GET /api/auth/me` retorna `{ authenticated: true }`. O estado é atualizado corretamente e os elementos aparecem.

### 2.5 Problema secundário: AUTH_PAGES inconsistente

Existem **duas listas separadas** de páginas públicas no código, e elas são diferentes:

| Página | RootLayout.tsx | Header.tsx |
|---|---|---|
| `/login` | ✅ listada | ✅ listada |
| `/register` | ✅ listada | ✅ listada |
| `/forgot-password` | ✅ listada | ✅ listada |
| `/reset-password` | ✅ listada | ✅ listada |
| `/resend-confirmation` | ❌ **ausente** | ✅ listada |
| `/change-password` | ❌ **ausente** | ✅ listada |

Consequência: em `/resend-confirmation` e `/change-password`, o Sidebar **apareceria** (pois RootLayout não considera essas páginas como auth), mas o UserMenu **não** (Header as exclui). Comportamento split-brain e inconsistente.

## 4. O Que Precisa Ser Feito

### 4.1 Solução escolhida: expor uma ação `login()` no AuthContext

Esta é a abordagem mais limpa e correta. Em vez de depender de navegação hard ou de um `useEffect` com dependência de pathname, o `AuthContext` expõe uma função `login(user)` que atualiza o estado diretamente e de forma síncrona. A página de login chama essa função após o sucesso do POST, antes de navegar.

**Vantagens:**
- Estado atualizado antes da navegação → componentes de layout renderizam corretamente na primeira exibição
- Sem hard refresh, sem recarregamento desnecessário
- Padrão React idiomático
- Não cria dependências de pathname no useEffect (o que causaria chamadas excessivas à API)

### 4.2 Mudanças necessárias

#### Mudança 1 — `src/context/AuthContext.tsx`

Adicionar uma função `login(userData)` ao contexto que atualiza `isAuthenticated` e `user` diretamente:

```tsx
// Adicionar ao AuthContextValue (interface/type do contexto):
login: (user: { sub: string; email?: string }) => void;
logout: () => void; // opcional, se não existir

// Implementação:
const login = useCallback((userData: { sub: string; email?: string }) => {
  setIsAuthenticated(true);
  setUser(userData);
}, []);

// Expor via context value:
value={{ isAuthenticated, isLoading, user, login }}
```

#### Mudança 2 — `app/(auth)/login/page.tsx`

Após receber `{ success: true }` da API, chamar `login()` antes de navegar:

```tsx
// Antes (atual):
router.push('/profile');

// Depois (corrigido):
const { login } = useAuth();
// ...
if (data.success) {
  login({ sub: data.sub, email: formData.email }); // atualiza contexto imediatamente
  router.push('/profile'); // navega com estado já correto
}
```

> **Nota:** A API de login precisará retornar o `sub` (userId) no body de resposta para que a página de login possa passá-lo ao contexto. Verificar se já retorna ou se precisa ser adicionado ao response body.

#### Mudança 3 — Unificar AUTH_PAGES em um único local

Criar uma constante compartilhada (ex: `src/lib/auth-pages.ts`) com a lista completa de páginas públicas, e importá-la em ambos `RootLayout.tsx` e `Header.tsx`:

```tsx
// src/lib/auth-pages.ts
export const AUTH_PAGES = [
  '/login',
  '/register',
  '/forgot-password',
  '/reset-password',
  '/resend-confirmation',
  '/change-password',
];
```

Substituir as duas listas locais por essa importação.
