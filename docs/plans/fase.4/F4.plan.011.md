# Plano de Ação: F4.plan.011 [CONCLUÍDO] - Sincronização de Auth & RLS

**Status:** ✅ CONCLUÍDO

> [!NOTE]
> Este plano complementa o **F4.plan.010**, detalhando a estratégia técnica para resolver a "crise de identidade" entre o frontend e o banco de dados.

## Contexto do Problema
Atualmente, o sistema possui uma "crise de identidade" entre o frontend e o banco de dados:
1. **Identidade Interna:** O usuário faz login via `/api/auth/login`, que gera um JWT interno e o armazena em um cookie HTTP-only. O frontend reconhece o `tenant_id`, mas este token não é enviado automaticamente para as chamadas diretas do Supabase via SDK.
2. **Segurança do Banco (RLS):** As políticas de Row Level Security (RLS) do Supabase são rigorosas e protegem os dados do Kanban exigindo que o `tenant_id` esteja presente no `app_metadata` do JWT enviado na requisição.
3. **A Falha:** Como o navegador não possui a sessão oficial do Supabase ativa (apenas o nosso cookie interno), as chamadas para o Kanban são executadas como "Anônimo". Como resultado, o banco de dados nega o acesso aos dados, resultando em "Nenhum quadro encontrado", mesmo que os dados existam.

**Objetivo:** Fazer com que o Supabase entenda e aceite o nosso token interno como uma credencial válida e segura, sem burlar o RLS.

---

## Passo a Passo para Resolução

### 1. Alinhamento de Claims do JWT (Backend)
Para que o Supabase aceite nosso token, ele precisa "parecer" um token do Supabase.
- **Arquivo:** `src/lib/jwt.ts` e `app/api/auth/login/route.ts`.
- **Ação:** Atualizar a geração do JWT para incluir os claims obrigatórios do Supabase:
    - `aud`: "authenticated"
    - `role`: "authenticated"
    - `app_metadata`: `{ "tenant_id": "...", "role": "..." }`
- **Importante:** Garantir que o `JWT_SECRET` usado pelo servidor seja exatamente o mesmo `JWT_SECRET` (Supabase JWT Secret) configurado no banco de dados.

### 2. Exposição Segura do Token para o Frontend
O SDK do Supabase no browser precisa ter acesso ao token para enviá-lo no header `Authorization`.
- **Ação:** Modificar a rota `/api/auth/login` para, além de setar o cookie HTTP-only (para segurança da API), retornar o token no corpo da resposta (JSON). 
- **Ação:** Armazenar temporariamente este token no `AuthContext` (em memória, nunca em localStorage por segurança).

### 3. Factory Dinâmica do Cliente Supabase (Frontend)
Substituir o cliente estático por um que saiba injetar o token.
- **Arquivo:** `src/lib/supabase-client.ts`.
- **Ação:** Criar uma função `getSupabaseClient(token?: string)` que inicializa o cliente com o header `Authorization: Bearer <token>` se o token existir.

### 4. Atualização dos Hooks de Dados
Fazer os hooks consumirem o token do contexto.
- **Arquivos:** `src/hooks/useConversations.ts` e `src/hooks/useKanbans.ts`.
- **Ação:** Utilizar o token do `AuthContext` para inicializar o cliente Supabase dentro dos hooks.

### 5. Verificação de Segurança
- Testar o isolamento multi-tenant garantindo que o RLS está bloqueando acessos indevidos e permitindo apenas os do tenant correto.

---

## User Review Required

> [!IMPORTANT]
> Esta alteração exige que o segredo JWT do Supabase esteja disponível no `.env` como `JWT_SECRET`. Se estivermos em produção, isso é crítico. Em ambiente local, usaremos o segredo padrão do Supabase Docker (se aplicável).

> [!WARNING]
> Ao passar o token para o frontend em memória, ele fica disponível para o código JS da página. Esta é a abordagem padrão para Single Page Applications (SPAs), mas deve ser feita com cuidado. O cookie HTTP-only continuará protegendo as rotas de API do Next.js.
