# Plano de Ação: F4.plan.014 [CONCLUÍDO] - Estudo de Transição para Algoritmo ECC / ES256

**Status:** ✅ CONCLUÍDO

> [!NOTE]
> Este plano complementa o **F4.plan.013**, expandindo a investigação de RLS para a incompatibilidade técnica de algoritmos de assinatura JWT.

## Contexto
O usuário identificou que o projeto Supabase estava configurado para utilizar o algoritmo **ECC (P-256)** para assinatura de JWTs. O sistema atual utilizava HS256, gerando incompatibilidade de assinatura (PGRST301).

**Objetivo:** Adaptar o sistema para utilizar assinaturas assimétricas (ES256) caso o Custom Auth fosse mantido.

---

## Descobertas Técnicas
1. **Algoritmo Assimétrico:** O ES256 exige um par de chaves (Pública e Privada).
2. **Requisito de Private Key:** Para o backend gerar (mint) tokens aceitáveis pelo Supabase, ele precisaria da Chave Privada correspondente ao JWK configurado no Dashboard.
3. **Ponto de Bloqueio:** O usuário possuía apenas o JWK de Verificação (Chave Pública), tornando matematicamente impossível a assinatura de tokens manuais válidos via backend sem a obtenção da Chave Privada.

---

## Decisão de Projeto
Após análise, o plano 014 foi **substituído pelo plano 015**, que optou por alinhar o sistema com a autenticação nativa do Supabase, eliminando a necessidade de gerenciar chaves assimétricas manualmente.

> [!NOTE]
> Este plano serve como documentação da tentativa de implementação de algoritmos assimétricos e os requisitos técnicos encontrados.
