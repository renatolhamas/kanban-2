# Plano de Ação: F4.plan.016 [CONCLUÍDO] - Verificação Nativa de Sessão e Restauração da UI

> [!NOTE]
> Este plano complementa o **F4.plan.015**, resolvendo os problemas de autenticação e UI que surgiram após a migração para a verificação nativa de sessão.

## Contexto do Problema
Após a migração para o token oficial do Supabase (Plano 15), as páginas que utilizam o middleware de autenticação (`/profile`, `/settings`, etc.) começaram a retornar **401 Unauthorized**. Isso ocorre porque o nosso middleware atual tenta verificar o token usando **HS256**, enquanto os tokens oficiais do projeto estão em **ES256 (ECC)**.

Como efeito colateral, a `RootLayout` esconde a Sidebar e o User Menu, pois entende que o usuário não está autenticado.

---

## Proposta de Solução
Mudar a estratégia de verificação de manual para **nativa**. Em vez de tentarmos validar a assinatura nós mesmos usando segredos no Next.js, chamaremos o método oficial do Supabase.

### 1. Backend: Middleware de Autenticação
- **Arquivo:** `lib/middleware/auth.ts`
- **Ação:** Substituir `verifyJWT(token)` por uma chamada ao `supabase.auth.getUser(token)`.
- **Ação:** Extrair os metadados (`tenant_id`, `role`) do objeto de usuário retornado pelo Supabase.

### 2. Frontend: Restauração da UI
- **Ação:** Com o middleware funcionando, o endpoint `/api/auth/me` voltará a retornar status `200 OK`.
- **Resultado:** A `RootLayout` detectará o estado autenticado e renderizará novamente a Sidebar e o User Menu.

---

## Verificação de Sucesso
1.  **Sidebar/Menu:** Devem reaparecer após o refresh da página.
2.  **Página /settings:** Deve carregar o status do WhatsApp sem erro 401.
3.  **Logs:** O console do servidor deve confirmar que o `tenant_id` está sendo extraído corretamente via `app_metadata`.
