# F5.plan.003 — Migração para Supabase Auth Nativa (Sessão Completa)

**Status:** Approved  
**Tipo:** Feature Complete  
**Escopo:** Auth Layer Refactor  
**Complexidade:** MEDIUM (M - 8pts)  
**Executor Proposto:** @dev  

---

## Problema

A autenticação atual é um **sistema híbrido inconsistente** onde:

1. Login usa `supabase.auth.signInWithPassword()` ✅
2. Sessão é armazenada em **cookie httpOnly customizado** ❌
3. `AuthContext` chama `/api/auth/me` (rota customizada) ❌
4. Front-end injeta JWT manualmente em cada cliente Supabase ❌
5. Logout só limpa cookie, não chama `supabase.auth.signOut()` ❌

**Impacto:** Este sistema híbrido causa:
- GoTrueJS instanciado mas sem controle → locks orphanados (`Lock "lock:sb-..." was not released within 5000ms`)
- Inconsistência entre backend (Supabase auth) e frontend (cookie customizado)
- Duplicação de lógica (rota `/api/auth/me` + `AuthContext`)
- Manutenção complexa (duas "fontes de verdade" de sessão)

**Visão Anterior:** Em conversas anteriores foi decidido migrar para auth nativa do Supabase (com exceção de register).

---

## Solução Proposta

Migrar **completamente** para autenticação nativa do Supabase usando `@supabase/ssr` (recomendado para Next.js), permitindo que o GoTrueJS gerencie sessões, tokens e refresh automático.

### Arquitetura Alvo

| Camada | Estado Atual | Estado Alvo |
|--------|-------------|-----------|
| **Login** | `supabase.auth.signInWithPassword()` em rota | ✅ Manter |
| **Sessão** | Cookie httpOnly customizado | **GoTrueJS** (via `@supabase/ssr`) |
| **AuthContext** | Chama `/api/auth/me` | `supabase.auth.getUser()` nativo |
| **Front-end client** | Injeta token manualmente | Usa `supabase.auth.getSession()` |
| **Logout** | Limpa cookie customizado | `supabase.auth.signOut()` |

## Riscos e Mitigações

| Risco | Mitigation |
|-------|-----------|
| Quebrar RLS em quereis | Verificar auth.jwt() nas policies antes de remover injeção manual |
| Perder lógica de refresh | `@supabase/ssr` + middleware Next.js maneja automaticamente |
| Sessão não persiste entre abas | `@supabase/ssr` usa localStorage (padrão) |
| Conflito com register (server-side) | Manter rota `/api/auth/register`, desacoplar de login |

## Decisões Tomadas

1.  **Login:** **Client-side**. Utilizaremos `supabase.auth.signInWithPassword()` diretamente no componente de Login. Isso permite que o GoTrueJS gerencie a sessão e os cookies nativamente sem interferência de rotas API customizadas.
    
2.  **Register:** **Server-side (MANTER)**. A rota `/api/auth/register` será mantida para garantir a atomicidade da criação do usuário e do Tenant em uma operação segura no servidor, protegendo a integridade do banco de dados.

3.  **Alteração de Senha:** **Client-side**. Migraremos para o uso exclusivo de `supabase.auth.updateUser()`, eliminando a necessidade de rotas customizadas para esta função.

4.  **Middleware:** **SSR Nativo**. Substituiremos a lógica customizada de `lib/middleware/auth.ts` pelo middleware padrão do `@supabase/ssr`, garantindo o refresh automático de tokens e proteção consistente de rotas.
