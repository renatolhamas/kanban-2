# F5.plan.004 — Corrigir Integração Webhook Evolution GO

**Data:** 2026-04-21  
**Status:** CONCLUÍDO  
**Autor:** Atlas (Analyst)  
**Motivação:** Webhook nunca chegou na Vercel. Análise da documentação real revelou 4 erros de implementação.

---

## 📋 Contexto

Durante o teste da Story 5.1 (Webhook Contact Auto-Registration), constatou-se que:

1. Mensagens enviadas ao WhatsApp **não geram dados no banco**
2. Logs da Vercel **não mostram nenhuma requisição POST** para `/api/webhooks/evo-go`
3. O webhook **nunca chegou** à aplicação

Após análise da documentação real do Evolution GO (`docs/evogo/`) e do código atual, foram identificados **4 erros de implementação** que, juntos, impedem completamente o funcionamento.

---

## 🔴 Problemas Identificados

### Problema 1 — Webhook Secret não existe no Evolution GO

A Story 3.1 (AC 5) definiu validação HMAC-SHA256 com `EVO_GO_WEBHOOK_SECRET`. Porém, a documentação oficial do Evolution GO **não menciona** HMAC, X-Signature, ou qualquer forma de webhook secret.

**Efeito no código atual:**
```typescript
// route.ts linha 35
if (!supabaseUrl || !supabaseServiceRoleKey || !evoGoWebhookSecret) {
  // EVO_GO_WEBHOOK_SECRET está vazio → entra aqui → retorna 200 sem processar nada
  return NextResponse.json({ statusCode: 200 }, { status: 200 });
}
```
Mesmo que o webhook chegasse, seria silenciosamente descartado.

---

### Problema 2 — Webhook nunca foi configurado na instância

O Evolution GO **não envia webhook automaticamente**. É necessário chamar `POST /instance/connect` com o campo `webhookUrl` para registrar a URL de destino. Esse passo nunca foi executado com a URL correta.

**O que a documentação diz (api-instances.md):**
```json
POST /instance/connect
Headers: { "apikey": "TOKEN-DA-INSTANCIA" }
Body: {
  "webhookUrl": "https://kanban.renatolhamas.com.br/api/webhooks/evo-go",
  "subscribe": ["MESSAGE", "CONNECTION", "QRCODE"]
}
```

Sem isso, o Evolution GO simplesmente não sabe para onde mandar os eventos.

---

### Problema 3 — Nomes dos eventos inconsistentes

O código atual usa nomes de eventos no estilo da Evolution API v2 (que é diferente do Evolution GO):

| No código atual | Na documentação Evolution GO |
|----------------|------------------------------|
| `MESSAGES_UPSERT` | `messages.upsert` ou `Message` |
| `CONNECTION_UPDATE` | `connection.update` ou `Connected` / `Disconnected` |
| `QRCODE_UPDATED` | `qrcode` ou `PairSuccess` |

Mesmo que o webhook chegasse, o `switch (event)` não bateria em nenhum case.

---

### Problema 4 — `NEXT_PUBLIC_APP_URL` aponta para localhost

Este é o **problema raiz** que explica por que o webhook nunca chegou.

O código em `app/api/settings/evo-go/qr/route.ts` (linha 157) monta a URL assim:
```typescript
const webhookUrl = `${process.env.NEXT_PUBLIC_APP_URL}/api/webhooks/evo-go?tenantId=${tenantId}`;
```

E o `.env.local` tem:
```env
NEXT_PUBLIC_APP_URL=http://localhost:3017
```

**Portanto a URL registrada na instância Evolution GO foi:**
```
http://localhost:3017/api/webhooks/evo-go?tenantId=4b54f906-6567-462e-a4fa-3e40da1eadfc
```

O Evolution GO tenta enviar o webhook para `localhost` — que é a máquina local de desenvolvimento — e obviamente **não consegue chegar na Vercel**.

A variável `NEXT_PUBLIC_APP_URL` já está correta na Vercel (`https://kanban.renatolhamas.com.br`), porém a instância foi conectada **em ambiente local** (onde vale `http://localhost:3017`), então o Evolution GO gravou a URL errada.

---

## ✅ Solução Proposta

### Mudança 1 — Remover validação HMAC permanentemente

**Arquivo:** `app/api/webhooks/evo-go/route.ts`

- Remover verificação de `evoGoWebhookSecret` do bloco de validação inicial
- Remover `parseSignatureHeader` e `validateEvoGoSignature` do handler
- Manter apenas validação de `supabaseUrl` e `supabaseServiceRoleKey`
- Remover imports não utilizados de `evo-go-signature.ts`

**Arquivo:** `lib/api/evo-go-env-validation.ts`

- Remover `EVO_GO_WEBHOOK_SECRET` da lista de variáveis obrigatórias

---

### Mudança 2 — Reconectar a instância via produção

`NEXT_PUBLIC_APP_URL` já está correto na Vercel. O código já está correto. O problema é que a instância foi conectada em ambiente local, então o Evolution GO tem `localhost` gravado.

**Verificado via API do Evolution GO:**
```json
"webhook": "http://localhost:3017/api/webhooks/evo-go?tenantId=4b54f906..."
```

**Solução:** Após o deploy das correções de código, o usuário deve **reconectar a instância WhatsApp** acessando `https://kanban.renatolhamas.com.br` e gerando um novo QR Code. O sistema chamará `callEvoGoConnect()` com a URL correta automaticamente, sobrescrevendo o `localhost` no Evolution GO.

---

### Mudança 3 — Confirmar que `webhookUrl` é passado no connect

**Arquivo:** `app/api/settings/evo-go/qr/route.ts` (linha 152-160)

O código já faz isso corretamente — monta a URL a partir de `NEXT_PUBLIC_APP_URL` e passa para `callEvoGoConnect()`. Não precisa mudar o código, apenas a variável de ambiente.

---

### Mudança 4 — Corrigir nomes dos eventos no switch case

**Arquivo:** `app/api/webhooks/evo-go/route.ts`

Adicionar suporte aos nomes reais usados pelo Evolution GO:

```typescript
switch (event) {
  // Mensagens
  case 'messages.upsert':
  case 'Message':
    await handleMessagesUpsert(supabase, resolvedTenantId, data);
    break;

  // Conexão
  case 'connection.update':
  case 'Connected':
  case 'Disconnected':
    await handleConnectionUpdate(supabase, resolvedTenantId, data);
    break;

  // QR Code
  case 'qrcode':
  case 'PairSuccess':
    await handleQRCodeUpdated(supabase, resolvedTenantId, data);
    break;
}
```

## 📁 Arquivos a Modificar

| Arquivo | Mudança |
|---------|---------|
| `app/api/webhooks/evo-go/route.ts` | Remover HMAC, corrigir switch de eventos |
| `lib/api/evo-go-env-validation.ts` | Remover `EVO_GO_WEBHOOK_SECRET` |
| `.env.local` | Remover `EVO_GO_WEBHOOK_SECRET`, corrigir `NEXT_PUBLIC_APP_URL` para dev |
| **Vercel ENV VARS** | Definir `NEXT_PUBLIC_APP_URL=https://kanban.renatolhamas.com.br` |

**Nota:** `lib/api/evo-go-client.ts` e `app/api/settings/evo-go/qr/route.ts` **não precisam de mudança de código** — já estão corretos. O problema era apenas a variável de ambiente.

## 🧪 Como Verificar

1. Configurar `NEXT_PUBLIC_APP_URL=https://kanban.renatolhamas.com.br` na Vercel
2. Deploy na Vercel após mudanças no código
3. **Reconectar instância** WhatsApp no sistema (gerar novo QR Code) — obrigatório para registrar a nova URL no Evolution GO
4. Enviar mensagem no WhatsApp
5. **Vercel logs** devem mostrar: `POST 200 /api/webhooks/evo-go`
6. **Banco de dados** deve mostrar:
   - `contacts.wa_name` preenchido com o nome real do contato
   - `contacts.is_group = false`
   - `conversations.unread_count = 1`
   - `conversations.last_message_at` atualizado
   - `conversations.evolution_message_id` preenchido

---

## ⚠️ Riscos

| Risco | Mitigação |
|-------|-----------|
| Sem validação HMAC, qualquer pessoa poderia chamar o endpoint | Endpoint retorna 200 sem expor dados. Aceitar por ora — Evolution GO não suporta assinatura. |
| Nome dos eventos pode variar por versão do servidor | Testar em staging antes. Adicionar log de evento desconhecido com o payload completo para debug. |
| URL pública incorreta no webhookUrl | `NEXT_PUBLIC_APP_URL=https://kanban.renatolhamas.com.br` na Vercel. Reconectar instância após mudança. |
