## ⚠️ Divergências CRÍTICAS (podem quebrar o fluxo)

### DIV-12 — Campo `instance` no webhook: nome vs UUID (MAIOR RISCO)

**O que é a divergência:**

A documentação (`events-system.md`) confirma que o campo `instance` no payload do webhook contém o **nome** da instância:

```json
{ "event": "MESSAGE", "instance": "vendas", "data": { ... } }
{ "event": "GROUP_UPDATE", "instance": "suporte", "data": { ... } }
{ "event": "CALL", "instance": "nome-da-instancia", "data": { ... } }
```

O nome da instância gerado pelo nosso código em `lib/api/evo-go-client.ts` (linha 96) é:

```typescript
name: `kanban-instance-${tenantId.substring(0, 8)}`
// Para tenantId "4b54f906-...", gera: "kanban-instance-4b54f906"
```

Ou seja, o nome é **determinístico e previsível** — é sempre `kanban-instance-` + os 8 primeiros caracteres do `tenantId`.

O lookup no webhook em `app/api/webhooks/evo-go/route.ts` busca assim:

```typescript
.eq('evolution_instance_id', instance)  // busca UUID, recebe nome
```

Mas o banco armazena em `evolution_instance_id` o **UUID retornado pelo Evolution GO** no create:

```
evolution_instance_id = "7957e835-16de-44e9-92ba-3a161209c49f"  (UUID do EvoGO)
instance no webhook   = "kanban-instance-4b54f906"              (nome gerado pelo código)
```

**Por que pode atrapalhar:**

A lookup nunca encontra o tenant. O webhook cai no fallback da URL (`?tenantId=...`), que funciona **somente porque a URL já inclui o tenantId**. Em um sistema multi-tenant com webhook global (sem tenantId na URL), o webhook seria descartado completamente.

**Como corrigir:**

O nome é determinístico — o código sempre gera `kanban-instance-${tenantId.substring(0, 8)}`. Não é necessário criar uma nova coluna nem alterar o create. Basta **calcular o nome esperado no momento do lookup** e usá-lo para identificar o tenant.

Modificar o lookup em `app/api/webhooks/evo-go/route.ts`:

```typescript
// Atual (lookup por UUID — nunca encontra):
.eq('evolution_instance_id', instance)

// Corrigido (derivar tenantId a partir do nome da instância):
if (instance && instance.startsWith('kanban-instance-')) {
  const tenantPrefix = instance.replace('kanban-instance-', ''); // ex: "4b54f906"
  const { data: tenantRow } = await supabase
    .from('tenants')
    .select('id')
    .ilike('id', `${tenantPrefix}%`)  // busca tenant cujo id começa com o prefixo
    .maybeSingle();

  if (tenantRow) {
    resolvedTenantId = tenantRow.id;
  }
}
```

**Ou**, alternativamente, adicionar coluna `evolution_instance_name` e populá-la no create:

```sql
ALTER TABLE tenants ADD COLUMN evolution_instance_name TEXT UNIQUE;
UPDATE tenants SET evolution_instance_name = 'kanban-instance-' || substring(id::text, 1, 8);
```

E fazer lookup direto:

```typescript
.eq('evolution_instance_name', instance)
```

**Recomendação:** A segunda opção (coluna dedicada) é mais robusta e explícita. A primeira (derivar do nome) funciona mas depende do padrão de nomenclatura não mudar.

### DIV-4 — Campos do `/instance/status` em PascalCase vs camelCase

**O que é a divergência:**

A documentação local (`api-instances.md`) diz que `GET /instance/status` retorna:
```json
{
  "connected": true,
  "loggedIn": true,
  "name": "Minha Empresa"
}
```

Campos em **camelCase**: `connected`, `loggedIn`.

O código em `lib/api/evo-go-client.ts` tenta:
```typescript
const connected = (statusData.Connected ?? statusData.connected ?? false);
const loggedIn = (statusData.LoggedIn ?? statusData.logged_in ?? false);
const name = (statusData.Name ?? statusData.name);
```

Prioriza **PascalCase** (`Connected`, `LoggedIn`, `Name`) com fallback para outros formatos. O comentário no código diz: `// Note: Evo GO returns "Connected" with uppercase C`.

**Por que pode atrapalhar:**

Se a API real retornar em camelCase (conforme documentação) mas o código priorizar PascalCase, a leitura pode falhar se não houver fallback. Porém, o código **tem fallback** (`??`), então não quebrará:

```
statusData.Connected → undefined
statusData.connected → true  ← usa fallback
```

O risco é se a API não retornar nenhum dos dois formatos (ex: snake_case `connected_status`), ambos falham.

**Como corrigir:**

Confirmar empiricamente qual formato a API real usa. Adicionar log ao chamar `/instance/status`:

```typescript
async function getEvoGoStatus(instanceToken: string): Promise<EvoGoInstanceStatus> {
  const response = await fetch(`${EVOGO_API_BASE}/instance/status`, {
    headers: { "apikey": instanceToken },
  });
  
  const json = await response.json();
  console.log('[EvoGo] /instance/status raw response:', JSON.stringify(json.data, null, 2));
  
  const statusData = json.data || {};
  // ... resto do código
}
```

Verificar nos logs Vercel qual formato é retornado e ajustar o código conforme necessário.

### DIV-3 — Campos do `/instance/qr` em PascalCase vs camelCase

**O que é a divergência:**

A documentação local (`api-instances.md`) diz que `GET /instance/qr` retorna:
```json
{
  "qrcode": "iVBORw0KGgoAAAANSUhEUgAABA...",
  "code": "2@12345abcdef,..."
}
```

Campos em **lowercase**: `qrcode`, `code`.

O código em `lib/api/evo-go-client.ts` tenta:
```typescript
const qrCode = (qrData.Qrcode || qrData.qrcode);
const code = (qrData.Code || qrData.code);
```

Prioriza **PascalCase** (`Qrcode`, `Code`) com fallback para lowercase. O comentário diz: `// Note: Evo GO returns "Qrcode" with uppercase Q`.

**Por que pode atrapalhar:**

Mesmo problema que DIV-4 — se a API enviar em camelCase (conforme doc) mas código priorizar PascalCase, pode haver ambiguidade. O fallback protege, mas a divergência mostra que o desenvolvedor **testou na prática e descobriu que a API diverge da documentação**.

**Como corrigir:**

Mesma abordagem de DIV-4 — adicionar log e confirmar empiricamente:

```typescript
async function getEvoGoQRCode(instanceToken: string): Promise<{ qr_code: string; code: string }> {
```

### DIV-5 — Resposta do `POST /instance/create` não documenta campo `id`

**O que é a divergência:**

A documentação local (`api-instances.md`) diz que `POST /instance/create` retorna:
```json
{
  "message": "success",
  "data": {
    "name": "vendas",
    "token": "token-seguro-unico",
    "webhook": "...",
    "status": "created",
    "createdAt": "2025-01-15T10:30:00Z"
  }
}
```

Campos retornados: `name`, `token`, `webhook`, `status`, `createdAt`. Não há menção a `id` ou `qrcode`.

O código em `lib/api/evo-go-client.ts` tenta:
```typescript
if (!responseData.id) {
  throw new EvoGoError(
    "Invalid response from Evo GO — missing instance ID",
    "MALFORMED_RESPONSE",
    500,
  );
}

return {
  instance_id: responseData.id,
  token: responseData.token,
  qr_code: responseData.qrcode || '',
};
```

O código **exige** que a resposta contenha `id`. Se não vier, a requisição falha.

**Por que pode atrapalhar:**

Se a API real não retornar `id` no create (conforme documentação), a função `callEvoGoCreateInstance` lança erro e a criação de instância falha completamente.

**Como corrigir:**

Confirmar empiricamente se a resposta do create retorna `id`. Se não retornar, usar o nome da instância como identificador ou fazer lookup posterior via `GET /instance/all` para obter o `id`:

```typescript
async function callEvoGoCreateInstance(
  apiKey: string,
  params: EvoGoCreateInstanceRequest,
): Promise<EvoGoCreateInstanceResponse> {
  // ... criar instância
  const json = await response.json();
  const responseData = json.data || {};
  
  console.log('[EvoGo] /instance/create raw response:', JSON.stringify(json, null, 2));
  
  // Se não retornar id, fazer lookup
  let instanceId = responseData.id;
  if (!instanceId && responseData.name) {
    // Buscar na lista de instâncias
    const instances = await listEvoGoInstances(apiKey);
    const created = instances.find(i => i.name === responseData.name);
    instanceId = created?.id;
  }
  
  if (!instanceId) {
    throw new EvoGoError(
      "Could not determine instance ID from create response",
      "MALFORMED_RESPONSE",
      500,
    );
  }
  
  return {
    instance_id: instanceId,
    token: responseData.token,
    qr_code: responseData.qrcode || '',
  };
}
```

### DIV-7 — Formato do `subscribe`: UPPERCASE vs dot.notation

**O que é:**

A documentação local é **inconsistente internamente**:

- `events-system.md` mostra: `subscribe: ["MESSAGE", "CONNECTION", "QRCODE"]` (UPPERCASE)
- `api-instances.md` mostra: `subscribe: ["messages.upsert", "connection.update"]` (dot.notation)

O código usa UPPERCASE:

```typescript
subscribe: ['QRCODE', 'CONNECTION', 'MESSAGE']
```

**Por que pode atrapalhar:**

Se a API espera um formato e recebe outro, pode não subscrever aos eventos corretamente. Nenhum evento seria recebido via webhook.

**Como corrigir:**

Verificar qual formato a Evolution GO realmente aceita testando no connect:

```typescript
// Teste com ambos os formatos em sequência
const formats = [
  ['MESSAGE', 'CONNECTION', 'QRCODE'],
  ['messages.upsert', 'connection.update', 'qrcode'],
];

for (const subscribe of formats) {
  const response = await fetch(`${EVOGO_API_BASE}/instance/connect`, {
    method: 'POST',
    headers: { 'apikey': instanceToken, 'Content-Type': 'application/json' },
    body: JSON.stringify({ webhookUrl, subscribe }),
  });
  
  const result = await response.json();
  console.log(`[EvoGo] connect attempt with ${subscribe[0]}:`, result);
  
  if (response.ok) break; // Primeiro que funciona é o correto
}
```

### DIV-8 — "Desconectar" chama logout destrutivo em vez de disconnect suave

**O que é:**

A documentação distingue:
- `POST /instance/disconnect` — pausa a conexão, mantém sessão
- `DELETE /instance/logout` — logout completo, remove sessão

O código em `app/api/settings/evo-go/disconnect/route.ts` chama `callEvoGoLogout()` (DELETE), que é **destrutivo**:

```typescript
// Código do projeto — chama DELETE /logout quando usuário clica em "Desconectar"
const result = await callEvoGoLogout(evolutionInstanceToken);
```

**Por que pode atrapalhar:**

Quando o usuário clica em "desconectar", ele pode esperar poder "conectar de novo" sem re-escanear QR Code. Mas o logout destrutivo remove a sessão, exigindo reconexão completa.

**Como corrigir:**

Decidir se o comportamento é intencional. Se for, está OK (é mais seguro). Se quiser "desconectar suave", usar `POST /instance/disconnect`:

```typescript
// Se quiser desconectar suave (mantém sessão)
export async function callEvoGoDisconnect(instanceToken: string): Promise<void> {
  const response = await fetch(`${EVOGO_API_BASE}/instance/disconnect`, {
    method: "POST",
    headers: {
      "apikey": instanceToken,
      "Content-Type": "application/json",
    },
  });

  if (!response.ok) {
    throw new EvoGoError(
      `Failed to disconnect instance: ${response.statusText}`,
      "API_ERROR",
      response.status,
    );
  }
}

// Depois chamar isso em vez de logout
await callEvoGoDisconnect(evolutionInstanceToken);
```

### DIV-11 — `GET /instance/all` pode não retornar `token`

**O que é:**

A documentação local (`api-instances.md`) mostra que `GET /instance/all` retorna objetos com campos: `id`, `name`, `connected`, `jid`.

O código em `lib/api/evo-go-client.ts` espera que cada instância na lista contenha `token`:

```typescript
export interface EvoGoInstance {
  id: string;
  name: string;
  token: string;  // ← Esperado, mas talvez não venha no GET /all
  connected: boolean;
}
```

**Por que pode atrapalhar:**

Se o `GET /instance/all` não retornar `token`, a função `getOrCreateInstance()` vai tentar reutilizar uma instância existente mas sem token. Isso quebraria o fluxo de QR Code (que precisa do token para chamar `GET /instance/qr`).

**Como corrigir:**

Verificar o que `GET /instance/all` realmente retorna e fazer lookup de token se necessário:

```typescript
async function getOrCreateInstance(
  apiKey: string,
  tenantId: string,
): Promise<EvoGoInstance> {
  const instances = await listEvoGoInstances(apiKey);
  
  if (instances.length > 0) {
    const instance = instances[0];
    
    // Se não tiver token, buscar via GET /instance/info/:id
    if (!instance.token) {
      const info = await getEvoGoInstanceInfo(apiKey, instance.id);
      instance.token = info.token;
    }
    
    return instance;
  }
  
  // Se não existir, criar nova
  return await callEvoGoCreateInstance(apiKey, { ... });
}

async function getEvoGoInstanceInfo(apiKey: string, instanceId: string): Promise<any> {
  const response = await fetch(`${EVOGO_API_BASE}/instance/info/${instanceId}`, {
    headers: { "apikey": apiKey },
  });
  
  const json = await response.json();
  return json.data;
}
```
