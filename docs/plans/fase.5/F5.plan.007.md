# F5.plan.007: Corrigir Mapeamento de Status CONNECTION_UPDATE no Webhook

## Contexto

O webhook Evolution GO envia eventos `CONNECTION_UPDATE` com status `'open'`, mas a tabela `tenants`. No banco de dados em produção, a coluna `tenants.connection_status` possui uma `CHECK CONSTRAINT` que aceita apenas:
  - `'connected'`
  - `'connecting'`
  - `'disconnected'`
  - `'error'`
  - `'pending'`
- O Evolution GO envia status `'open'` (e possivelmente `'close'`, `'connecting'`, `'created'`).
, a tentativa de atualizar o banco falha com erro:
```
violates check constraint "tenants_connection_status_check"
```

**Evidência nos logs (Vercel, 2026-04-22 14:48:53):**
```
[Webhook] CONNECTION_UPDATE database error
{
  tenantId: '4b54f906-6567-462e-a4fa-3e40da1eadfc',
  status: 'open',
  error: 'new row for relation "tenants" violates check constraint "tenants_connection_status_check"',
  timestamp: '2026-04-22T17:48:54.735Z'
}
```

**Predecessores:** Momentos antes (14:48:50), o mesmo webhook com status `'disconnected'` foi processado com sucesso, confirmando que a constraint existe e está funcionando.

---

## Análise do Problema

### Status Evolution GO vs. Banco

| Evo GO Status | DB Value | Descrição |
|---|---|---|
| `open` | `connected` | Conexão estabelecida |
| `connecting` | `connecting` | Em processo de conexão |
| `close` / `closed` | `disconnected` | Desconectado pelo usuário/sistema |
| `created` | `pending` | Instância criada, aguardando QR |
| *fallback* | `error` | Erro desconhecido ou falha |

### Localização do Código Problemático

**Arquivo:** `app/api/webhooks/evo-go/route.ts`

**Função:** `handleConnectionUpdate()`

**Linha problemática:**
```typescript
const status = data?.status || 'disconnected';

const { error } = await supabase
  .from('tenants')
  .update({
    connection_status: status,  // ← status vem direto do payload, sem normalização
    ...
  })
  .eq('id', tenantId);
```

### Por que falha

1. Evolution GO CLI envia `status: 'open'` no payload
2. Webhook handler usa esse valor direto: `connection_status: 'open'`
3. Banco rejeita porque `'open'` não está na lista de valores válidos da constraint
4. Update falha silenciosamente (log do erro, mas webhook retorna 200 OK)

---

## Solução

### 1. Adicionar Mapeamento de Status

No arquivo `app/api/webhooks/evo-go/route.ts`, na função `handleConnectionUpdate()`:

**ANTES:**
```typescript

  supabase: any,
  tenantId: string,
  data?: Record<string, unknown>,
): Promise<void> {
  try {
    const status = data?.status || 'disconnected';
    
    const { error } = await supabase
      .from('tenants')
      .update({
        connection_status: status,
        ...
```

**DEPOIS:**
```typescript
async function handleConnectionUpdate(
  supabase: any,
  tenantId: string,
  data?: Record<string, unknown>,
): Promise<void> {
  try {
    // Mapeamento de status Evolution GO para status do banco
    const statusMap: Record<string, string> = {
      'open': 'connected',
      'closed': 'disconnected',
      'connecting': 'connecting',
    };
    
    const rawStatus = (data?.status as string | undefined) || 'disconnected';
    const status = statusMap[rawStatus] || 'failed';
    
    console.log('[EvoGo] CONNECTION_UPDATE status mapping', {
      rawStatus,
      mappedStatus: status,
      timestamp: new Date().toISOString(),
    });
    
    const { error } = await supabase
      .from('tenants')
      .update({
        connection_status: status,
        ...(status === 'connected' && { updated_at: new Date().toISOString() }),
      })
      .eq('id', tenantId);
```

### 2. Verificar Schema da Tabela

Confirmar que a constraint está correta em `supabase/migrations/`:

```sql
ALTER TABLE tenants
ADD CONSTRAINT tenants_connection_status_check
CHECK (connection_status IN ('connected', 'disconnected', 'connecting', 'failed'));
```

Se a constraint não existir, criar:

```sql
CREATE TABLE IF NOT EXISTS tenants (
  ...
  connection_status VARCHAR(20) DEFAULT 'disconnected' 
    CHECK (connection_status IN ('connected', 'disconnected', 'connecting', 'failed')),
  ...
);
```

### 3. Adicionar Documentação

No arquivo `app/api/webhooks/evo-go/route.ts`, adicionar comentário explicando os mapeamentos:

```typescript
/**
 * Evolution GO Connection Status Mapping
 * 
 * Evolution GO sends connection events with status values:
 * - 'open': WhatsApp connected and ready
 * - 'closed': WhatsApp disconnected
 * - 'connecting': In the process of connecting
 * 
 * These must be mapped to database valid values:
 * - 'connected': Logged in and ready
 * - 'disconnected': Not connected
 * - 'connecting': Connection in progress
 * - 'failed': Error or unknown state
 */
```
