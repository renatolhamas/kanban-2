### 1.2 Query Analysis

**Current Problem:**
The API endpoint `GET /api/conversations` needs to fetch the last message per conversation efficiently:

```sql
-- INEFFICIENT (without proper index)
SELECT 
  c.id,
  c.contact_id,
  c.tenant_id,
  (SELECT content FROM messages 
   WHERE conversation_id = c.id 
   ORDER BY created_at DESC LIMIT 1) AS last_message,
  (SELECT created_at FROM messages 
   WHERE conversation_id = c.id 
   ORDER BY created_at DESC LIMIT 1) AS last_message_at,
  (SELECT sender_type FROM messages 
   WHERE conversation_id = c.id 
   ORDER BY created_at DESC LIMIT 1) AS last_sender_type
FROM conversations c
WHERE c.tenant_id = $1
ORDER BY last_message_at DESC;
```

**Performance Issue:**
- Without index: Full Table Scan on `messages` for each conversation
- With 100+ conversations: Up to 100+ FTS operations = 3000-5000ms
- Target: ≤ 200ms for full result set

**Solution:**
Index supporting the correlated subquery pattern.

### 1.3 Index Strategy

**Decision:** Composite index on `(conversation_id, created_at DESC)`

**Why this index:**
1. Filters by `conversation_id` (WHERE clause)
2. Orders by `created_at DESC` (ORDER BY in subquery)
3. Both columns needed together for efficiency
4. Index-Only Scan possible (forward direction)

**Rationale vs alternatives:**
- ❌ `idx_messages_created` — missing conversation_id filter
- ❌ `idx_messages_conversation` — missing order optimization
- ✅ `idx_messages_conversation_created` — covers both conditions

**Status:** Migration file successfully created in repository

**Content:**
```sql
-- Migration: Add index for last-message-per-conversation queries (Story 5.4)
-- Purpose: Optimize correlated subquery performance for Kanban preview display
-- Target: ≤ 200ms for 100+ conversations per tenant

-- Create composite index
CREATE INDEX IF NOT EXISTS idx_messages_conversation_created 
ON public.messages(conversation_id ASC, created_at DESC)
INCLUDE (sender_type, content)
WHERE deleted_at IS NULL;

-- Note on INCLUDE clause:
-- - Adds sender_type + content to index without affecting sort order
-- - Enables index-only scans for the subquery pattern
-- - Trades index size for query performance (justified for high-read workload)

-- Comment for documentation
COMMENT ON INDEX public.idx_messages_conversation_created IS 
'Composite index for last-message-per-conversation queries in Kanban board.
Supports correlated subqueries: WHERE conversation_id = c.id ORDER BY created_at DESC LIMIT 1
Performance target: ≤ 200ms for 100+ conversations per page.
Story 5.4: Last Message Preview optimization.';
```

**Rollback:**
```sql
DROP INDEX IF EXISTS public.idx_messages_conversation_created;
```

## Notes for @architect

**Architecture Review Points:**

1. **Correlated Subquery vs Window Function:**
   - Story chose correlated subquery for simplicity
   - Index optimizes this pattern well
   - CTE with window function available as fallback

2. **Real-Time Subscription Pattern:**
   - RLS already covers tenant isolation
   - Subscription should filter by tenant_id in WHERE clause
   - Consider message rate limits to prevent subscription thrashing

3. **Future Optimization:**

   - Current index approach is pragmatic for now
