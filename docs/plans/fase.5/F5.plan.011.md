# F5.plan.011 — Correção do Schema: Campo Renomeado Quebrou o Webhook

**Data:** 2026-04-24  
**Severidade:** CRÍTICA  
**Status:** ✅ CONCLUÍDO (Executado via Story 5.4.2)  

---

## O Problema

Alguém, em algum momento, meteu a mão nas migrations e renomeou um campo da tabela `tenants`. O resultado? **O webhook de recebimento de mensagens está completamente cego desde então.** Toda mensagem que chega é silenciosamente descartada. O PO testou, estava funcionando — e agora "misteriosamente" parou. Não é mistério: renomearam a porra da coluna e não atualizaram o código.

---

### ✅ Conclusão do Plano

Este plano de correção crítica foi executado e validado por meio da **[Story 5.4.2 — Schema Correction & RPC Recovery: Webhook & Dashboard Fix](../../stories/5.4.2.story.md)**.

## Evidência: O Código Busca Um Campo Que Não Existe

### O que o código faz

**Arquivo:** `app/api/webhooks/evo-go/route.ts`, linha 76

```typescript
const { data: tenantData } = await supabase
  .from('tenants')
  .select('id')
  .eq('evolution_instance_name', instance)  // ← BUSCA ESSE CAMPO
  .single()
```

O webhook recebe o nome da instância Evolution GO e precisa descobrir qual tenant é dono dela. Para isso, filtra a tabela `tenants` pelo campo `evolution_instance_name`.

### O que o banco tem

**Arquivo:** `docs/db/schema.sql`, linha 23

```sql
CREATE TABLE tenants (
  id uuid DEFAULT gen_random_uuid() PRIMARY KEY,
  ...
  evolution_instance_id text UNIQUE,   -- ← O CAMPO SE CHAMA ASSIM
  ...
);
```

**O campo `evolution_instance_name` não existe.** O campo correto no banco se chama `evolution_instance_id`.

---

## Consequência em Cascata

O Supabase/PostgREST **não retorna erro** quando você filtra por uma coluna inexistente — ele simplesmente retorna zero resultados. Então o webhook:

1. Busca o tenant pelo campo errado → retorna vazio
2. Tenta usar o `tenantId` da query string como fallback
3. Se esse fallback também falhar → `resolvedTenantId` é nulo
4. **Nenhuma conversa é criada. Nenhuma mensagem é gravada. Silêncio total.**

Do ponto de vista externo: o webhook retorna HTTP 200 (sempre), nenhum erro visível, mas absolutamente nada é persistido.

## Segunda Bomba: RPC Ausente do Banco

**Arquivo:** `app/api/conversations/route.ts`, linha 33

```typescript
supabase.rpc('get_conversations_with_last_message', { p_kanban_id: kanbanId })
```

Essa RPC é chamada toda vez que o frontend carrega o board. Ela **não está documentada em `docs/db/functions.md`** (que lista apenas 4 funções, sem ela). Se a função não existir no banco atual, toda tela `/home` retorna erro silencioso e o board aparece vazio.

### Fix 2 — Criar a RPC `get_conversations_with_last_message` no banco

**Confirmado:** a função **não existe** no banco atual (`docs/db/functions.md` lista apenas 3 funções; essa RPC não está entre elas). Precisa ser criada via nova migration:

```sql
CREATE OR REPLACE FUNCTION public.get_conversations_with_last_message(
  p_kanban_id uuid
)
RETURNS TABLE (
  id uuid,
  wa_phone text,
  status text,
  last_message_at timestamptz,
  column_id uuid,
  contact_name text,
  last_message_content text,
  last_sender_type text,
  last_media_url text,
  last_media_type text,
  unread_count int
) AS $$
DECLARE
  v_tenant_id uuid;
BEGIN
  v_tenant_id := ((SELECT auth.jwt()) -> 'app_metadata' ->> 'tenant_id')::uuid;

  RETURN QUERY
  SELECT
    c.id,
    c.wa_phone,
    c.status,
    c.last_message_at,
    c.column_id,
    con.name AS contact_name,
    (SELECT m.content FROM public.messages m
     WHERE m.conversation_id = c.id
     ORDER BY m.created_at DESC LIMIT 1) AS last_message_content,
    (SELECT m.sender_type FROM public.messages m
     WHERE m.conversation_id = c.id
     ORDER BY m.created_at DESC LIMIT 1) AS last_sender_type,
    (SELECT m.media_url FROM public.messages m
     WHERE m.conversation_id = c.id
     ORDER BY m.created_at DESC LIMIT 1) AS last_media_url,
    (SELECT m.media_type FROM public.messages m
     WHERE m.conversation_id = c.id
     ORDER BY m.created_at DESC LIMIT 1) AS last_media_type,
    c.unread_count
  FROM public.conversations c
  LEFT JOIN public.contacts con ON c.contact_id = con.id
  WHERE c.kanban_id = p_kanban_id
    AND c.tenant_id = v_tenant_id
    AND c.status = 'active'
  ORDER BY c.last_message_at DESC NULLS LAST;
END;
$$ LANGUAGE plpgsql STABLE SECURITY DEFINER;
```

## Respostas às Perguntas do PO

### "Cada mensagem sempre vai estar referenciada com uma conversation?"

**Sim — por design e por constraint.** A coluna `messages.conversation_id` é `NOT NULL` com `FK → conversations(id) ON DELETE CASCADE`. É impossível inserir uma mensagem sem uma conversation válida. O webhook só insere em `messages` depois de garantir que tem um `conversationId`:

```typescript
if (!fromMe && conversationId) {  // só insere se conversationId existe
  await supabase.from('messages').insert({ conversation_id: conversationId, ... })
}
```

### "A questão de a conversa estar ativa ou não está dentro de conversations?"

**Sim.** O campo `conversations.status` controla isso com dois valores: `'active'` e `'archived'`. A RPC `get_conversations_with_last_message` já filtra por `status = 'active'` — conversas arquivadas não aparecem no board. O webhook nunca altera o status (novas conversas nascem como `'active'` pelo default do banco).
