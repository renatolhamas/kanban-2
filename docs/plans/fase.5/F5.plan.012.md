# F5.plan.011 — Search Path Hardening em Funções PL/pgSQL

**Status:** 📋 Planejamento  
**Prioridade:** 🟡 MEDIUM (Segurança defensiva)  
**Impacto:** Baixo (zero breaking changes)  
**Esforço:** ⏱️ 15-30 min

---

## 🔴 Problema

Duas funções PL/pgSQL no banco **não fixam `search_path`**, permitindo potencial ataque de manipulação de search_path:

```
✗ upsert_contact()
✗ get_conversations_with_last_message()
```

**Risco:** Um atacante poderia criar tabelas fake em outro schema e fazer a função usar essas tabelas em vez das legítimas.

### Exemplo de Ataque

```sql
-- Passo 1: Atacante cria tabela fake
CREATE TABLE attacker.contacts AS SELECT * FROM public.contacts;

-- Passo 2: Modifica search_path (se tiver permissão)
SET search_path = 'attacker, public';

-- Passo 3: Quando upsert_contact() executa:
INSERT INTO contacts (...)  -- ← vai para attacker.contacts!
```

### Por Que Acontece?

Quando uma função **não especifica search_path**, ela herda do **contexto do chamador**. Se um usuário malicioso mudar seu search_path antes de chamar a função, a função segue esse search_path.

---

## ✅ O Que Fazer

Adicionar **`SET search_path = 'public'`** na definição de cada função para **forçar uso do schema correto** independentemente do contexto do chamador.

### Funções a Corrigir

| Função | Tabelas Usadas | Ação |
|--------|----------------|------|
| `upsert_contact` | contacts | ADD `SET search_path = 'public'` |
| `get_conversations_with_last_message` | conversations, messages, contacts | ADD `SET search_path = 'public'` |

---

## 📋 Impacto Zero

✅ **Não quebra nada:**
- Funções continuam funcionando exatamente igual
- RLS policies não são afetadas
- Queries de aplicação não precisam mudar
- Sem mudança de lógica, apenas hardening de segurança

✅ **Aplicável imediatamente:**
- Fix é isolado (cada função é independente)
- Pode ser aplicado em ambiente de produção com zero risco
- Sem migrações complexas

#### 3a. Função: `upsert_contact`

**ANTES:**
```sql
CREATE FUNCTION upsert_contact(
  p_tenant_id uuid,
  p_phone text,
  p_name text,
  p_wa_name text,
  p_is_group boolean

RETURNS uuid
LANGUAGE plpgsql
AS $$
DECLARE
  v_id uuid;
BEGIN
  INSERT INTO contacts (...)
  ...
END;
$$;
```

**DEPOIS:**
```sql
CREATE OR REPLACE FUNCTION upsert_contact(
  p_tenant_id uuid,
  p_phone text,
  p_name text,
  p_wa_name text,
  p_is_group boolean
)
RETURNS uuid
LANGUAGE plpgsql
SET search_path = 'public'  -- ← ADICIONA ISTO
AS $$
DECLARE
  v_id uuid;

  ...
END;
$$;
```

## 🎯 Padrão Para Futuras Funções

**Sempre incluir `SET search_path` ao criar funções:**

```sql
CREATE FUNCTION minha_funcao(...)
  LANGUAGE plpgsql
  SET search_path = 'public'  -- ← SEMPRE ADICIONAR
AS $$
  ...
$$;
```

**Por que?**
- ✅ Previne ataques de manipulação de search_path
- ✅ Torna função determinística (não depende do contexto do chamador)
- ✅ Melhor performance (sem overhead de lookup de schema)
- ✅ Padrão recomendado por PostgreSQL docs

## ⚠️ Cuidados

### ❌ O Que NÃO Fazer

1. **Não modifique outras funções ao mesmo tempo**
   - Risco: Se algo quebrar, não sabemos qual função causou
   - ✅ Fazer uma por vez

2. **Não pule testes**
   - Risco: Fix silencioso que quebra em produção
   - ✅ Rodar `*smoke-test` após cada mudança

3. **Não use DROP + CREATE em produção**
   - Risco: Downtime se função está em uso
   - ✅ Usar `CREATE OR REPLACE`

4. **Não confunda com aplicação code**
   - Search path em banco ≠ search path em aplicação Node.js
   - ✅ Este fix é APENAS banco de dados

### ✅ O Que Fazer

1. **Aplicar uma função por migration**
   ```sql
   -- migration: 20260424_add_search_path_upsert_contact.sql
   CREATE OR REPLACE FUNCTION upsert_contact(...) ...
   
   -- migration: 20260424_add_search_path_get_conversations.sql
   CREATE OR REPLACE FUNCTION get_conversations_with_last_message(...) ...
   ```

2. **Testar após cada migration**
   ```bash
   *dry-run supabase/migrations/20260424_add_search_path_upsert_contact.sql
   *apply-migration supabase/migrations/20260424_add_search_path_upsert_contact.sql
   ```

3. **Documentar no commit**
   ```
   fix: add search_path hardening to upsert_contact function
   
   - Set search_path = 'public' to prevent search_path injection attacks
   - Zero breaking changes, security enhancement only
   - Closes Supabase linter advisory
   ```
