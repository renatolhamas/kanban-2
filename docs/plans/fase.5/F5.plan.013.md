# F5.plan.013 — Correção da Lógica de Vinculação de Mensagens com Conversas

**Data:** 2026-04-24  
**Severidade:** ALTA  
**Status:** Pendente implementação

---

## O Problema

O webhook atual busca uma conversa existente filtrando por `(tenant_id, contact_id, kanban_id)`. Essa lógica está **errada** — a vinculação de uma mensagem deve ser com a **conversa ativa do contato**, não com o kanban.

O resultado prático: quando múltiplas mensagens chegam de um mesmo contato, o código cria uma nova conversa para cada mensagem ao invés de acumular todas na mesma conversa.

---

## Evidência: A Pergunta Errada no Código

**Arquivo:** `app/api/webhooks/evo-go/route.ts`, linhas 376–382

```typescript
const { data: existingConv } = await supabase
  .from('conversations')
  .select('id')
  .eq('tenant_id', tenantId)
  .eq('contact_id', contact.id)
  .eq('kanban_id', kanban.id)   // ← ERRADO: limita pelo kanban
  .maybeSingle();
```

O código pergunta:
> "Existe uma conversa desse contato **nesse kanban**?"

A pergunta correta é:
> "Existe uma conversa **ativa** desse contato?"

---

## Por Que `kanban_id` Não Deve Estar Aqui

A conversa **já possui** `kanban_id` associado desde quando foi criada. Quando uma nova mensagem chega via webhook, o objetivo é:

1. Encontrar a **conversa ativa** daquele contato
2. Vincular a mensagem a essa conversa

O kanban é um atributo da conversa — não um critério de busca para vinculação de mensagens.

---

## A Correção

### Mudança na busca (linha 376–382)

**De:**
```typescript
.eq('tenant_id', tenantId)
.eq('contact_id', contact.id)
.eq('kanban_id', kanban.id)
```

**Para:**
```typescript
.eq('tenant_id', tenantId)
.eq('contact_id', contact.id)
.eq('status', 'active')
```

### Criação de conversa nova (sem mudança)

Quando não existir conversa ativa, cria uma nova normalmente — com `kanban_id`, `column_id`, `status = 'active'` (default do banco).

---

## Lógica Correta Após a Correção

```
Chegou uma mensagem via webhook:
  1. Upsert do contato (idempotente)
  2. Busca conversa ATIVA do contato para esse tenant
     - se SIM → vincula mensagem à conversa ativa
     - se NÃO → cria conversa nova (ativa), vincula mensagem
```

---

## Impacto

| Componente | Mudança |
|-----------|---------|
| `app/api/webhooks/evo-go/route.ts` | Trocar filtro `kanban_id` por `status = 'active'` na busca |
| Banco de dados | Avaliar se constraint UNIQUE precisa ser atualizada |
| Comportamento | Mensagens do mesmo contato acumulam em 1 conversa ativa |

---

## O Que NÃO Muda

- A criação de conversa nova continua usando `kanban_id` e `column_id`
- A estrutura da tabela `conversations` não muda
- O fluxo de inserção de mensagens não muda

---

## Arquivos Impactados

| Arquivo | Ação |
|---------|------|
| `app/api/webhooks/evo-go/route.ts` (linha 376–382) | Substituir filtro `kanban_id` por `status = 'active'` |
