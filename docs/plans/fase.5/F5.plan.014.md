## 📋 Executivo

A função RPC `get_conversations_with_last_message` foi criada em 2026-04-23, teve problemas, foi removida e restaurada em 2026-04-24. Está **ATIVA e USADA em produção** no endpoint `/api/conversations`, mas com **3 vulnerabilidades críticas** e **1 problema de performance**.

## 🚨 Vulnerabilidades & Problemas Identificados

### 1. 🔴 CRÍTICO — Segurança: SECURITY DEFINER + Acesso Público

**Localização:** `functions.md` - Security Advisors (18 findings)

**Status:**
```
❌ anon_security_definer_function_executable
   → "Public can execute get_conversations_with_last_message as SECURITY DEFINER"
   
❌ authenticated_security_definer_function_executable
   → "Signed-in users can execute get_conversations_with_last_message as SECURITY DEFINER"
```

**O Problema:**
- Função é `SECURITY DEFINER` → executa com postgres role (superuser)
- Está exposta como RPC pública → `anon` + `authenticated` roles podem chamar
- **NÃO há validação que kanban_id pertence ao tenant do user**
- Risco: Usuário autenticado em tenant A pode listar conversas de kanban em tenant B se conhecer o ID

**Cenário de Ataque:**
```sql
-- Usuário (tenant A) consegue fazer:
SELECT * FROM get_conversations_with_last_message(
  p_kanban_id := 'UUID-from-tenant-B'  -- Kanban de outro tenant!
)
-- RPC valida apenas: tenant_id do JWT
-- Mas NÃO valida: kanban_id pertence ao tenant_id do JWT
-- → Data leak: vê conversas de outro tenant
```

**Mitigation:**
```sql
-- Opção 1: REVOKE EXECUTE FROM public (menor risco)
REVOKE EXECUTE ON FUNCTION public.get_conversations_with_last_message(p_kanban_id uuid) 
  FROM public;
GRANT EXECUTE ON FUNCTION public.get_conversations_with_last_message(p_kanban_id uuid) 
  TO authenticated;

-- Opção 2: Switch to SECURITY INVOKER (melhor, mas precisa validação RLS)
ALTER FUNCTION public.get_conversations_with_last_message(p_kanban_id uuid) 
  SECURITY INVOKER;
```

### 2. 🟠 ALTO — Segurança: Validação de Kanban Ownership Falta na RPC

**Status:**
- ✅ Endpoint `/api/conversations` valida `tenant_id` via JWT
- ❌ Endpoint NÃO valida que `kanban_id` pertence ao `tenant_id`
- ❌ RPC não faz essa validação

**Code Analysis:**

```typescript
// app/api/conversations/route.ts (linhas 26-35)
const tenantId = user.app_metadata?.tenant_id
if (!tenantId) {
  return NextResponse.json({ error: 'Tenant context missing' }, { status: 403 })
}

// 🚨 Missing: Validar que kanban_id pertence a tenantId
const { data, error } = await supabase.rpc('get_conversations_with_last_message', {
  p_kanban_id: kanbanId  // ← Sem validação!
})
```

**Fix:**

```typescript
// ANTES: Chamar RPC sem validação
const { data, error } = await supabase.rpc(...)

// DEPOIS: Validar kanban_id pertence ao tenant
const { data: kanban, error: kbError } = await supabase
  .from('kanbans')
  .select('id')
  .eq('id', kanbanId)
  .eq('tenant_id', tenantId)
  .single()

if (kbError || !kanban) {
  return NextResponse.json({ error: 'Kanban not found or unauthorized' }, { status: 403 })
}

// Agora seguro: chamar RPC
const { data, error } = await supabase.rpc(...)
```

### 3. ⚠️ PERFORMANCE — N+1 Queries (4 subqueries correlated)

**Status:**
```
❌ unindexed_foreign_keys (3 findings)
   → automatic_messages.scheduled_kanban_id
   → conversations.column_id
   → conversations.contact_id
   
❌ unused_index (11 findings)
   → Multiple indexes on failed_registrations, tenants, contacts, conversations
   → Advisor: "queries never used this index"
```

**RPC Function Detail (funções.md):**

```sql
SELECT
  c.id,
  c.wa_phone,
  c.status,
  c.last_message_at,
  c.column_id,
  con.name AS contact_name,
  (SELECT m.content FROM messages m         -- Subquery 1: last message content
   WHERE m.conversation_id = c.id
   ORDER BY m.created_at DESC LIMIT 1),
  (SELECT m.sender_type FROM messages m     -- Subquery 2: sender type
   ...),
  (SELECT m.media_url FROM messages m       -- Subquery 3: media URL
   ...),
  (SELECT m.media_type FROM messages m      -- Subquery 4: media type
   ...),
  c.unread_count
FROM conversations c
LEFT JOIN contacts con ON c.contact_id = con.id
WHERE c.kanban_id = p_kanban_id
  AND c.tenant_id = v_tenant_id
  AND c.status = 'active'
ORDER BY c.last_message_at DESC NULLS LAST;
```

**Impact:**
- **1 conversation** → 1 SELECT + 4 subqueries = 5 queries
- **100 conversations** → 1 + 400 subqueries = **401 database round trips** 😱
- Latency: ~200-400ms for typical load (assuming 50ms per query)

**Root Cause:** Denormalization sem índice composite on `messages(conversation_id, created_at DESC)`

**Current Index (existe mas pode não ser usado):**
```sql
CREATE INDEX idx_messages_conversation_created 
  ON public.messages (conversation_id, created_at DESC);
```

### FASE 2: Otimização de Performance (2-4 horas, próxima sprint)

**Objetivo:** Remover N+1 queries, melhorar latency de 300ms → 30ms.

#### 2.1 Option A: LATERAL JOIN (Sem schema change)

**Vantagem:** Deploy imediato, sem migração  
**Desvantagem:** Ainda precisa de join com messages table

**Implementação:**

```sql
-- Recriar RPC com LATERAL JOIN em vez de subqueries
CREATE OR REPLACE FUNCTION public.get_conversations_with_last_message(p_kanban_id uuid)
RETURNS TABLE (
  id uuid,
  wa_phone text,
  status text,
  last_message_at timestamptz,
  column_id uuid,
  contact_name text,
  last_message_content text,
  last_sender_type text,
  last_media_url text,
  last_media_type text,
  unread_count integer
) AS $$
DECLARE
  v_tenant_id uuid;
BEGIN
  v_tenant_id := ((SELECT auth.jwt()) -> 'app_metadata' ->> 'tenant_id')::uuid;

  RETURN QUERY
  SELECT
    c.id,
    c.wa_phone,
    c.status,
    c.last_message_at,
    c.column_id,
    con.name,
    m.content,
    m.sender_type,
    m.media_url,
    m.media_type,
    c.unread_count
  FROM public.conversations c
  LEFT JOIN public.contacts con ON c.contact_id = con.id
  LEFT JOIN LATERAL (
    SELECT content, sender_type, media_url, media_type
    FROM public.messages
    WHERE conversation_id = c.id
    ORDER BY created_at DESC
    LIMIT 1
  ) m ON true
  WHERE c.kanban_id = p_kanban_id
    AND c.tenant_id = v_tenant_id
    AND c.status = 'active'
  ORDER BY c.last_message_at DESC NULLS LAST;
END;
$$ LANGUAGE plpgsql SECURITY DEFINER;
```

**Ganho:** ~80% menos queries (5 queries → 1 query com lateral join)

#### 2.2 Option B: Denormalization (Melhor, mas requer trigger)

**Vantagem:** Melhor performance (~10x), 1 query puro sem joins  
**Desvantagem:** Schema change, requer manutenção via trigger

**Implementação:**

1. **Adicionar colunas em conversations:**

```sql
ALTER TABLE public.conversations ADD COLUMN IF NOT EXISTS (
  last_message_content text,
  last_sender_type text,
  last_media_url text,
  last_media_type text,
  last_message_id uuid
);

-- Index para find last message rápido
CREATE INDEX IF NOT EXISTS idx_messages_conversation_latest
  ON public.messages(conversation_id, created_at DESC)
  WHERE status != 'deleted'; -- Optional: exclude deleted
```

2. **Criar trigger para manter denormalization:**

```sql
CREATE OR REPLACE FUNCTION update_conversation_last_message()
RETURNS TRIGGER AS $$
BEGIN
  UPDATE public.conversations
  SET
    last_message_content = (
      SELECT content FROM messages 
      WHERE conversation_id = NEW.conversation_id 
      ORDER BY created_at DESC LIMIT 1
    ),
    last_sender_type = (
      SELECT sender_type FROM messages 
      WHERE conversation_id = NEW.conversation_id 
      ORDER BY created_at DESC LIMIT 1
    ),
    last_media_url = (
      SELECT media_url FROM messages 
      WHERE conversation_id = NEW.conversation_id 
      ORDER BY created_at DESC LIMIT 1
    ),
    last_media_type = (
      SELECT media_type FROM messages 
      WHERE conversation_id = NEW.conversation_id 
      ORDER BY created_at DESC LIMIT 1
    )
  WHERE id = NEW.conversation_id;
  
  RETURN NEW;
END;
$$ LANGUAGE plpgsql;

CREATE TRIGGER trg_update_conversation_last_message
  AFTER INSERT ON public.messages
  FOR EACH ROW
  EXECUTE FUNCTION update_conversation_last_message();
```

3. **Simplificar RPC:**

```sql
CREATE OR REPLACE FUNCTION public.get_conversations_with_last_message(p_kanban_id uuid)
RETURNS TABLE (...) AS $$
DECLARE v_tenant_id uuid;
BEGIN
  v_tenant_id := ((SELECT auth.jwt()) -> 'app_metadata' ->> 'tenant_id')::uuid;

  RETURN QUERY
  SELECT c.* FROM public.conversations c
  LEFT JOIN public.contacts con ON c.contact_id = con.id
  WHERE c.kanban_id = p_kanban_id
    AND c.tenant_id = v_tenant_id
    AND c.status = 'active'
  ORDER BY c.last_message_at DESC NULLS LAST;
END;
$$ LANGUAGE plpgsql SECURITY DEFINER;
```

**Ganho:** 1 query, sub-10ms latency, 0 subqueries.

### Trigger para Denormalization (FASE 2, Se Opção B)

**IMPORTANTE:** Trigger deve ser **idempotent** (safe to run multiple times)

```sql
-- Backfill existing conversations com última mensagem
UPDATE conversations c
SET
  last_message_content = m.content,
  last_sender_type = m.sender_type,
  last_media_url = m.media_url,
  last_media_type = m.media_type
FROM (
  SELECT 
    conversation_id,
    content, sender_type, media_url, media_type,
    ROW_NUMBER() OVER (PARTITION BY conversation_id ORDER BY created_at DESC) as rn
  FROM messages
) m
WHERE m.rn = 1 AND c.id = m.conversation_id;
```
