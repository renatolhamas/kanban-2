## Decisões Consolidadas

### 🔴 CRÍTICO 1 — Tenant Resolution

**Decisão:** `payload.instance` → lookup no banco → fallback de emergência para URL param

**Padrão de nomenclatura da instância (confirmado em `lib/api/evo-go-client.ts:325`):**
O nome da instância é gerado deterministicamente pela própria aplicação:
```typescript
const instanceName = `kanban-instance-${tenantId.substring(0, 8)}`;
```

O campo no payload é `instance` na raiz (não `instanceId`). Exemplo real:
```json
{
  "event": "MESSAGE",
  "instance": "kanban-instance-a1b2c3d4",  // ← padrão: kanban-instance-{8 chars do tenantId}
  "data": { ... }
}
```

**Lógica de resolução:**
```
1. Ler payload.instance  →  ex: "kanban-instance-a1b2c3d4"
2. SELECT tenant_id FROM tenants WHERE evolution_instance_id = ?
3. Se encontrado → seguir normalmente
4. Se NÃO encontrado → LOG WARNING ("instância sem tenant mapeado") + tentar ?tenantId= como emergência
5. Se ambos ausentes → log error + return (webhook retorna 200)
```

**Observação importante:** Como a instância é sempre criada por `getOrCreateInstance()` e
`evolution_instance_id` é salvo no banco nesse momento, o lookup **vai sempre encontrar**
o tenant em condições normais. O fallback para `?tenantId=` é um safety net para
inconsistências de dados (bug, registro deletado), não um fluxo alternativo legítimo.

### 🔴 CRÍTICO 2 — `name` vs `wa_name`: Upsert parcial (atômico)

**Decisão:** SQL `INSERT ... ON CONFLICT DO UPDATE SET wa_name=...` (sem tocar `name`)

**Implementação via `supabase.rpc()`:**

```sql
CREATE OR REPLACE FUNCTION upsert_contact(
  p_tenant_id uuid,
  p_phone text,
  p_name text,
  p_wa_name text,
  p_is_group boolean
) RETURNS uuid AS $$
DECLARE v_id uuid;
BEGIN
  INSERT INTO contacts (tenant_id, phone, name, wa_name, is_group, updated_at)
  VALUES (p_tenant_id, p_phone, p_name, p_wa_name, p_is_group, NOW())
  ON CONFLICT (phone, tenant_id)
  DO UPDATE SET
    wa_name    = EXCLUDED.wa_name,
    is_group   = EXCLUDED.is_group,
    updated_at = NOW()
    -- name NOT updated: preserva edições manuais
  RETURNING id INTO v_id;
  RETURN v_id;
END;
$$ LANGUAGE plpgsql SECURITY DEFINER;
```

**TypeScript:**
```typescript
const { data: contactId, error } = await supabase
  .rpc('upsert_contact', {
    p_tenant_id: tenantId,
    p_phone:     contactInfo.waPhone,
    p_name:      contactInfo.name,       // INSERT apenas
    p_wa_name:   contactInfo.waName,     // sempre atualizado
    p_is_group:  contactInfo.isGroup,
  });
```

### 🔴 CRÍTICO 5 — Contratos de `parseJid` e `normalizePhone`

**Decisão:** Tabela completa de contratos input/output

**`parseJid(jid: string): { phone: string; isGroup: boolean } | null`**

| Input | Output |
|-------|--------|
| `"5511987654321:1@s.whatsapp.net"` | `{ phone: "5511987654321", isGroup: false }` |
| `"5511987654321@s.whatsapp.net"` | `{ phone: "5511987654321", isGroup: false }` |
| `"120363XXXXXXXXXX@g.us"` | `{ phone: "120363XXXXXXXXXX", isGroup: true }` |
| `""` | `null` |
| `null` / `undefined` | `null` |

**Regras:**
1. Remover sufixo multi-device (tudo após `:` antes de `@`)
2. Remover domínio (tudo após `@`)
3. Se termina em `@g.us` → `isGroup = true`

**`normalizePhone(phone: string): string | null`**

| Input | Output |
|-------|--------|
| `"5511987654321"` | `"+5511987654321"` |
| `"+5511987654321"` | `"+5511987654321"` (idempotente) |
| `""` | `null` |
| `null` / `undefined` | `null` |

**Regras:**
1. Se começa com `+` → retornar como está
2. Senão → adicionar `+` como prefixo
3. String vazia ou null → retornar `null`

**Escopo de entrada:** Esta função recebe exclusivamente números extraídos de JIDs do
WhatsApp via `parseJid`. JIDs do WhatsApp **sempre incluem o código do país** — ex:
`5511987654321@s.whatsapp.net` (com `55` do Brasil). Portanto, o input nunca chegará
com zero inicial (`011...`) nem sem código de país. A regra de "apenas adicionar `+`"
é suficiente para todos os casos reais de webhook.

Casos com formato fora do padrão (ex: entrada manual pela UI) são tratados pela camada
de validação da UI, não por esta função.
