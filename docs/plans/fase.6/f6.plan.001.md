# F6.PLAN.001 — Relatório Técnico: Auditoria da Estrutura Híbrida (lib/ vs src/lib/)

## 1. Visão Geral da Situação
A aplicação encontra-se atualmente em um estado de "arquitetura de transição". Existe uma pasta `lib/` na raiz do projeto e uma pasta `src/lib/` dentro do diretório de fontes. Esta configuração é considerada um débito técnico de alta prioridade, pois cria ambiguidades para o compilador do TypeScript, para o Next.js e, especialmente, para as ferramentas de teste (Vitest).

### 1.1. O Conflito de Resolução (Shadowing) e Não-Determinismo
No `tsconfig.json`, o alias `@/*` está configurado para olhar primeiro em `src/` e depois na raiz. Isso cria um problema de **não-determinismo**: o comportamento da aplicação pode mudar dependendo de como o bundler (Webpack/Turbo) ou o test runner (Vitest) resolve esses caminhos em diferentes sistemas operacionais ou ambientes (Vercel vs Local).

### 1.2. Divergência de Lógica (O Risco Invisível)
Identifiquei que arquivos com o mesmo nome possuem implementações diferentes. 
*   **Exemplo Crítico (`kanban.ts`):** A versão em `lib/` (raiz) possui tratamento de erros robusto (`try/catch`), logs detalhados e rollback manual de banco de dados. A versão em `src/lib/` é uma implementação simplificada sem essas proteções. Se o Next.js estiver usando a versão de `src`, o sistema está operando com **baixa tolerância a falhas** sem o conhecimento da equipe.

---

## 2. Malefícios de Manter a Divisão

1.  **Fragilidade nos Testes:** Ferramentas como o Vitest precisam de configurações complexas de aliases. Qualquer erro impede a execução dos testes unitários (como visto na Story 5.4.3).
2.  **Inconsistência de Padrões e Observabilidade:** A pasta `lib/` da raiz contém lógica de infraestrutura com logs vitais para debugging. A pasta `src/lib/` foca em UI e helpers leves. Ao manter ambos, perdemos a visão unificada de logs e tratamento de erros.
3.  **Dificuldade de Manutenção e Bundle Size:** Risco de importar ambas as versões acidentalmente, aumentando o tamanho do bundle final e criando estados inconsistentes na memória da aplicação.
4.  **Risco de Falsos Positivos (QA Nightmare):** Podemos estar testando exaustivamente a versão da raiz (`lib/`) enquanto o Next.js ignora esse arquivo e usa a versão simplificada em `src/lib/`.
5.  **A Armadilha do Auto-import (DX):** O VS Code frequentemente sugere imports da raiz (`../../lib/...`) em vez do alias `@/lib/...`.
6.  **Drift de Ambiente e Case-Sensitivity:** Risco de o build passar em Windows e quebrar em Linux (Vercel) devido a maiúsculas/minúsculas.
7.  **Inconsistência Transacional e Registros Órfãos:** Se a lógica robusta de rollback da raiz for ignorada, falhas na criação de objetos complexos (ex: Kanban) podem deixar "lixo" no banco (quadros sem colunas), travando a interface do usuário.
8.  **Conflito de Clientes Supabase (Segurança):** Se `lib/supabase/` e `src/lib/supabase/` inicializarem o cliente com chaves ou configurações de cache diferentes, corremos o risco de **bypass de RLS** (segurança) ou **exaustão de conexões** com o banco de dados.

## 4. Plano de Ação: "The Great Migration"

Para evitar que a aplicação pare de funcionar, a migração deve ser feita em ondas:

### Onda 1: Consolidação de Utilitários Únicos (Risco Baixo)
Mover arquivos que não possuem duplicatas e que são pouco referenciados (ex: `email.ts`, `link-validation.ts`, `rate-limit.ts`).
*   **Como:** Mover arquivo -> Rodar `npm run typecheck` -> Corrigir imports se necessário.

### Onda 2: Infraestrutura Crítica (Risco Alto)
Mover a pasta `api/` e os arquivos de `auth/jwt/supabase`.
*   **Como:** Mover para `src/lib/` -> Ajustar aliases no `vitest.config.ts` -> Rodar suíte completa de testes.

### Onda 3: Resolução de Conflitos (Risco Crítico)
Comparar `lib/kanban.ts` vs `src/lib/kanban.ts` e `lib/utils.ts` vs `src/lib/utils.ts`.
*   **Como:** Identificar funções que só existem na raiz -> Copiar para a versão de `src` -> Deletar o arquivo da raiz.

## 5. Matriz de Riscos e Impactos

| Risco | Impacto | Mitigação |
| :--- | :--- | :--- |
| **Quebra de Imports** | Aplicação não compila ou testes falham. | Uso de `npm run typecheck` constante e aliases globais no `tsconfig`. |
| **Regressão de Lógica** | Perda de funcionalidades de segurança e rollback (ex: `kanban.ts`). | **AUDITORIA OBRIGATÓRIA**: Copiar a lógica robusta da raiz para `src` antes de deletar. |
| **Silent Failures** | O sistema parece funcionar mas ignora erros de banco de dados. | Validar logs de erro após a migração para garantir que o "motor" continua reportando falhas. |
| **Falsos Positivos** | Testes passam em arquivos que não estão em uso. | Verificação manual de cobertura (`coverage report`) para garantir que os arquivos em `src/` estão sendo exercitados. |
| **Falha no Build (CI/CD)** | Deployment bloqueado ou inconsistente com o ambiente local. | Validar localmente com `npm run build` e limpar cache do Next.js/Vitest. |

## 6. Checklist de Validação de Integridade (Pós-Ondas)
Para garantir que a "merda" não aconteça após movermos os arquivos:

1.  [ ] **Smoke Test de API:** Disparar webhooks manuais e verificar se a gravação no banco segue o fluxo robusto (logs presentes).
2.  [ ] **Verificação de Coverage:** Rodar `npm run coverage` e confirmar que NENHUM arquivo da pasta raiz `lib/` está sendo referenciado nos testes.
3.  [ ] **Protocolo de Auditoria de Dados (Dara):** Verificar via SQL se não existem registros órfãos (ex: `kanban_boards` sem colunas associadas) após as operações de teste.
4.  [ ] **Protocolo de Limpeza de Cache (Obrigatório):**
    *   `rm -rf .next` (Cache do Next.js)
    *   `rm -rf node_modules/.vitest` (Cache do Vitest)
    *   `npm run dev` (Reinício limpo)
5.  [ ] **Typecheck Global:** O comando `npm run typecheck` deve retornar zero erros em todo o projeto.
6.  [ ] **Build de Validação:** Executar `npm run build` localmente para garantir que o rastreamento de artefatos não foi quebrado.

## 8. Diretrizes de Desenvolvimento para a Migração
Como desenvolvedores, seguiremos estas regras para garantir que o histórico do Git seja preservado e o código fique limpo:

1.  **Preferência por Alias:** Nunca usar `../../lib` ou `../../src/lib`. Sempre usar `@/lib`.
2.  **Merge de Lógica:** Ao encontrar funções em `lib/kanban.ts` que não estão em `src/lib/kanban.ts`, faremos o porting manual preservando os comentários e os logs de erro.
3.  **Deleção Atômica:** Só deletar o arquivo da raiz após confirmar via `grep` que não existem referências diretas (`import ... from 'lib/...'`) em nenhum lugar do projeto.
4.  **Preservação de Histórico:** Usar `git mv` sempre que possível para que o Git entenda que o arquivo mudou de lugar e não que foi deletado e recriado.

## 9. Conclusão
A situação atual é sustentável a curto prazo, mas é a causa raiz da instabilidade recente nas ferramentas de teste e da confusão no desenvolvimento diário. A recomendação é iniciar a **Onda 3** imediatamente para os arquivos `kanban.ts` e `utils.ts`, pois são os que geram maior confusão por estarem "sombreados" (shadowed) pela estrutura do Next.js.
