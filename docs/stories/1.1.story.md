# Story 1.1: Database Schema Creation

## Story Statement

As a developer, I need to create the complete PostgreSQL schema so that the application has a proper data model to support multi-tenancy, authentication, and conversation management.

## Description

Create the foundational PostgreSQL schema on Supabase Cloud with all required tables, relationships, indexes, and Row Level Security (RLS) policies. This schema supports multi-tenant architecture where each tenant has isolated data through RLS enforcement.

## Acceptance Criteria

- [ ] `tenants` table created with id, name, subscription_status, evolution_instance_id, connection_status, created_at, updated_at
- [ ] `users` table created with id, tenant_id (FK), email, role, name, password_hash, created_at, updated_at
- [ ] `kanbans` table created with id, tenant_id (FK), name, is_main, order_position, created_at, updated_at
- [ ] `columns` table created with id, kanban_id (FK), name, order_position, created_at, updated_at
- [ ] `contacts` table created with id, tenant_id (FK), name, phone (E.164 format), created_at, updated_at
- [ ] `conversations` table created with id, tenant_id (FK), contact_id (FK), kanban_id (FK), column_id (FK), wa_phone, status, last_message_at, created_at, updated_at
- [ ] `messages` table created with id, conversation_id (FK), sender_type, content, media_url, media_type, created_at
- [ ] `automatic_messages` table created with id, tenant_id (FK), name, message, scheduled_interval_minutes, scheduled_kanban_id (FK), created_at, updated_at
- [ ] All foreign keys properly reference parent tables with correct ON DELETE behavior
- [ ] Indexes created on: tenant_id, status, phone, conversation_id, created dates per performance requirements
- [ ] RLS enabled on all tenant-scoped tables (tenants, users, kanbans, columns, contacts, conversations, messages, automatic_messages)
- [ ] RLS policies implemented: auth.get_tenant_id() function, tenant isolation policies for SELECT/UPDATE/DELETE
- [ ] Unique constraints on: users(email, tenant_id), contacts(phone, tenant_id), kanbans(tenant_id, is_main) WHERE is_main = TRUE
- [ ] Schema documented in `/docs/architecture/schema.md` with table descriptions and relationships
- [ ] Migration file created and tested in dev environment (reversible)
- [ ] No schema errors on `SELECT pg_catalog.pg_get_ddl('t'::regtype, 'table_name'::regclass)` for each table

## Dev Notes

### Architecture Context

**Database Choice:**

- PostgreSQL via Supabase Cloud (native RLS support, multi-tenancy first-class citizen)
- [Source: architecture/1-tech-stack-strategic-decisions.md#backend]

**Multi-Tenancy Pattern:**

- Row-level security (RLS) for data isolation
- Every tenant-scoped table has `tenant_id` column
- JWT includes `tenant_id` claim extracted by `auth.get_tenant_id()` function
- [Source: architecture/7-security-architecture.md#rls-enforcement]

**RLS Implementation Details:**

- Extract tenant_id from JWT: `SELECT (auth.jwt()->>'tenant_id')::UUID`
- Policy pattern: `WHERE tenant_id = auth.get_tenant_id()`
- All SELECT, UPDATE, DELETE operations governed by RLS
- [Source: architecture/3-database-schema-complete-reference.md#row-level-security]

**Authentication Context:**

- JWT payload includes: sub (user UUID), tenant_id, email, role, iat, exp (1 hour)
- httpOnly cookies store JWT in production (secure flag enabled)
- [Source: architecture/7-security-architecture.md#authentication-flow]

### Table Relationships (ER Model)

```
tenants (root)
  ├─ users (owned by tenant)
  ├─ contacts (owned by tenant)
  ├─ conversations (links contact to kanban column)
  │   ├─ kanbans (owned by tenant)
  │   │   └─ columns (board stages)
  │   └─ messages (conversation history)
  └─ automatic_messages (message templates)
```

### Index Strategy

**Performance-Critical Indexes (from architecture):**

- `idx_conversations_tenant_status` — Filter active conversations: `WHERE tenant_id = ? AND status = 'active'`
- `idx_messages_conversation_created` — Load message history DESC: `WHERE conversation_id = ? ORDER BY created_at DESC`
- `idx_contacts_phone_tenant` — Prevent duplicates: `WHERE phone = ? AND tenant_id = ?`
- `idx_kanbans_main_tenant` — Find main kanban: `WHERE tenant_id = ? AND is_main = TRUE`
- `idx_conversations_kanban_column` — Board render: `WHERE kanban_id = ? AND column_id = ?`
- `idx_users_email_tenant` — Unique email per tenant
- `idx_automatic_messages_tenant` — Tenant's auto-messages

[Source: architecture/3-database-schema-complete-reference.md#indexes]

### Nullable Columns & Constraints

- `users.password_hash` — Nullable for OAuth future support
- `conversations.kanban_id, column_id` — ON DELETE SET NULL (conversation survives kanban/column deletion)
- `automatic_messages.scheduled_kanban_id` — Nullable for tenant-wide templates
- `conversations.last_message_at` — Nullable until first message
- All timestamps use `DEFAULT NOW()` and are immutable post-creation

### Testing Requirements

**Unit Tests:**

- RLS policy enforcement: attempt to SELECT another tenant's data → denied
- Foreign key constraints: insert invalid FK → error
- Unique constraints: insert duplicate phone in same tenant → error; different tenant → allowed

**Migration Tests:**

- Forward migration applies without error
- Rollback migration removes all tables and functions cleanly
- Re-apply forward migration succeeds on clean DB
