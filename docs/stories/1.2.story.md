# Story 1.2: Supabase Auth (Register, Login, Profile)

## Story Statement

As an owner (tenant admin), I need to register my account with a secure password and login to access the system, so that I can authenticate and manage my tenant's data with proper session handling and security.

## Description

Implement complete authentication flow using Supabase Cloud Auth with JWT tokens stored in httpOnly cookies. This includes:

1. **Register endpoint** — Create owner account, auto-create tenant, issue JWT
2. **Login endpoint** — Validate credentials, issue JWT with tenant_id claim
3. **Profile page** — Display and edit owner name and password
4. **Logout functionality** — Destroy JWT session
5. **Password validation** — Enforce 8+ chars, uppercase, lowercase, number
6. **JWT payload** — Include sub (user UUID), tenant_id, email, role, iat, exp (1 hour)

Authentication forms the security foundation for RLS policies and multi-tenant isolation.

## Acceptance Criteria

- [ ] Register endpoint (`POST /api/auth/register`) accepts email, name, password; validates password strength (8+ chars, mixed case + number)
- [ ] Register creates Supabase Auth user, generates JWT with sub + tenant_id + email + role claims, returns JWT in httpOnly cookie
- [ ] Register auto-creates tenant and owner user record in `users` table (role='owner')
- [ ] Register redirect to `/settings/connection` after successful signup
- [ ] Login endpoint (`POST /api/auth/login`) accepts email, password; validates via Supabase Auth
- [ ] Login returns JWT in httpOnly cookie with 1-hour expiration (iat + 3600s)
- [ ] Profile page (`/profile`) displays owner name and email (read-only), password change form
- [ ] Profile page password form validates strength in real-time (UI + server)
- [ ] Profile page "Save Password" updates `users.password_hash` via authenticated API
- [ ] Logout endpoint (`POST /api/auth/logout`) clears httpOnly cookie and redirects to login
- [ ] JWT payload structure: `{sub, tenant_id, email, role, iat, exp}` (no additional claims)
- [ ] httpOnly cookie set with Secure flag in production, SameSite=Lax, Path=/
- [ ] All auth endpoints return 400/401/500 errors with clear messages (no stack traces)
- [ ] Register/Login form validation shows inline errors (email format, password strength, required fields)
- [ ] Session persists across page reloads (cookie-based, no localStorage)
- [ ] Unauthenticated requests to protected routes redirect to login page

## Risks & Mitigation

| Risk                                                 | Likelihood | Impact   | Mitigation                                                            |
| ---------------------------------------------------- | ---------- | -------- | --------------------------------------------------------------------- |
| Password stored insecurely (plain text or weak hash) | Low        | Critical | Use Supabase Auth's built-in password hashing; never store plain text |
| JWT token leaked (stored in localStorage)            | Medium     | High     | Use httpOnly cookies exclusively; never expose token to JavaScript    |
| Session hijacking via cookie theft                   | Medium     | High     | Secure flag + SameSite=Lax in production; HTTPS only                  |
| Weak password validation allows common passwords     | Low        | Medium   | Enforce 8+ chars, mixed case, number; consider entropy check          |
| Cross-site request forgery (CSRF)                    | Low        | Medium   | Use SameSite=Lax + CSRF tokens for state-changing operations          |
| Token expiration not enforced on backend             | Medium     | High     | Validate exp claim on every protected request                         |

## Dev Notes

### Architecture Context

**Authentication Stack:**

- Supabase Cloud Auth (JWT token generation, user management)
- Next.js API routes for /api/auth/\* endpoints
- httpOnly cookies for token storage (no localStorage)
- [Source: architecture/1-tech-stack-strategic-decisions.md#backend]

**JWT Payload Structure:**

```json
{
  "sub": "user-uuid",
  "tenant_id": "tenant-uuid",
  "email": "owner@example.com",
  "role": "owner",
  "iat": 1234567890,
  "exp": 1234567890 + 3600
}
```

- Expiration: 1 hour (3600 seconds)
- Claims: sub, tenant_id, email, role, iat, exp ONLY
- No sensitive data in JWT (password hash, API keys, etc.)
- [Source: architecture/7-security-architecture.md#authentication-flow]

**Password Requirements:**

- Minimum 8 characters
- At least one uppercase letter (A-Z)
- At least one lowercase letter (a-z)
- At least one digit (0-9)
- NO special characters required (keep it simple)
- [Source: architecture/7-security-architecture.md#password-requirements]

**httpOnly Cookie Configuration:**

- Name: `auth_token`
- Path: `/`
- HttpOnly: true (prevent JavaScript access)
- Secure: true (HTTPS only in production)
- SameSite: Lax (prevent CSRF, allow cross-site requests)
- MaxAge: 3600 (1 hour, matches JWT exp)

### Registration Flow

1. User submits email, name, password
2. Validate password strength (client + server)
3. Call Supabase Auth `signUp()` with email + password
4. On success, create tenant record in `tenants` table
5. Create owner user record in `users` table with tenant_id
6. Generate JWT with sub + tenant_id + email + role
7. Set httpOnly cookie
8. Redirect to `/settings/connection`

### Login Flow

1. User submits email, password
2. Call Supabase Auth `signInWithPassword()`
3. On success, fetch user's tenant_id from `users` table
4. Generate JWT with sub + tenant_id + email + role
5. Set httpOnly cookie
6. Redirect to `/` (home)

### Profile Page Flow

1. Display owner's name and email (read-only)
2. Show password change form (empty fields)
3. On "Save Password": validate strength, send to POST /api/auth/profile
4. Backend: verify JWT, update `users.password_hash`, return success
5. Clear password form, show "Success" toast

### Logout Flow

1. User clicks "Logout"
2. POST /api/auth/logout
3. Backend: clear httpOnly cookie (set MaxAge=0)
4. Redirect to /login

### Error Handling

- **Invalid email format:** "Please enter a valid email address"
- **Weak password:** "Password must be 8+ characters with uppercase, lowercase, and number"
- **Email already registered:** "Email already in use. Try login instead."
- **Invalid credentials (login):** "Invalid email or password"
- **Server error:** "An error occurred. Please try again later."

**NEVER return:**

- Stack traces
- Database error messages
- Internal server details

## 🔴 **CRITICAL ISSUE: RLS Policy Bypass in Login Flow (2026-04-02 Post-QA-Review)**

**Discovered:** 2026-04-02 (During manual testing)  
**Status:** BLOCKING  
**Severity:** CRITICAL  

### Issue: User Cannot Login After Session Expires

#### Symptom
- ✅ User can register and login immediately (JWT cached in cookie)
- ❌ User CANNOT login after closing browser/clearing cookies
- ❌ Returns "Invalid email or password" error (but credentials are correct)
- ⚠️ Multi-tenant RLS policy not being enforced

#### Root Cause

**Login endpoint (`app/api/auth/login/route.ts`) bypasses RLS on users query:**

The endpoint uses `supabaseAnonKey` WITHOUT including the user's JWT in the Authorization header:

```typescript
// ❌ BROKEN: No JWT in Authorization header
const { data: userData, error: userError } = await supabase
  .from("users")
  .select("tenant_id, role")
  .eq("id", userId)
  .single();
```

When no JWT is present:
- `auth.jwt()` evaluates to `NULL` in RLS policy context
- RLS policy: `auth.jwt()['tenant_id'] == users.tenant_id` returns FALSE
- Query is BLOCKED by security policy ❌
- User cannot login

#### Why It Worked Previously

After initial registration, user's JWT was cached in httpOnly cookie. Browser automatically included cookie → Authorization header had user's JWT → RLS allowed query. **But once cookie expires or is cleared, this breaks.**

#### Fix

Use the user's JWT (returned from `signInWithPassword()`) in the Authorization header:

```typescript
// ✅ CORRECT: User's JWT in Authorization header
const accessToken = authData.session?.access_token;

const userSupabase = createClient(supabaseUrl, supabaseAnonKey, {
  global: {
    headers: {
      Authorization: `Bearer ${accessToken}` // ✅ USER'S JWT
    }
  }
});

const { data: userData, error: userError } = await userSupabase
  .from("users")
  .select("tenant_id, role")
  .eq("id", userId)
  .single();
```

#### Fix #1: Apply Correct RLS Policy Migration (Database)

```sql

SELECT (tenant_id)::text = (auth.jwt() ->> 'tenant_id'::text)

USING (auth.uid() = id);
```
