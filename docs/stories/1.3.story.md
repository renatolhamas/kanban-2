# Story 1.3: Email Confirmation Flow (Registration + Resend)

## Story Statement

As a new user registering for the system, I need to confirm my email address before being able to login, so that we ensure email validity and prevent spam accounts while providing a frictionless signup experience.

## Description

Implement email confirmation flow that bridges the gap between Supabase Admin API user creation (which doesn't auto-send confirmation emails) and user login validation. This story builds on Story 1.2's registration endpoint by adding:

1. **Email Confirmation Pipeline** — After `admin.createUser()`, generate confirmation link via `admin.generateLink()` and send via Resend
2. **Resend Integration** — Use Resend free tier (100 emails/day) to reliably send confirmation emails with 99%+ deliverability
3. **Link Validation** — Validate confirmation link format before sending to ensure Supabase can confirm user
4. **Rate Limiting** — Protect against spam registration (10 attempts/15min per IP, 3 attempts/60min per email)
5. **Cleanup Strategy** — Best-effort rollback if email sending fails, with audit logging
6. **Environment Validation** — Prevent production deploys with wrong domain configuration
7. **Sandbox Testing** — Use Resend sandbox in development to avoid consuming quota during testing

This ensures users cannot login until email is confirmed (`confirmed_at NOT NULL`), improving security posture and data quality.

## Acceptance Criteria

### Core Functionality

- [ ] Modified `POST /api/auth/register` creates user, tenant, user record as in Story 1.2
- [ ] After user record creation, endpoint generates confirmation link via `admin.generateLink(type: 'signup', email)`
- [ ] Confirmation link is validated (robust URL parsing: path contains `/auth/v1/verify`, type param is exactly `'signup'`, token exists and not empty)
- [ ] If link validation fails, log warning but continue (link may still be valid, don't block UX)
- [ ] Endpoint sends confirmation email via Resend API with subject "Confirm Your Signup"
- [ ] Email contains clickable link that redirects to `{NEXT_PUBLIC_APP_DOMAIN}/login?confirmed=true` after user clicks
- [ ] User clears email, `confirmed_at` timestamp is set to NOW() in Supabase auth users table
- [ ] User receives HTTP 202 (Accepted) with message: "Registration successful. Please check your email to confirm your address."

### Cleanup & Reliability

- [ ] If email send fails, performs best-effort cleanup: attempts to delete auth user AND tenant (not sequentially, so both are tried)
- [ ] Cleanup attempts logged individually (✅ or ❌) for observability
- [ ] If cleanup fails completely, entry is logged to `failed_registrations` table for manual audit
- [ ] Cleanup does NOT block with errors; returns 500 to user if email fails

### Rate Limiting (Spam Protection)

- [ ] Endpoint enforces rate limit: max 10 registration attempts per IP per 15 minutes (blocks at #11)
- [ ] Endpoint enforces rate limit: max 3 registration attempts per email per 60 minutes (blocks at #4)
- [ ] Rate-limited requests return HTTP 429 with `Retry-After` header (seconds until reset)
- [ ] Rate limit applies BEFORE input validation (prevents enumeration attacks)
- [ ] Rate limit correctly extracts client IP from `X-Forwarded-For` header (proxy/load balancer aware)

### Environment & Domain Validation

- [ ] Server startup validates `NEXT_PUBLIC_APP_DOMAIN` matches expected value for NODE_ENV (e.g., `localhost:3000` for dev, `staging.kanban.com` for staging)
- [ ] If domain mismatch in staging/dev, logs warning but allows execution
- [ ] If domain mismatch in production, **fails immediately** (throws error, prevents deployment)
- [ ] Production cannot launch with sandbox Resend emails

### Testing & Observability

- [ ] Confirmation link logged in server console at info level (for manual verification)
- [ ] Link validation results logged including path, type param, token param (for debugging)
- [ ] Email send logged with timestamp and status (delivered vs error)
- [ ] Rate limit blocks logged with IP/email and reset time
- [ ] All errors caught and logged; no unhandled promises or exceptions

## In Scope

## Risks & Mitigations

| Risk | Severity | Mitigation |
|------|----------|-----------|
| Email never arrives (Resend API down) | MEDIUM | Implement retry with exponential backoff; monitor Resend status; sandbox testing prevents quota waste |
| Confirmation link format wrong | MEDIUM | Robust URL parsing validation before sending; debug logging of full link; manual test step documented |
| Domain mismatch in production | CRITICAL | Environment validation that FAILS deployment if mismatch detected |
| Rate limit too liberal (still allows abuse) | LOW | Start conservative (10/15min), monitor real usage, adjust in Phase 1.5 |
| Data orphans from cleanup failure | MEDIUM | Audit table (`failed_registrations`) enables discovery and manual cleanup |
| Race condition: tenant created but user record fails | LOW | 100ms delay before user record creation resolves FK constraint sync |

## Dev Notes

**Architecture Decisions Made by @architect (Phase 0 Refinement):**

1. **Resend Free Tier Selected** — Over SendGrid/AWS SES for MVP (100/day free, 99%+ deliverability, requires zero setup vs SMTP complexity)
   - Source: plan-002.md§1 (Opção C - Hybrid)

2. **Robust URL Parsing for Link Validation** — Not substring matching
   - Prevents false positives if SDK changes param order
   - Source: plan-002.md§2.2d + §2.5 (STEP 6.5)

3. **In-Memory Rate Limiting for MVP** — Not Redis
   - Sufficient for single-instance deployments
   - TODO Phase 1.5: Migrate to Redis for multi-instance
   - Source: plan-002.md§2.1a

4. **Best-Effort Cleanup** — Not sequentially blocked
   - Each delete operation wrapped in try/catch independently
   - If deleteUser fails, deleteTenant still attempts (vs cascading failure)
   - Source: plan-002.md§2.2b.1

5. **Audit Table for Failed Registrations** — Enables manual cleanup discovery
   - Tracks `created_resources` JSON to know what was partially created
   - Source: plan-002.md§2.2b.1 + §3.0

6. **Timeout + Retry on Link Validation** — Supabase replication lag
   - After user clicks link, retry 3x with 1s backoff before declaring confirmed
   - Source: plan-002.md§2.2d

7. **Environment Domain Validation** — Fails deployment if mismatch in production
   - Uses env-validation.ts module to check NEXT_PUBLIC_APP_DOMAIN
   - Source: plan-002.md§2.4a

8. **Resend Sandbox in Development** — Uses delivered@resend.dev, real email in prod
   - Prevents quota waste during development
   - Source: plan-002.md§2.4b
