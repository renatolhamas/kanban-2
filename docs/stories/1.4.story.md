# Story 1.4: RLS Policies Validation (Testing & Verification)

## Story Statement

As a security auditor, I need to validate that Row-Level Security (RLS) policies implemented in Story 1.1 are correctly enforced across all data tables, so that cross-tenant data access is impossible and the system is secure for production use.

## Description

Comprehensively validate and test the RLS policies that were implemented in Story 1.1 (Database Schema Creation). This story verifies that the security layer is working correctly by:

1. **Testing RLS Enforcement** — Verify that SELECT, INSERT, UPDATE, DELETE queries respect tenant isolation
2. **Cross-Tenant Attack Prevention** — Simulate 10+ attack scenarios and confirm they all fail safely
3. **Performance Validation** — Measure query performance under RLS to ensure acceptable overhead
4. **Documentation & Auditing** — Document test results and create maintenance guide for RLS policies

**Important Note:**
- Story 1.1 already IMPLEMENTED the RLS policies (`auth.get_tenant_id()` function, policies on all 8 tables)
- Story 1.4 VALIDATES that implementation through comprehensive testing, not reimplementation

## Acceptance Criteria

- [ ] Verify `auth.get_tenant_id()` function exists and correctly extracts tenant_id from JWT
- [ ] Verify RLS is enabled on all 8 tables: tenants, users, kanbans, columns, contacts, conversations, messages, automatic_messages
- [ ] Verify policy on `tenants`: SELECT only returns user's own tenant row
- [ ] Verify policy on `users`: SELECT only returns users in user's tenant, INSERT/UPDATE/DELETE blocked for other tenants
- [ ] Verify policy on `kanbans`: SELECT only returns kanbans in user's tenant, mutations blocked for other tenants
- [ ] Verify policy on `columns`: SELECT only returns columns for kanbans in user's tenant, mutations blocked
- [ ] Verify policy on `contacts`: SELECT only returns contacts in user's tenant, mutations blocked
- [ ] Verify policy on `conversations`: SELECT only returns conversations in user's tenant, mutations blocked

## Dev Notes

### RLS Architecture [Source: architecture/3-database-schema-complete-reference.md#3.2, architecture/7-security-architecture.md#7.3]

**Tenant Extraction:**
- All users authenticate via Supabase Auth (Story 1.2)
- JWT payload includes `sub` (user ID) and `tenant_id` claims
- Function `auth.get_tenant_id()` extracts tenant_id from JWT: `(auth.jwt()->>'tenant_id')::UUID`
- Returns NULL if unauthenticated

**RLS Enforcement:**
- Every tenant-scoped table has `tenant_id` column
- RLS policies applied at query execution — no application-layer filtering
- Supabase validates JWT signature before extracting claims

### Testing Framework & Execution

**Framework:** Vitest (not Jest or MCP)
- Tests in `tests/rls-validation.test.ts` connect **directly to Supabase** via `@supabase/supabase-js` library
- No MCP usage (to avoid complications)
- Run tests: `npm run test tests/rls-validation.test.ts`
- Each test simulates authenticated request with JWT claims (`sub`, `tenant_id`)

**Test Pattern:**
```typescript
import { createClient } from '@supabase/supabase-js'

test('TC-RLS-001: User A cannot see Tenant B data', async () => {
  const supabase = createClient(url, key, { auth: { persistSession: false } })
  // Set auth context with User A's JWT
  const { data } = await supabase.from('kanbans').select()
  expect(data).toHaveLength(0) // RLS blocks other tenant's rows
})
```

### Testing Strategy — Attack Scenarios

**Cross-Tenant Data Leakage Prevention:**
1. **Privilege Escalation:** User A impersonates User B → Supabase signature validation prevents
2. **Direct SQL Injection:** Attacker injects SQL that bypasses RLS → PostgreSQL RLS enforced at DB layer
3. **Concurrent Modification:** User A modifies data while User B views it → RLS isolation prevents
4. **Nested Relationship Bypass:** User A queries columns via kanban lookup → nested RLS policy prevents
