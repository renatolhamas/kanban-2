# Story 1.5: Onboarding (Tenant Auto-Create & Default Kanban Setup)

## Story Statement

As a new Owner, I need to register with email/password and have my tenant account automatically created with a default "Main" kanban pre-configured with standard columns, so that I can immediately start managing conversations without manual setup.

## Description

Implement the complete onboarding flow that creates a new tenant and its default workspace structure:

1. **Register Page** — Owner creates account with email and password
2. **Automatic Tenant Creation** — Tenant row created in `tenants` table with default name
3. **Default Kanban Setup** — "Main" kanban auto-created with standard columns (Novo, Qualificado, Em Negociação, Fechado)
4. **User Record Creation** — Owner user record created with tenant association and RLS metadata
5. **Post-Register Redirect** — User redirected to Home (dashboard) or Config page

**Important Note:**
- Story 1.1 (Database Schema with RLS) + Story 1.2 (Auth Layer) are prerequisites
- Story 1.5 ties them together at the application layer (Register UI + seed logic)
- Kanban "Main" must be marked as the default kanban for auto-routing

## Dev Notes

### Auth Architecture [Source: architecture/5-auth-and-user-management.md]

**Supabase Auth Integration:**
- Email/password signup via Supabase Auth
- JWT issued on successful auth
- JWT contains: `sub` (user ID), `email`, `tenant_id` (custom claim from DB metadata)
- RLS policies use `auth.uid()` and `auth.jwt()->>'tenant_id'`

**Custom JWT Claims:**
- Story 1.2 (Auth Layer) configured JWT claim injection
- `tenant_id` claim added via Supabase Realtime subscription metadata or DB trigger
- Verify claim is present before issuing JWT

### Database Tables [Source: docs/architecture/3-database-schema-complete-reference.md]

**Tables involved in onboarding:**

1. **auth.users** (Supabase Auth internal)
   - Managed by Supabase Auth
   - Email, password hash, aud="authenticated"

2. **tenants** (custom)
   ```sql
   id UUID PRIMARY KEY
   name TEXT NOT NULL
   created_at TIMESTAMP DEFAULT now()
   updated_at TIMESTAMP DEFAULT now()
   ```
   - One row per organization/workspace
   - No auth.users link (tenant is workspace container)

3. **users** (custom, app-level)
   ```sql
   id UUID PRIMARY KEY (from auth.users.id)
   email TEXT NOT NULL
   tenant_id UUID FK → tenants.id
   role TEXT DEFAULT 'OWNER' -- or 'USER', 'ADMIN'
   created_at TIMESTAMP
   updated_at TIMESTAMP
   last_login TIMESTAMP
   ```
   - Links auth.users to tenants
   - RLS: (auth.uid() = users.id) AND (users.tenant_id matches auth.jwt()->>'tenant_id')

4. **kanbans** (custom)
   ```sql
   id UUID PRIMARY KEY
   tenant_id UUID FK → tenants.id
   name TEXT NOT NULL -- e.g., "Main"
   is_main BOOLEAN DEFAULT false
   order INT DEFAULT 1
   created_at TIMESTAMP
   updated_at TIMESTAMP
   ```
   - One or more per tenant
   - is_main = true for default kanban (only 1 per tenant)

5. **columns** (custom, kanban columns)
   ```sql
   id UUID PRIMARY KEY
   kanban_id UUID FK → kanbans.id
   name TEXT NOT NULL -- e.g., "Novo", "Qualificado"
   order INT DEFAULT 1
   created_at TIMESTAMP
   updated_at TIMESTAMP
   ```
   - 4+ columns per kanban
   - order determines display sequence

### Onboarding Flow Sequence

```
1. User fills register form (email, password)
   ↓
2. Client submits to Supabase Auth signup endpoint
   ↓
3. Supabase Auth creates auth.users record (if email not already registered)
   ↓
4. Auth hook/function triggers:
   a. Insert tenants row (name = "{email-prefix}-workspace")
   b. Insert users row (id from auth.users, tenant_id, role='OWNER')
   c. Insert kanbans row (name='Main', is_main=true, order=1)
   d. Insert 4 columns rows (Novo, Qualificado, Em Negociação, Fechado)
   ↓
5. Supabase Auth issues JWT (with tenant_id claim from DB metadata)
   ↓
6. Client receives JWT, stores in session/localStorage
   ↓
7. Client redirects to `/` (Home page)
   ↓
8. Home page queries kanbans + columns (RLS filters by tenant_id from JWT)
   ↓
9. User sees "Main" kanban with 4 columns, ready to use
```

### Implementation Approach

**Frontend (Register Component):**
- Use Supabase Auth JavaScript client (`@supabase/auth-helpers-react` or `@supabase/supabase-js`)
- Form validation (email format, password strength) client-side
- On submit: `supabaseClient.auth.signUp({ email, password })`
- Catch signup errors (email already registered, weak password, etc.)
- On success: redirect to `/`

**Backend (Tenant/Kanban Creation):**
- **Option A (Supabase Edge Function):** Create Edge Function that runs after auth.users insert
- **Option B (Database Trigger):** PostgreSQL trigger on auth.users INSERT → creates tenants + kanbans + columns
- **Option C (Node.js Backend):** Custom signup endpoint that calls Supabase Auth + creates records

**Recommended:** Option B (trigger) for atomicity and consistency. If trigger not possible, Option A (Edge Function) with transaction-like semantics.

**Tenant Name Generation:**
```javascript
const tenantName = email.split('@')[0] + '-workspace';
// "john.doe@example.com" → "john.doe-workspace"
```

**Default Columns:**
```javascript
const defaultColumns = [
  { name: 'Novo', order: 1 },
  { name: 'Qualificado', order: 2 },
  { name: 'Em Negociação', order: 3 },
  { name: 'Fechado', order: 4 }
];
```

**Unit Tests:**

- Password validation: strength rules (8+ chars, mix case/numbers)

**Integration Tests:**

- Concurrent signups: prevent race conditions (unique email constraint)
- Error scenarios: already registered, weak password, network errors
