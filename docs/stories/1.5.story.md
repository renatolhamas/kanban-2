## Description

Implement the complete onboarding flow that creates a new tenant and its default workspace structure:

1. **Register Page** — Owner creates account with email and password
2. **Automatic Tenant Creation** — Tenant row created in `tenants` table with default name
3. **Default Kanban Setup** — "Main" kanban auto-created with standard columns (Novo, Qualificado, Em Negociação, Fechado)
4. **User Record Creation** — Owner user record created with tenant association and RLS metadata
5. **Post-Register Redirect** — User redirected to Home (dashboard) or Config page

**Important Note:**
- Story 1.1 (Database Schema with RLS) + Story 1.2 (Auth Layer) are prerequisites
- Story 1.5 ties them together at the application layer (Register UI + seed logic)
- Kanban "Main" must be marked as the default kanban for auto-routing

**Rationale (based on codebase analysis):**

The application uses **application-layer orchestration exclusively** (no triggers, no Edge Functions):
- `register/route.ts` — Multi-step creation: auth → tenant → user (with rollback)
- `login/route.ts` — Multi-step fetch + compute: auth → fetch user → generate JWT
- `profile/route.ts` — Single operations: fetch or update
- All patterns handle errors in-app, not at DB layer

**For Story 1.5:**
Extend `register/route.ts` with new STEP 5.5-5.6:
```
STEP 4: Create tenant ✅ (exists)

STEP 5.6: Create 4 columns ← NEW
STEP 7: Send email (existing)
```

**Benefits:**
- ✅ Reuses proven pattern from tenant creation
- ✅ Same rollback mechanism: if kanban fails → cleanup auth user + tenant
- ✅ Centralized error handling and logging
- ✅ Fully testable (integration tests can mock kanban creation)
- ✅ No new database objects needed (tables exist in schema.sql)

**Implementation:**
1. Create helper: `lib/kanban.ts: createDefaultKanban(supabase, tenantId)`
2. Call from register route after user creation (STEP 5.5)
3. Add rollback: if kanban creation fails, cleanup auth user + tenant
4. Add integration test: register → verify kanban + 4 columns created
