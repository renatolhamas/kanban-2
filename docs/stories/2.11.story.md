# Story 2.11 — Application Layout & Navigation

## 📓 Decision Log (Atlas Analyst)

| Item | decision | Rationale |
|------|----------|-----------|
| **Logo** | Dynamic Component | Facilitates token-based dark mode control without multiple image assets. |
| **Mobile Sidebar** | shadcn/ui Sheet (Drawer) | Standard premium mobile pattern; avoids layout shifts and improves focus. |
| **User Menu Trigger** | Avatar + Chevron | Balanced "affordance" and identity; saves horizontal space compared to Name. |
| **Data Flow** | Context API (Direct) | Leverages existing Auth/Theme contexts; keeps RootLayout clean and modular. |
| **Sidebar Persistence** | LocalStorage | Remembers user preference (collapsed/expanded) across sessions. |
| **Navigation Links** | Button + `asChild` | shadcn pattern; ensures consistent design system tokens (hover/focus). |
| **Modularity** | Isolated `UserMenu.tsx` | Better maintainability, separate logic from Header, and easier testing. |

## 📋 Overview

Refatorar os componentes de navegação **Header**, **Sidebar** e **User Menu** aplicando os componentes validados do Design System (Stories 2.1-2.10). O objetivo é criar uma navegação funcional, consistente, e totalmente alinhada com o "Architectural Ledger" design que foi estabelecido e auditado em dark mode.

**Objetivo:** Garantir que a aplicação apresente um layout robusto e coerente em todas as páginas protegidas, pronto para as próximas epics (Kanban Board, Contacts, Settings).

## 🔧 Technical Approach

### Phase 1: Header Component Refactor (Tasks 1-2)

**Task 1: Analyze Current Header Structure and Design System Components**

1. Revisar estrutura atual de Header (se existente em RootLayout)
2. Mapear componentes do Design System reutilizáveis:
   - Button (para ícones, logo)
   - Dropdown (para UserMenu)
   - Typography (para Page Title)
3. Definir layout visual: Logo (left) | Page Title (center) | UserMenu (right)
4. Planejamento de props e estado (tema, usuário logado, etc.)

**Task 2: Implement Header Component with Design System**

1. Criar `src/components/layout/Header.tsx`
2. Renderizar Logo usando componente dinâmico (Icon + Text) para suporte total a tokens.
3. Renderizar Page Title (Typography)
4. Integrar UserMenu component (extraído em arquivo próprio, ver Task 5)
5. Aplicar Tailwind classes para responsive (flex, gap, items-center, etc.)
6. Suportar dark mode via Tailwind `dark:` prefixes
7. Adicionar ARIA labels apropriados (role="banner", aria-label="Application header")

### Phase 2: Sidebar Component Refactor (Tasks 3-4)

**Task 3: Implement Sidebar Navigation**

1. Criar `src/components/layout/Sidebar.tsx`
2. Renderizar links de navegação usando `shadcn Button` com prop `asChild` envolvendo Next.js `Link`.
3. Definir links: Home (kanban icon), Contacts (people icon), Settings (gear icon).
4. Ativo state baseado em rota atual (useRouter hook).
5. Hover/active states via Tailwind tokens (bg-primary, text-white, etc.).
6. Persistir estado colapsado no **LocalStorage**.
7. Suportar dark mode corporativo.
8. Mobile collapse: usar Tailwind responsive classes (`hidden md:block`).

**Task 4: Responsive Sidebar Behavior**

1. Desktop (md+): Sidebar sempre visível.
2. Mobile (<md): Sidebar colapsado por padrão, trigger via hamburger menu usando o componente **Sheet (Drawer)** da shadcn/ui.
3. Keyboard nav: Tab order correto, focus visible em todos links.

### Phase 3: User Menu Component (Task 5)

**Task 5: Implement User Menu Dropdown**

1. Criar `src/components/layout/UserMenu.tsx` como componente isolado.
2. Renderizar trigger button com **Avatar + Chevron Down** (consumindo de `useAuth`).
3. Usar `shadcn DropdownMenu` para consistência.
4. Menu items: Profile, Logout.
5. Logout action: Limpar JWT, redirect to /login.
6. Suportar dark mode.
7. ARIA labels: role="menubutton", aria-expanded, aria-haspopup="menu".
8. Keyboard nav: Enter/Space abre menu, Arrow keys navegam, Esc fecha.

### Phase 4: Layout Integration & Testing (Tasks 6-8)

**Task 6: Integrate Header + Sidebar into RootLayout**

1. Criar/atualizar `src/components/layout/RootLayout.tsx`
2. Structure:
   ```
   <RootLayout>
     <Header />
     <div class="flex">
       <Sidebar />
       <main class="flex-1">{children}</main>
     </div>
   </RootLayout>
   ```
3. Consumir dados (usuário, tema, etc.) diretamente via **Context API** nos componentes.
4. Aplicar Sidebar conditionally (apenas em protected pages, não em /login, /register)

**Task 7: Dark Mode & Responsive Testing**

1. Testar Header + Sidebar + UserMenu em light mode
2. Testar em dark mode (via ThemeToggle do Story 2.10)
3. Testar responsividade: desktop (1280px+), tablet (768px), mobile (320px)
4. Validar focus states em ambos os temas
5. Keyboard navigation: Tab through links, focus visible

**Task 8: Accessibility & Quality Validation**

1. Rodar AXE DevTools na página (Home ou Settings com layout completo)
2. Validar WCAG AA contrast em Header/Sidebar/UserMenu em light e dark mode
3. Gerar keyboard navigation test matrix
4. Verificar ARIA roles e labels
5. Documentar findings
