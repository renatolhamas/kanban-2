# Story 3.2: Evo GO WhatsApp Integration — Complete Smart Pairing Flow

## Story Statement

**As a** tenant owner,  
**I want to** establish a complete, intelligent WhatsApp pairing flow with real-time synchronization and instance reuse,  
**so that** I can connect my WhatsApp account seamlessly, avoid duplicate instances, and receive live status updates directly from Evo GO.

---

## Acceptance Criteria

### QR Code Generation (Smart Instance Management)
1. [ ] **AC 1:** `POST /api/settings/evo-go/qr` reuses existing instances (no duplicates via `getOrCreateInstance()`)
2. [ ] **AC 2:** Instance token is unique per connection attempt (randomUUID, not global API key)
3. [ ] **AC 3:** Endpoint calls `callEvoGoConnect(token, webhookUrl)` to register webhook automatically
4. [ ] **AC 4:** QR generation includes retry logic (5 attempts, 2s delay between attempts)
5. [ ] **AC 5:** Endpoint returns 409 if WhatsApp already logged in (`logged_in === true`)
6. [ ] **AC 6:** Endpoint detects "client disconnected" and treats as disconnected state (continue to generate QR)

### Status Polling (Real-Time Sync with Evo GO)
7. [ ] **AC 7:** `GET /api/settings/evo-go/status` calls `getEvoGoStatus()` to fetch REAL status from Evo GO
8. [ ] **AC 8:** Status endpoint syncs `logged_in=true` → auto-update DB to `connection_status='connected'`
9. [ ] **AC 9:** Status endpoint detects `logged_in=false` → auto-update DB to `connection_status='disconnected'`
10. [ ] **AC 10:** 401/403 errors (invalid token) → auto-clears instance from DB (instance_id=null, token=null)
11. [ ] **AC 11:** Polling only calls Evo GO when status is `'connecting'` or `'connected'` (not on disconnect)
12. [ ] **AC 12:** Returns `{ connection_status, evolution_instance_id, qr_code, qr_code_expires_at }`

### Disconnect (Instance Preservation for Reconnect)
13. [ ] **AC 13:** `POST /api/settings/evo-go/disconnect` calls `callEvoGoLogout(token)` (Opção 2: Logout)
14. [ ] **AC 14:** Disconnect preserves `evolution_instance_id` and `evolution_instance_token` in DB
15. [ ] **AC 15:** Only `connection_status` changes to `'disconnected'` (enables reconnect with same instance)
16. [ ] **AC 16:** No "Instance already exists" error on reconnect after disconnect

### Webhook Integration
17. [ ] **AC 17:** `POST /api/webhooks/evo-go` validates HMAC-SHA256 signature via `X-Signature` header
18. [ ] **AC 18:** Webhook extracts `tenantId` from query parameter: `?tenantId={tenantId}`
19. [ ] **AC 19:** Webhook processes `CONNECTION_UPDATE` event → updates `tenants.connection_status`
20. [ ] **AC 20:** Webhook auto-registered via `callEvoGoConnect(token, webhookUrl)` during QR generation

### Frontend
21. [ ] **AC 21:** Settings page displays WhatsApp Connection status (connected/disconnected/connecting)
22. [ ] **AC 22:** "Conectar WhatsApp" button triggers QR generation via `POST /api/settings/evo-go/qr`
23. [ ] **AC 23:** QR displays with 5-minute countdown timer (updates every 1s)
24. [ ] **AC 24:** Status updates automatically via polling (no manual refresh needed)
25. [ ] **AC 25:** "Desconectar" button appears when connected with confirmation modal

### Database
26. [ ] **AC 26:** Migrations add `qr_code`, `qr_code_expires_at`, `evolution_instance_token` columns
27. [ ] **AC 27:** All mutations use tenant isolation (WHERE `id = tenantId`)

## Implementation Details

### Key Features Actually Implemented

#### 1. Smart Instance Management (`getOrCreateInstance`)
- Avoids instance duplication by reusing existing instances
- Generates unique token per connection (randomUUID, not global API key)
- Automatically registers webhook URL during connect

**Flow:**
```
POST /api/settings/evo-go/qr
  ↓
getOrCreateInstance(tenantId)
  ├─ Check: does instance exist?
  ├─ YES: reuse (return existing instance_id, token)
  └─ NO: create new with unique token (randomUUID)
  ↓
Save to DB: instance_id, token, status='connecting'
  ↓
callEvoGoConnect(token, webhookUrl)  ← auto-register webhook
  ↓
getEvoGoQRCode with retry (5×, 2s delay)
```

#### 2. Real-Time Status Polling (Not Just DB Reads)
- Calls Evo GO directly on every poll (not stale DB data)
- Syncs on `logged_in === true` (user scanned QR, WhatsApp session active)
- Detects session loss (`logged_in === false`)
- Handles token lifecycle (401/403 → clear invalid tokens)

**Flow:**
```
GET /api/settings/evo-go/status
  ↓
IF connection_status in ['connecting', 'connected']:
  ↓
  Call getEvoGoStatus(token) ← real-time from Evo GO
    ↓
    IF logged_in === true:
      Update DB: status='connected'
    ↓
    IF logged_in === false:
      Update DB: status='disconnected'
    ↓
    IF 401/403 (unauthorized):
      Clear DB: instance_id=null, token=null, status='disconnected'
ELSE:
  Just return DB status
```

#### 3. Disconnect Strategy (Opção 2: Logout)
- Calls Evo GO `POST /instance/disconnect/{instanceName}`
- Ends WhatsApp session, but **keeps instance alive**
- Preserves `instance_id` and `token` in DB
- Enables reconnect with same instance (no "Instance already exists" error)

**Flow:**
```
POST /api/settings/evo-go/disconnect
  ↓
callEvoGoLogout(token)
  ↓
Update DB:
  connection_status = 'disconnected'
  (instance_id and token PRESERVED)
  ↓
Result: Fast reconnect using same instance
```

#### 4. Webhook Integration (Auto-Registered)
- Webhook URL auto-registered during pairing: `/api/webhooks/evo-go?tenantId={tenantId}`
- `callEvoGoConnect(token, webhookUrl)` registers webhook on Evo GO side
- Zero manual webhook configuration needed
- HMAC-SHA256 validation required (non-optional)

**Flow:**
```
QR endpoint calls:
  callEvoGoConnect(token, webhookUrl)
    ↓
    Evo GO registers: webhook at ${APP_URL}/api/webhooks/evo-go?tenantId=${tenantId}
    ↓
When user scans QR:
  Evo GO POSTs CONNECTION_UPDATE to webhook
    ↓
    Webhook validates HMAC-SHA256 via X-Signature
    ↓
    Extracts tenantId from query param
    ↓
    Updates: tenants.connection_status (best-effort)
```

## Technical Architecture

### Core Client Library Functions (`lib/api/evo-go-client.ts`)

1. **`getOrCreateInstance(tenantId)`**
   - Checks if instance exists for tenant
   - Reuses or creates new (unique token)
   - Returns: `{ instance_id, token }`

2. **`callEvoGoConnect(token, webhookUrl)`**
   - Initializes WhatsApp session
   - Registers webhook URL on Evo GO
   - Returns: connection confirmation

3. **`getEvoGoQRCode(token)`**
   - Fetches QR code from Evo GO
   - Returns: `{ qr_code: string }`

4. **`getEvoGoStatus(token)`**
   - Fetches real-time status from Evo GO
   - Returns: `{ connected: boolean, logged_in: boolean }`

5. **`callEvoGoLogout(token)`**
   - Ends WhatsApp session (Logout, not Delete)
   - Keeps instance alive
   - Returns: confirmation

### API Endpoints

| Method | Endpoint | Behavior |
|--------|----------|----------|
| POST | `/api/settings/evo-go/qr` | getOrCreateInstance → connect → retry getQR |
| GET | `/api/settings/evo-go/status` | Poll real status from Evo GO + sync DB |
| POST | `/api/settings/evo-go/disconnect` | Logout + preserve instance |
| POST | `/api/webhooks/evo-go` | HMAC validation + CONNECTION_UPDATE handler |

### Database Schema Changes

Migrations added:
- `qr_code` — Base64-encoded QR image
- `qr_code_expires_at` — Expiration timestamp
- `evolution_instance_token` — Unique per connection (randomUUID)

All mutations use tenant isolation: `WHERE tenants.id = {tenantId}`

## Key Technical Decisions

### Why "logged_in" not "Connected"?
- `Connected=true` means infrastructure active (not user-ready)
- `logged_in=true` means WhatsApp session active (user can receive messages)
- Prevents false "Conectado" status when instance exists but has no active WhatsApp

### Why Retry Logic for QR Generation?
- Evo GO QR generation is async (1-3s after connect call)
- Retry with 2s delay (5× max) catches transient timing
- User sees QR faster, timeout-proof

### Why Preserve Instance on Disconnect?
| Option | Reconnect | Instance Count | Notes |
|--------|-----------|-----------------|-------|
| Opção 1 (Delete) | Creates new | Grows | Bloat, auth issues |
| Opção 2 (Logout) | Reuses | Stable | Fast, clean ✅ |
| Opção 3 (Pause) | Complex | N/A | Overcomplicated |

**Chosen:** Opção 2 (Logout)

### Why Auto-Register Webhook?
- Avoids manual webhook configuration step
- `callEvoGoConnect()` registers webhook URL during pairing
- One-shot setup, production-ready
