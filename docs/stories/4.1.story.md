# Story 4.1: Correção do Isolamento Multi-Tenant via RLS

**Status**: Draft  
**Epic**: Epic 4 — Preparação para Realtime & Compliance

## 📋 Story Statement

**As a** System Security Officer,  
**I want** to fix the multi-tenant isolation by migrating RLS policies from `user_metadata` (user-editable) to `app_metadata` (immutable),  
**so that** the database enforces true tenant isolation and Realtime subscriptions cannot leak data across tenants.

---

## ✅ Acceptance Criteria

1. **Custom Access Token Hook implementado e ativo**
   - Função PostgreSQL `public.custom_access_token_hook` criada em migration
   - Hook habilitado no dashboard de Authentication > Hooks
   - JWT sempre contém `app_metadata.tenant_id` injetado do banco

2. **`app_metadata` preenchido em todos os usuários**
   - Script backfill executa sem erros
   - Todos os usuários (existentes + novos) têm `app_metadata.tenant_id` e `app_metadata.role` definidos
   - Novos usuários recebem `app_metadata` via `register/route.ts` STEP 5.1

3. **28 RLS policies migradas para `app_metadata`**
   - Todas as policies nas tabelas `tenants`, `kanbans`, `contacts`, `conversations`, `automatic_messages`, `columns`, `messages` atualizadas
   - Padrão `user_metadata` substituído por `app_metadata` em TODAS as policies
   - Subqueries adicionadas para performance (`SELECT ...` inline)

4. **Isolamento testado e validado**
   - Query com JWT correto retorna dados esperados
   - Query com JWT incorreto retorna 0 linhas (não vaza dados)
   - Query setando apenas `user_metadata` retorna 0 linhas (não usa campo editável)
   - Nenhuma regressão nas APIs existentes

5. **Security audit passa sem ERRORs**
   - `*security-audit full` executa sem violações de RLS
   - Nenhum data leak detectado
   - Relatório de auditoria gerado e documentado

6. **Hook falha de forma segura (Strict)**
   - Em qualquer erro de query no banco, o hook lança exceção explícita (`RAISE EXCEPTION`)
   - Login falha com erro tratável — nunca emite JWT parcial (sem `tenant_id`)
   - Testável: simular falha de banco → login retorna erro, não JWT vazio

## 📝 Dev Notes

### Diagnóstico do Problema

As 28 RLS policies atuais usam `user_metadata` para isolamento de tenant:
```sql
-- INSEGURO — user_metadata é editável via supabase.auth.updateUser()
tenant_id = (auth.jwt() -> 'user_metadata' ->> 'tenant_id')::uuid
```

**Dois problemas simultâneos:**
1. `user_metadata` é editável pelo usuário final → bypass de isolamento
2. `tenant_id` não é preenchido em `user_metadata` no registro → claim vazia

**Por que agora é crítico:** Story 4.2 implementa Supabase Realtime, que depende de RLS para filtrar eventos por tenant. Sem correção, toda subscrição retornaria dados de TODOS os tenants.

[Source: plans/fase.4/F4.plan.001.md#1-diagnóstico]

### Solução em 3 Componentes

#### **Componente 1: Injeção de `app_metadata` no Registro**
- **Arquivo**: `app/api/auth/register/route.ts`
- **Onde**: STEP 5.1 (após criação do user record, antes de STEP 5.5)
- **O que**: Chamar `supabase.auth.admin.updateUserById()` para setar `app_metadata.tenant_id` e `app_metadata.role` (imutáveis)
- **Fallback**: Se falhar, o Custom Access Token Hook insere na próxima autenticação

#### **Componente 2: Custom Access Token Hook (PostgreSQL)**
- **Arquivo**: `supabase/migrations/20260414_create_custom_access_token_hook.sql`
- **Função**: `public.custom_access_token_hook(event jsonb) → jsonb`
- **Comportamento**: 
  - Ao gerar JWT, lê `users.tenant_id` e `users.role` do banco
  - Injeta no claims como `app_metadata.tenant_id` e `app_metadata.role`
  - Garante JWT sempre consistente com o banco
- **Permissões**: GRANT EXECUTE a `supabase_auth_admin` only
- **Ativação Manual**: Dashboard → Authentication → Hooks → Selecionar `public.custom_access_token_hook`

#### **Decisões de Design do Hook**

> ⚠️ **ATENÇÃO**: O código SQL em `F4.plan.001.md` NÃO reflete estas decisões. A story 4.1 é o spec autoritativo.

**Comportamento em Falha — Strict (DECISÃO: 2026-04-14)**

O hook usa `EXCEPTION WHEN OTHERS THEN RAISE EXCEPTION` para qualquer falha de banco:

```sql
BEGIN
  -- ... lógica do hook ...
EXCEPTION WHEN OTHERS THEN
  RAISE EXCEPTION 'custom_access_token_hook failed: %', SQLERRM;
END;
```

- Falha de banco = login falha (erro 500) — usuário não recebe JWT sem `tenant_id`
- **Justificativa**: Story 4.2 (Realtime) conecta WebSocket usando JWT. JWT sem `tenant_id` conectaria sem filtro de tenant → potencial data leak em Realtime. Risco inaceitável.
- **Alternativa rejeitada**: Permissive (emitir JWT parcial) — incompatível com requisito de Realtime.

**Sobrescrita de Role — Sempre (DECISÃO: 2026-04-14)**

O merge JSONB (`||`) sempre sobrescreve `tenant_id` e `role` com valores do banco:

```sql
COALESCE(claims -> 'app_metadata', '{}'::jsonb) ||
  jsonb_build_object('tenant_id', user_tenant_id, 'role', user_role)
```

- DB é sempre source of truth — mudanças de role refletem no próximo login do usuário
- **Justificativa**: Consistência e segurança — usuário não pode manter role desatualizado via JWT cacheado.
- **Alternativa rejeitada**: Só injetar se vazio — permitiria divergência entre DB e JWT.

#### **Componente 3: Migration de 28 RLS Policies**
- **Arquivo**: `supabase/migrations/20260414_fix_rls_use_app_metadata.sql`
- **Padrão**:
  ```sql
  -- ANTES:
  tenant_id = (auth.jwt() -> 'user_metadata' ->> 'tenant_id')::uuid
  
  -- DEPOIS:
  tenant_id = ((SELECT auth.jwt()) -> 'app_metadata' ->> 'tenant_id')::uuid
  ```
- **Tabelas Afetadas** (28 policies):
  - `tenants` (3), `kanbans` (4), `contacts` (3), `conversations` (3), `automatic_messages` (3), `columns`, `messages`

### Script de Backfill
- **Arquivo**: `scripts/backfill-app-metadata.ts`
- **O que**: Lê usuários da tabela `users`, injeta `app_metadata` via admin API
- **Quando**: PASSO 2 (após migration do Hook)

### Sequência Crítica de Execução

```
PASSO 1: Aplicar create_custom_access_token_hook.sql
PASSO 2: Executar npx ts-node scripts/backfill-app-metadata.ts
PASSO 3: Ativar Hook no Dashboard (Authentication → Hooks)
PASSO 4: Testar login → verificar JWT com app_metadata.tenant_id
PASSO 5: Aplicar fix_rls_use_app_metadata.sql
PASSO 6: Testar RLS isolamento (JWT correto/incorreto)
PASSO 7: Atualizar register/route.ts (STEP 5.1)
PASSO 8: Executar *security-audit full
```

### Testes SQL para Validação

```sql
-- ✅ Correto tenant_id:
SET LOCAL request.jwt.claims = '{"app_metadata": {"tenant_id": "TENANT-UUID"}}';
SELECT count(*) FROM public.kanbans;

-- ❌ Errado tenant_id:
SET LOCAL request.jwt.claims = '{"app_metadata": {"tenant_id": "00000000-0000-0000-0000-000000000000"}}';
SELECT count(*) FROM public.kanbans;

-- ❌ user_metadata (não deve funcionar):
SET LOCAL request.jwt.claims = '{"user_metadata": {"tenant_id": "TENANT-UUID"}}';
SELECT count(*) FROM public.kanbans;
```

- Location: `tests/rls.test.ts`
- Framework: Jest
- Strategy: Integration tests contra PostgreSQL real (RLS não funciona em mocks)

**Test Cases:**
1. Correct tenant_id access → returns data
2. Wrong tenant_id access → returns 0 rows
3. user_metadata access → returns 0 rows (ignora)
4. Multiple tenants isolation
5. New user creation with app_metadata injection
6. Backfill script execution
7. JWT decode contains app_metadata
8. **Hook failure (Strict)**: Simular banco indisponível → login deve falhar com erro, NÃO emitir JWT sem `tenant_id`
