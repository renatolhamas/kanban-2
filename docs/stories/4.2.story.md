# Story 4.2: Home Page Kanban Board

## Story Statement

**As a** owner,  
**I want to** see my kanban board on the home page with all conversations organized by column,  
**so that** I can visualize my workflow and manage conversations efficiently.

---

## Acceptance Criteria

1. [ ] Home page (`/`) displays selected kanban with all columns rendered (Novo, Em Andamento, Resolvido, etc.)
2. [ ] Conversations are queried from `conversations` table filtered by `kanban_id` and `tenant_id` (RLS isolation)
3. [ ] Each conversation card displays:
   - Contact name (if exists in DB) OR phone number (if no contact created yet)
   - WhatsApp phone number (always shown)
   - Last message preview (max 80 chars, truncated with ellipsis)
   - Timestamp of last message (relative or absolute format)
4. [ ] Conversations are grouped by their `column_id` and rendered in correct column position
5. [ ] Kanban selector dropdown allows switching between tenant's kanbans
6. [ ] Empty state displayed when no conversations exist
7. [ ] WCAG AA accessibility: semantic HTML, ARIA labels, keyboard navigation, focus management
8. [ ] No regressions in existing functionality (Epic 3 webhooks still work)
9. [ ] RLS policies prevent cross-tenant access

## Dev Notes

### EVO GO Integration Flow
[Source: docs/evogo/github-evo-go/docs/wiki/recursos-avancados/events-system.md (2026-04-14)]

**Data Flow for Story 4.2:**
1. **Webhook Event** (Story 3.3): EVO GO sends `MESSAGE` event to `/api/webhooks/messages`
   - Event name: `MESSAGE` (not `messages.upsert` — that's Supabase internal)
   - Payload includes: contact JID, message text, timestamp
2. **DB Write** (Story 5.1 will implement): Webhook handler:
   - Extracts `remoteJid` (e.g., `5511999999999@s.whatsapp.net`)
   - Parses phone number from JID
   - Finds or creates `contacts` record
   - Finds or creates `conversations` record (linked to contact + kanban + column)
   - Inserts `messages` record
   - Sets `conversations.last_message_at` to current timestamp
3. **Real-time Display** (Story 4.2 - THIS): Frontend listens to Supabase real-time for:
   - New conversation INSERTs
   - Conversation UPDATEs (last_message_at, column_id changes)
   - Message INSERTs (for last_message_content update)
   - Renders on kanban board

**EVO GO Event Configuration:**
- **Base URL:** `https://evogo.renatop.com.br` (external VPS)
- **Docs:** https://docs.evolutionfoundation.com.br/evolution-go
- **Webhook Setup** (per instance):
  ```bash
  POST /instance/connect
  {
    "webhookUrl": "https://our-domain.com/api/webhooks/messages",
    "subscribe": ["MESSAGE"],  # Category to subscribe to
    "apikey": "{instance-token}"
  }
  ```
- **Webhook Payload Structure:**
  ```json
  {
    "event": "MESSAGE",
    "instance": "vendas",
    "data": {
      "key": { 
        "remoteJid": "5511999999999@s.whatsapp.net",  // Phone@domain (individual or group)
        "fromMe": false,
        "id": "3EB0C5A277F7F9B6C599"                  // Unique message ID from EVO GO
      },
      "message": { 
        "conversation": "Olá, tudo bem?"              // Message text
      },
      "messageTimestamp": "1699700400",               // Unix timestamp
      "pushName": "João Silva"                        // Contact name from WhatsApp
    }
  }
  ```
- **Retry Behavior:**
  - Automatic retries: 5 attempts
  - Interval: 30 seconds between attempts
  - Your endpoint MUST return 200 OK quickly (don't process sync)
- **Event Categories Available:** MESSAGE, SEND_MESSAGE, READ_RECEIPT, PRESENCE, GROUP, CALL, CONNECTION, QRCODE, etc.

**Important: Webhook Idempotency**
- EVO GO retries failed webhooks (up to 5 times, 30s intervals)
- Same event can arrive multiple times
- Your webhook handler MUST be idempotent:
  - Use `key.id` (message ID) for deduplication
  - Check if message already exists before INSERT
  - UPSERTs are preferred over INSERTs

**For Story 4.2 (Display):**
- Conversations table is **populated by Story 5.1 webhook handler**, not queried directly from EVO GO API
- Story 4.2 reads from local `conversations` + `messages` tables (data synced via webhooks)
- NO direct EVO GO API calls needed for display — Supabase real-time listeners sufficient
- Webhook data flow: EVO GO → our `/api/webhooks/messages` → Story 5.1 handler → Supabase DB → Story 4.2 frontend

### Database Schema
[Source: docs/db/schema.md (2026-04-14)]

**Conversations table structure:**
```sql
CREATE TABLE conversations (
  id uuid PRIMARY KEY,
  tenant_id uuid NOT NULL,          -- FK → tenants (RLS filtering)
  contact_id uuid NOT NULL,          -- FK → contacts
  kanban_id uuid (SET NULL on delete),  -- FK → kanbans
  column_id uuid (SET NULL on delete),  -- FK → columns
  wa_phone text NOT NULL,            -- WhatsApp phone (5511999...)
  status text DEFAULT 'active',      -- 'active' or 'archived'
  last_message_at timestamptz,       -- For sorting by recency
  created_at, updated_at timestamptz
);
```

**Messages table structure:**
```sql
CREATE TABLE messages (
  id uuid PRIMARY KEY,
  conversation_id uuid NOT NULL,     -- FK → conversations CASCADE
  sender_type text,                  -- 'incoming' or 'outgoing'
  content text NOT NULL,             -- Message body
  media_url text,                    -- If media attachment
  media_type text,                   -- 'image', 'document', etc.
  created_at timestamptz
);
```

**Contacts table structure:**
```sql
CREATE TABLE contacts (
  id uuid PRIMARY KEY,
  tenant_id uuid NOT NULL,           -- FK → tenants CASCADE
  name text NOT NULL,                -- Contact display name
  phone text NOT NULL,               -- Phone number (unique per tenant)
  created_at, updated_at timestamptz
);
```

**Recommended query for Story 4.2:**
```sql
SELECT 
  c.id, c.wa_phone, c.status, c.last_message_at,
  co.name as contact_name,           -- Will be NULL if contact not yet created
  col.order_position,
  (SELECT content FROM messages WHERE conversation_id = c.id 
   ORDER BY created_at DESC LIMIT 1) as last_message_content,
  (SELECT created_at FROM messages WHERE conversation_id = c.id 
   ORDER BY created_at DESC LIMIT 1) as last_message_at_detail
FROM conversations c
LEFT JOIN contacts co ON c.contact_id = co.id
LEFT JOIN columns col ON c.column_id = col.id
WHERE c.kanban_id = $1 AND c.tenant_id = $2 AND c.status = 'active'
ORDER BY col.order_position, c.last_message_at DESC
LIMIT 50
```

**RLS Policy (Current ✅ SECURE):**
```sql
-- Pattern implemented in Story 4.1:
tenant_id = ((SELECT auth.jwt()) -> 'app_metadata' ->> 'tenant_id')::uuid
```

**⚠️ IMPORTANT: RLS Security Status**
- RLS fix completed in Story 4.1 using imuttable `app_metadata`.
- Frontend queries MUST rely on this claim for automatic filtering.

All RLS policies across conversations, contacts, kanbans use:
```sql
tenant_id = (auth.jwt() -> 'user_metadata' ->> 'tenant_id')::uuid
```

**Problem:**
- `user_metadata` is **editable by the user** (via auth API)
- Users can bypass tenant isolation by modifying their JWT claims
- This is flagged as **CRITICAL security risk** in Supabase Advisors

**Required Fix:**
Replace all `user_metadata` references with `app_metadata` (server-only, immutable):
```sql
-- All policies should use:
tenant_id = (auth.jwt() -> 'app_metadata' ->> 'tenant_id')::uuid
```

**Affected Tables:**
- `kanbans` (4 policies)
- `conversations` (4 policies)
- `contacts` (4 policies)
- `automatic_messages` (4 policies)
- `tenants` (3 policies)

**Who:** @data-engineer must update RLS policies before @ux-design-expert starts Story 4.2

**Estimated Effort:** 1-2 hours

### Issue 2: Missing Webhook Processing Columns

**Current State:**
Conversations table has basic structure, but lacks columns for robust webhook processing.

**Required Additions:**

```sql
-- Add to conversations table:
ALTER TABLE conversations ADD COLUMN IF NOT EXISTS
  evolution_message_id TEXT UNIQUE,  -- Maps to webhook key.id
  CONSTRAINT check_evolution_msg_id CHECK (evolution_message_id ~ '^\w+$');

-- Add to messages table:
ALTER TABLE messages ADD COLUMN IF NOT EXISTS
  evolution_message_id TEXT UNIQUE,  -- Maps to webhook key.id for idempotency
  sender_jid TEXT,                    -- Maps to webhook key.remoteJid (full JID with @s.whatsapp.net)
  CONSTRAINT check_sender_jid CHECK (sender_jid ~ '^\d+@s\.whatsapp\.net$' OR sender_jid ~ '^\d+@g\.us$');
```

**Why:**
- `evolution_message_id`: Prevents duplicate message processing if webhooks retry
- `sender_jid`: Stores full JID for EVO GO API calls; supports both individual (5511999...@s.whatsapp.net) and group (120363...@g.us) chats
- Idempotency is critical: Webhook retry policy is 5 attempts × 30s intervals

**Who:** @data-engineer applies migration

**Estimated Effort:** 30 minutes

### Issue 3: Index Strategy for Webhook Performance

**Current State:**

**Recommended Additions:**

```sql
CREATE INDEX IF NOT EXISTS idx_conversations_wa_phone_tenant 
  ON conversations(wa_phone, tenant_id);

-- Speed up deduplication checks
CREATE INDEX IF NOT EXISTS idx_messages_evolution_id 
  ON messages(evolution_message_id);

CREATE INDEX IF NOT EXISTS idx_conversations_evolution_id 
  ON conversations(evolution_message_id);
```

**Impact:** 
- Webhook handler can find/create conversations faster
- Deduplication checks execute in <1ms
- No blocking under 100+ msg/min ingest
