# Story 4.3: Drag-and-Drop Between Kanban Columns

> [!IMPORTANT]
> Esta story implementa drag-and-drop funcional entre colunas do kanban usando a **versão estável mais recente do dnd-kit (v6.x+)** [Recomendação @analyst Atlas].
> A história anterior (4.2 - Home Page Kanban Board) estabeleceu a visualização estática das colunas e conversas.
> Esta história adiciona interatividade: mover uma conversa entre colunas atualizando seu `column_id` no banco de dados.
>
> **Escopo definido:** Apenas drag-and-drop entre colunas (reordenação dentro de coluna será implementada em Story 4.4). [Atlas: Cards devem ser posicionados no TOPO da nova coluna].

## Story Statement

**As a** kanban owner,
**I want to** drag and drop conversation cards between columns,
**so that** I can quickly reorganize conversations without manually editing each one.

---

## Acceptance Criteria

1. [ ] Conversation cards in `<KanbanBoard>` are draggable between columns (using dnd-kit)
2. [ ] Dropped card updates `column_id` in database via generic API endpoint `PUT /api/conversations/update-column` [Atlas Recommendation: Re-use generic endpoint from 6.6]
3. [ ] **[River: Otimistic UI]** Card moves immediately on drop (optimistic) to the **TOP** of the new column [Atlas Recommendation], then persists to DB async
4. [ ] **[River: No validation restrictions]** Any conversation can be moved to any column without workflow restrictions (Backend validation from 6.6 applies)
5. [ ] Drop zones (columns) are clearly highlighted on drag (visual feedback: border highlight, background tint)
6. [ ] Dragging card has reduced opacity (**0.8**) and **pronounced box-shadow** [Atlas Recommendation for WCAG AA contrast] and cursor changes to `grab`
7. [ ] **[River: RLS Isolation]** Multi-tenant isolation enforced: only conversations within user's tenant can be moved
8. [ ] **[River: Acessibilidade]** Keyboard navigation support: Arrow keys (←/→) to move between columns, Enter to confirm drop (WCAG AA). Includes **Horizontal Auto-scroll** when dragging [Atlas Recommendation].
9. [ ] On successful drop: Card appears at the top of the new column (optimistic), toast notification "Moved to {columnName}" (optional: Atlas recommends only on error or if multi-kanban)
10. [ ] On drop error: Card reverts to original position + error toast "Failed to move conversation"
11. [ ] Real-time sync: When other users move conversations, current user's board updates via Supabase listener
12. [ ] No regressions: Existing functionality (4.2 Kanban display, 6.x Chat Modal) still works after drag-drop integration
13. [ ] WCAG AA compliance: Semantic HTML, ARIA labels for columns, keyboard focus management

---

## Scope

### In Scope (Included)

- **Drag-and-Drop Interaction**
  - dnd-kit latest stable integration (v6.x+) with `<DragDropContainer>` wrapper [Atlas Adjustment]
  - Drag sensor: mouse + touch (mobile support)
  - Drop target validation (any column accepts any card)
  - **Horizontal Auto-scroll** support during drag [Atlas Recommendation]

- **Database Mutation**
  - UPDATE `conversations` SET `column_id = ?` WHERE `id = ?`
  - Optimistic UI: move card immediately, persist async
  - Error recovery: revert on failed mutation

- **Visual Feedback**
  - **[River: Minimalista + Acessível]**
    - Dragging: `opacity: 0.8`, `box-shadow: xl`, `cursor: grab` [Atlas Recommendation for contrast]
    - Hovering column: `border: 2px solid primary` highlight
    - Drop zone active: `background: rgba(primary, 0.1)`
  - Keyboard feedback: focus ring on columns, ARIA live region announces movement

- **Real-time Updates**
  - Supabase listener (from 4.2) propagates `column_id` changes
  - All users see conversation moved in their boards (no refresh needed)

- **Multi-tenant Isolation**
  - RLS policies prevent moving conversations from other tenants
  - Column access restricted to kanban owner

### Out of Scope (Future Stories)

- **Reordering within columns** (Story 4.4) — only column-to-column movement here
- **Undo/Redo** — future UX polish
- **Drag animations** (easing, spring physics) — basic transition sufficient for MVP
- **Drag handles** — entire card is draggable
- **Drag preview customization** — default dnd-kit preview
- **Search/filter during drag** — no interaction with other features

Real-time subscription pattern (already in use from 4.2):
```typescript
const conversations = useConversations(kanbanId);

// Including column_id updates — so real-time sync is automatic
```

**Impact for 4.3:** The existing `useConversations` listener will automatically detect `column_id` changes, so no additional subscription code needed.

#### **Tech Stack — dnd-kit Selection**
[Source: docs/architecture/1-tech-stack-strategic-decisions.md#1.1]

Drag-and-drop library choice:
- **Selected:** dnd-kit v8.0.0 (not explicitly listed in tech-stack.md but referenced in PRD)
- **Why dnd-kit:** Lightweight, accessible (WCAG AA), modern hooks-based API, no jQuery dependency
- **Alternative (rejected):** React Beautiful DnD (not actively maintained), React DnD (older, more complex API)

#### **Accessibility Requirements**
[Source: docs/architecture/8-coding-standards-conventions.md (assumed from 4.2 pattern)]

Story 4.2 established WCAG AA as mandatory requirement. For drag-drop:
- Semantic HTML: `<button>`, proper roles
- ARIA labels: `aria-label`, `aria-live` for dynamic updates
- Keyboard navigation: Tab, Arrow keys, Enter/Space
- Focus visibility: `:focus-visible` with primary color ring

#### **Database Schema & RLS**
[Source: docs/db/schema.md + docs/db/rls.md (referenced in 4.2)]

Conversations table structure (relevant fields):
```sql
conversations (
  id uuid PRIMARY KEY,
  tenant_id uuid NOT NULL REFERENCES tenants(id),
  kanban_id uuid NOT NULL REFERENCES kanbans(id),
  column_id uuid NOT NULL REFERENCES kanban_columns(id),  -- [River: THIS field gets updated]
  contact_id uuid REFERENCES contacts(id),
  ...
)
```

RLS Policy (from 4.2, applicable here):
```sql
-- Conversations visible only to their tenant
SELECT * FROM conversations
  WHERE tenant_id = auth.jwt() -> 'app_metadata' ->> 'tenant_id'
```

**For Task 5:** This RLS policy automatically prevents moving conversations between tenants (UPDATE also checks tenant_id).
