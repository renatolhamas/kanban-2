# Story 4.4.1: Kanban Column Management

> [!IMPORTANT]
> Esta story implementa a gestão completa de **colunas** (Create, Edit, Delete, Reorder) dentro de um Kanban específico através de uma nova página de configuração (`/settings/kanbans/[id]`).
>
> **Contexto precedente:**
> - Story 4.1: Multi-tenant isolation via RLS ✅
> - Story 4.2: Home page displays selected kanban ✅
> - Story 4.3: Drag-and-drop between columns ✅
> - Story 4.4: Kanban CRUD in Settings ✅
>
> **Escopo definido:** CRUD completo para colunas (max 10 por kanban) + reordenação via setas (↑ ↓) + integridade de dados (mínimo 1 coluna por kanban). Reatribuição automática de conversas ao deletar coluna via **Application-Layer Orchestration** (não DB Triggers). **[Atlas Decision]** Conversas são movidas para a primeira coluna disponível (posição 0) para garantir resiliência.

## Story Statement

**As a** kanban owner,
**I want to** create, edit, delete, and reorder columns within my kanban boards,
**so that** I can customize my workflow and organize conversations exactly as I need.

## Risks & Mitigation

1. **⚠️ Conversation Reassignment Complexity** — Conversations must be automatically reassigned to first available column when a column is deleted
   - **Potential Issue:** Reassignment logic may fail or count is inaccurate
   - **Mitigation:** Implement via **Application-Layer Orchestration in POST handler** (Dex); integration tests verify reassignment works correctly; return count in response. **[Atlas Decision]** Conversations always move to first available column (min order_position).

2. **⚠️ RLS Enforcement & Data Isolation** — Cross-tenant visibility if RLS policies incorrectly configured
   - **Potential Issue:** User A could modify User B's columns if kanban_id validation misses edge cases
   - **Mitigation:** RLS policies enforced at DB level; integration tests verify query with wrong kanban_id returns 403; API validates kanban_id from auth context

3. **⚠️ Minimum Column Safeguard** — Cannot delete last remaining column in kanban
   - **Potential Issue:** Conversations without a column cause data integrity issues
   - **Mitigation:** DB constraint + API validation; button disabled when only 1 column; error message "Cannot delete last column"

4. **⚠️ Navigation Context Loss** — User navigates to column management page, then returns to Settings
   - **Potential Issue:** Tab selection lost or incorrect context
   - **Mitigation:** "Voltar" button links to `/settings?tab=kanbans` explicitly; test navigation flow

## Dev Notes

### Database Structure [Source: Story 4.4 Dev Notes + Plan F4.plan.023]

**Columns Table:**
```sql
CREATE TABLE columns (
  id uuid PRIMARY KEY DEFAULT uuid_generate_v4(),
  kanban_id uuid NOT NULL REFERENCES kanbans(id) CASCADE,
  name text NOT NULL,
  order_position integer NOT NULL DEFAULT 0,
  created_at timestamptz NOT NULL DEFAULT now(),
  updated_at timestamptz NOT NULL DEFAULT now()
);

-- Constraint: UNIQUE (kanban_id, name)
-- Constraint: order_position >= 0
```

**Conversations Table (updated):**
```sql
ALTER TABLE conversations ADD COLUMN column_id uuid REFERENCES columns(id) SET NULL;
-- When a column is deleted, conversations.column_id → first_available_column_id
```

**Key Points:**
- Every kanban has at least ONE column (enforced by constraint)
- `order_position` controls sort order in table (0, 1, 2, ..., gaps allowed)
- RLS policies via `kanban_id → tenant_id` prevent cross-tenant access
- Conversations FK to `column_id` is nullable; delete column → conversations REASSIGNED to first available column via **Application-Layer** (POST handler) (Dex)
- DB Schema and RLS policies verified as already existing in docs/db/ (Dex)

### API Route Patterns [Source: Plan F4.plan.023]

**GET /api/kanbans/[id]/columns**
```
├─ Validate: kanban belongs to user's tenant
├─ Fetch all columns for this kanban (sorted by order_position)
└─ Return: [{id, name, order_position, created_at}, ...]
```

**POST /api/columns**
```
├─ Validate: kanban exists and belongs to user's tenant
├─ Validate: name (1-100 chars), no duplicate names in kanban
├─ Validate: max 10 columns per kanban
├─ Create column in DB with next order_position
└─ Return: {id, name, order_position, created_at}
```

**PUT /api/columns/{id}**
```
├─ Validate: column exists and kanban belongs to user's tenant
├─ Validate: name (1-100 chars), no duplicate names in kanban
├─ Update column name
└─ Return: {id, name, updated_at}
```

**DELETE /api/columns/{id}**
```
├─ Validate: column exists and kanban belongs to user's tenant
├─ Validate: NOT the last remaining column (return 409 if is)
├─ Get first available column (min order_position) **[Atlas Decision]**
├─ UPDATE conversations: column_id → first_available_column_id (Application-Layer) **[Atlas Decision]**
├─ Delete column from DB
└─ Return: {success: true, conversations_moved: N}
```

**PATCH /api/columns/reorder**
```
├─ Validate: column exists and kanban belongs to user's tenant
├─ Update order_position (swap with adjacent column - Directional Swap) (Dex: Payload is {columnId, direction})
├─ Return: {success: true, columns: [{id, order_position}, ...]}
```

**Error Handling:**
- 400 Bad Request: Invalid input (name validation)
- 401 Unauthorized: Not authenticated
- 403 Forbidden: Trying to modify another user's kanban columns
- 404 Not Found: Column or kanban not found in user's tenant
- 409 Conflict: Trying to delete last column OR duplicate name
- 500 Server Error: Database/unexpected errors

### Technical Constraints & Considerations

1. **Order Position Management:** When reordering, gaps in `order_position` are fine (0, 1, 3, ...). Use simple swap logic: `order_a <> order_b`.

2. **Column Limit:** Max 10 columns per kanban (enforced via API validation + frontend warning).

3. **Minimum Column Safeguard:**
   - Cannot delete last remaining column in kanban
   - Button disabled when only 1 column
   - Error message: "Cannot delete last column. Every kanban must have at least one column."

4. **Conversation Reassignment (Application-Layer):**
   - Deleting a column with conversations triggers automatic reassignment to **first available column** (min order_position) **[Atlas Decision]**
   - Implemented in POST handler (not DB trigger)
   - Return count of moved conversations in response
   - [Decisão Dex]: Implementado sem triggers SQL, mantendo lógica na aplicação

5. **Navigation Context:**
   - "Voltar" button links to `/settings?tab=kanbans` explicitly
   - Preserve user's scroll position or selected kanban (optional enhancement)

6. **Accessibility (WCAG AA):**
   - Modal focus trap (Tab key cycles within modal, not outside)

   - Delete confirmation uses semantic `<dialog>` or alert role
   - Keyboard: Esc closes modals, Enter submits forms
