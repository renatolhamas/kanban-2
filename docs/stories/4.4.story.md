# Story 4.4: Kanban CRUD in Settings

> [!IMPORTANT]
> Esta story implementa a seção **Kanbans** na página de Settings, permitindo que usuários gerenciem seus boards kanban (criar, editar, deletar, reordenar).
>
> **Contexto precedente:**
> - Story 4.1: Multi-tenant isolation via RLS ✅
> - Story 4.2: Home page displays selected kanban ✅
> - Story 4.3: Drag-and-drop between columns ✅
>
> **Escopo definido:** CRUD completo para kanbans + reordenação via setas (↑ ↓). Colunas padrão criadas automaticamente ao criar novo kanban. [Recomendação arquitetura: aplicar padrão de `order_position` para reordenação]

## Story Statement

**As a** kanban owner,
**I want to** create, edit, delete, and reorder my kanban boards in Settings,
**so that** I can organize multiple workflow boards and customize my workspace.

## Acceptance Criteria

1. [ ] **Kanbans Section visible in Settings** — Settings page displays "Kanbans" tab with table listing all user's kanbans
2. [ ] **List all kanbans** — Table displays:
   - Kanban name
   - "Main" indicator (badge) if `is_main = true`
   - Created date (relative or absolute format)
   - Actions buttons: Edit, Delete, Move Up (↑), Move Down (↓)
3. [ ] **Create new kanban** — "Create Kanban" button opens modal with:
   - Kanban name input field (required, 1-100 chars)
   - "Create" button (disabled until name provided)
   - On submit: POST `/api/kanbans/create` with `{name, tenant_id}`
   - On success: toast "Kanban created" + refresh table + modal closes
   - Default columns created automatically (Novo, Em Andamento, Resolvido) via **Application Layer** (Dex) [Decisão Atlas: Padrão Epic 1.5]
4. [ ] **Edit existing kanban** — Edit button opens modal with:
   - Kanban name pre-filled
   - Cannot edit "Main" kanban name (disabled input OR readonly)
   - On submit: PUT `/api/kanbans/{id}` with `{name, tenant_id}`
   - On success: toast "Kanban updated" + refresh table + modal closes
5. [ ] **Delete kanban** — Delete button triggers confirmation dialog with:
   - "Are you sure? This action cannot be undone."
   - Safeguard: **Cannot delete "Main" kanban** (button disabled OR not shown for main)
   - On confirm: DELETE `/api/kanbans/{id}` with tenant_id validation
   - Cascade: Conversations linked to deleted kanban are automatically reassigned to the "Main" kanban via **Application Layer Orchestration** (Dex) [Decisão Atlas: Garantia de visibilidade de dados]
   - On success: toast "Kanban deleted" + refresh table
6. [ ] **Reorder kanbans** — Move Up/Down buttons update `order_position`:
   - Move Up (↑): Decrement `order_position` of current, increment previous
   - Move Down (↓): Increment `order_position` of current, decrement next
   - PATCH `/api/kanbans/reorder` with `{kanban_id, direction}` OR `{kanban_id, new_order_position}`
   - Optimistic UI: Cards move immediately, then persist
   - On error: Revert to original order + error toast
7. [ ] **Multi-tenant isolation (RLS)** — Only user's own kanbans visible; cannot edit/delete other users' kanbans
8. [ ] **Keyboard accessibility** — WCAG AA compliance:
   - Modal has trap focus (Tab cycles within modal)
   - Buttons have proper ARIA labels
   - Delete confirmation is keyboard-navigable
   - Focus management when modal opens/closes
9. [ ] **Loading states** — Buttons show loading spinner during API calls; disabled to prevent double-submit
10. [ ] **Error handling** — API errors display descriptive toast messages (e.g., "Failed to create kanban: Duplicate name")
11. [ ] **Real-time updates** — When other users modify kanbans (future multi-user story), current user's table refreshes via Supabase listener
12. [ ] **No regressions** — Existing functionality (4.2 home kanban display, kanban selector dropdown) still works
13. [ ] **Graceful degradation** — If reassign fails, show warning: "Conversations linked to this kanban were not successfully reassigned to Main. Please contact support." [Atlas]

## Risks & Mitigation

1. **⚠️ Cascade Delete Complexity** — Conversations must be automatically reassigned to Main kanban when a kanban is deleted
   - **Potential Issue:** Trigger logic may fail silently or reassign count is inaccurate
   - **Mitigation:** Implement via `BEFORE DELETE` trigger with `GET DIAGNOSTICS` for count validation; integration tests verify reassignment works correctly

2. **⚠️ RLS Enforcement & Data Isolation** — Cross-tenant visibility if RLS policies incorrectly configured
   - **Potential Issue:** User A could modify User B's kanban if tenant_id validation misses edge cases
   - **Mitigation:** RLS policies enforced at DB level; integration tests verify query with wrong tenant_id returns 403; API validates tenant_id from auth context

3. **⚠️ Real-time Subscription Failure** — If Supabase listener unavailable, current user won't see updates from other users
   - **Potential Issue:** Offline sync or network issues cause stale UI
   - **Mitigation:** Real-time subscription optional for MVP; manual refresh sufficient; graceful degradation shows warning if listener unavailable

## Dev Notes

### Database Structure [Source: docs/db/schema.md]

**Kanbans Table:**
```sql
CREATE TABLE kanbans (
  id uuid PRIMARY KEY DEFAULT uuid_generate_v4(),
  tenant_id uuid NOT NULL REFERENCES tenants(id) CASCADE,
  name text NOT NULL,
  is_main boolean NOT NULL DEFAULT false,
  order_position integer NOT NULL DEFAULT 0,
  created_at timestamptz NOT NULL DEFAULT now(),
  updated_at timestamptz NOT NULL DEFAULT now()
);

-- Constraint: UNIQUE (tenant_id, is_main) WHERE is_main = true
```

**Columns Table:**
```sql
CREATE TABLE columns (
  id uuid PRIMARY KEY DEFAULT uuid_generate_v4(),
  kanban_id uuid NOT NULL REFERENCES kanbans(id) CASCADE,
  name text NOT NULL,
  order_position integer NOT NULL DEFAULT 0,
  created_at timestamptz NOT NULL DEFAULT now(),
  updated_at timestamptz NOT NULL DEFAULT now()
);
```

**Key Points:**
- Every tenant has exactly ONE "Main" kanban (enforced by unique constraint with WHERE clause)
- New kanbans should have default columns created immediately via **Application Layer (POST handler)** (Dex) — [Decisão Atlas: Padrão Epic 1.5]
- `order_position` controls sort order in Settings table (0, 1, 2, ... or gaps allowed)
- RLS policies via `kanbans.tenant_id` prevent cross-tenant access
- Conversations FK to `kanban_id` is nullable; delete kanban → conversations are REASSIGNED to Main kanban [Decisão Atlas]

### API Route Patterns [Source: docs/architecture/5-backend-architecture-api-routes-middleware.md]

**Route structure example:**
```
POST /api/kanbans/create
├─ Validate: name (1-100 chars), tenant_id
├─ Create kanban in DB
├─ Create default columns (application-layer per epic 1.5)
└─ Return: {id, name, is_main, order_position, created_at}

PUT /api/kanbans/{id}
├─ Validate: kanban belongs to user's tenant
├─ Validate: NOT Main kanban (if name edit attempted)
├─ Update name
└─ Return: {id, name, updated_at}

DELETE /api/kanbans/{id}
├─ Validate: kanban belongs to user's tenant
├─ Validate: NOT Main kanban
├─ Check conversations count (for cascade warning)
├─ Delete kanban (cascade deletes columns, nullifies conversation FKs)
└─ Return: {success: true}

PATCH /api/kanbans/reorder
├─ Validate: kanban belongs to user's tenant
├─ Update order_position (swap with adjacent kanban)
├─ Return: {success: true, kanbans: [{id, order_position}, ...]}
```

**Error Handling [Source: architecture]**
- 400 Bad Request: Invalid input (name validation)
- 401 Unauthorized: Not authenticated
- 403 Forbidden: Trying to modify other user's kanban
- 404 Not Found: Kanban not found in user's tenant
- 409 Conflict: Trying to delete Main kanban OR duplicate name
- 500 Server Error: Database/unexpected errors

### Technical Constraints & Considerations

1. **Order Position Management:** When reordering, gaps in `order_position` are fine (0, 1, 3, ...). No need to compact. Use simple swap logic: `order_a <> order_b`.

2. **Default Columns:** When creating new kanban, auto-create 3 default columns via:
   - **Option B: Application-layer in POST handler [Decisão Dex: Implementado com order_position fixo 0, 1, 2 para garantir ordenação inicial]**
   - **Columns:** ["Novo", "Em Andamento", "Resolvido"] (Dex)

3. **Main Kanban Safeguards:**
   - Cannot delete (button disabled OR hidden for is_main=true kanban)
   - Cannot rename (input disabled OR readonly)
   - Always shows "Main" badge
   - Automatically created during tenant onboarding (Story 1.5)

5. **Cascade Delete Behavior (Atlas/Dex):**
   - Deleting kanban cascades to delete columns via Database FK (Dex).
   - Conversations linked to the deleted kanban are REASSIGNED to the tenant's "Main" kanban via **Application Layer Orchestration in the API Route** (Dex).
   - **Técnico Dex:** A API Route executará o `UPDATE conversations` antes do `DELETE kanban`, capturando o count de registros afetados para retornar na resposta.
   - [Decisão Atlas]: Mantém a integridade do workflow e evita conversas "órfãs" no sistema.

5. **Accessibility (WCAG AA):**
   - Modal focus trap (Tab key cycles within modal, not outside)
   - Buttons have `aria-label` attributes
   - Delete confirmation uses semantic `<dialog>` or alert role
   - Keyboard: Esc closes modals, Enter submits forms
