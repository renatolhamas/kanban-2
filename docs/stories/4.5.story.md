# Story 4.5: Contacts page CRUD

> [!IMPORTANT]
> Esta story implementa a página `/contacts` com CRUD completo para gerenciar contatos do tenant.
>
> **Contexto precedente:**
> - Story 4.1: Real Tenant Isolation via RLS ✅
> - Story 4.2: Home page displays kanban board ✅
> - Story 4.3: Drag-and-drop between columns ✅

>
> **Escopo definido:** Criar página `/contacts` com lista, create/edit/delete modals, validação básica de input, e RLS enforcement.

## Story Statement

**As a** tenant owner,
**I want to** create, view, edit, and delete contacts in a dedicated Contacts page,
**so that** I can maintain a centralized directory of phone numbers and their associated information for outbound communication management.

## Acceptance Criteria

1. [ ] **Contacts Page accessible from Sidebar** — Page `/contacts` displays with Sidebar navigation; "Contacts" tab visible and clickable from main layout
2. [ ] **Table displays all contacts** — DataTable shows all contacts for current tenant with columns:
   - Phone number (E.164 format or raw input)
   - Name
   - Created date (relative format, e.g., "2 days ago")
   - Actions (Edit, Delete buttons)
3. [ ] **Create Contact modal** — "Add Contact" button opens modal with:
   - Phone input field (required, accepts flexible input with automatic normalization to digits-only before persistence using a hybrid approach) **[Atlas Decision / Dex Decision: Hybrid normalization via `lib/utils/phone.ts`]**
   - Name input field (required, 1-100 chars)
   - "Create" button (disabled until both fields filled)
   - "Cancel" button closes modal
   - On submit: POST `/api/contacts` with `{phone, name, tenant_id}`
   - On success: Toast "Contact created" + table refreshes + modal closes
   - On error: Toast shows error message (e.g., "Duplicate phone number")
4. [ ] **Edit Contact modal** — Edit button on row opens modal with:
   - Phone pre-filled (Editable with uniqueness validation against other contacts in the tenant) **[Atlas Decision / Dex Decision: Frontend validation via Zod + React Hook Form]**
   - Name pre-filled (editable)
   - "Save" button submits PATCH `/api/contacts/{id}` with `{name, tenant_id}`
   - On success: Toast "Contact updated" + table refreshes + modal closes
5. [ ] **Delete Contact** — Delete button triggers confirmation dialog (Shadcn AlertDialog) with: **[Dex Decision: Use Shadcn AlertDialog for destruction confirmation]**
   - Message: "Are you sure? This contact will be permanently deleted."
   - Cancel / Confirm buttons
   - On confirm: DELETE `/api/contacts/{id}` with tenant_id validation
   - On success: Toast "Contact deleted" + table refreshes
   - Cascade: Any conversations linked to deleted contact are preserved; UI components must fallback to raw phone number display if contact record is absent **[Atlas Decision]**
6. [ ] **Multi-tenant isolation (RLS)** — Only user's own contacts visible; cannot edit/delete other users' contacts
7. [ ] **Pagination** — Table displays 10 contacts per page; pagination controls visible if >10 contacts; Implementation must be server-side to ensure scalability **[Dex Decision: Server-side pagination via API query params]**
   - **Pattern:** Anterior | Página X/Y | Próxima (Shadcn DataTable native controls) **[Atlas Decision]**
   - No pagination controls shown if ≤10 contacts total
8. [ ] **Keyboard accessibility** — WCAG AA compliance:
   - Modal has focus trap (Tab cycles within modal)
   - Form fields keyboard-navigable
   - Delete confirmation keyboard-navigable
   - Focus management when modal opens/closes
   - Buttons have proper ARIA labels
9. [ ] **Loading states** — Buttons show loading spinner during API calls; disabled to prevent double-submit
10. [ ] **Error handling** — API errors display descriptive toast messages (e.g., "Failed to create contact: Server error")
11. [ ] **No regressions** — Existing functionality (kanban board, settings, chat modal) still works
12. [ ] **Empty state handling** — When no contacts exist, table shows "No contacts found" message with "Create your first contact" call-to-action

## Dev Notes

### Database Structure [Source: docs/architecture/5-backend-architecture-api-routes-middleware.md]

**Contacts Table Schema:**
```sql
CREATE TABLE contacts (
  id uuid PRIMARY KEY DEFAULT uuid_generate_v4(),
  tenant_id uuid NOT NULL REFERENCES tenants(id) CASCADE,
  phone text NOT NULL,
  name text NOT NULL,
  created_at timestamptz NOT NULL DEFAULT now(),
  updated_at timestamptz NOT NULL DEFAULT now(),
  UNIQUE(tenant_id, phone)  -- Prevent duplicate phone per tenant
);

-- RLS Policy: All operations filtered by tenant_id
CREATE POLICY contacts_tenant_isolation ON contacts
  USING (tenant_id = auth.jwt() ->> 'tenant_id');
```

**Key Points:**
- Every contact has a `tenant_id` for RLS enforcement
- Phone is unique per tenant (UNIQUE constraint)
- Name is required, phone is required
- Created/updated timestamps tracked automatically
- No FK to conversations (contact deletion doesn't cascade)

### API Route Patterns [Source: docs/architecture/5-backend-architecture-api-routes-middleware.md]

**Contacts API Endpoints:**

```typescript
// GET /api/contacts
// Returns: { contacts: Contact[], pagination: { page, limit, total } }
// Query params: ?page=1&limit=10
// Middleware: auth, tenant-isolation

// POST /api/contacts
// Body: { phone: string, name: string }
// Returns: { id, tenant_id, phone, name, created_at }
// Validation: phone not empty, name 1-100 chars, unique phone per tenant
// Middleware: auth, tenant-isolation

// PATCH /api/contacts/[id]
// Body: { name: string }
// Returns: { id, phone, name, updated_at }
// Middleware: auth, tenant-isolation, RLS validates id belongs to tenant

// DELETE /api/contacts/[id]
// Returns: { success: true } or { error: string }
// Middleware: auth, tenant-isolation, RLS validates id belongs to tenant
```
