## Implementation Notes

### Recommended Approach

**Option A: Custom Validation (recommended for simplicity)**
```typescript
export function validateE164(phone: string, countryHint?: string) {
  const normalized = phone.replace(/[^\d+]/g, '').trim();
  
  if (!normalized.startsWith('+')) {
    return { isValid: false, errorMessage: 'Phone must start with country code (e.g., +55)' };
  }
  
  const digits = normalized.slice(1);
  if (digits.length < 10 || digits.length > 15) {
    return { isValid: false, errorMessage: 'Invalid phone length (10-15 digits)' };
  }
  
  const detectedCountry = detectCountry(normalized);
  return { isValid: true, normalized, detectedCountry };
}
```

**Option B: libphonenumber-js (recommended for robustness)**
```typescript
import { parsePhoneNumber, isValidPhoneNumber } from 'libphonenumber-js';

export function validateE164(phone: string) {
  try {
    if (isValidPhoneNumber(phone)) {
      const parsed = parsePhoneNumber(phone);
      return { isValid: true, normalized: parsed.format('E.164'), detectedCountry: parsed.country };
    }
    return { isValid: false, errorMessage: 'Invalid phone number' };
  } catch (e) {
    return { isValid: false, errorMessage: 'Invalid format' };
  }
}
```

**Recommendation:** Start with Option A (simpler, no extra dependency). If edge cases discovered, upgrade to Option B.

### Backward Compatibility

- **Existing contacts:** Can be edited without errors (no forced revalidation on load)
- **Old format phones:** If stored without `+` prefix, user must add when editing
- **Migration:** Not needed (validation only on create/edit, not retroactive)
