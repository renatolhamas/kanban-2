# Story 5.1.1: Corrigir Divergências Críticas Webhook Evo GO

> [!CRITICAL]
> Esta story resolve 3 divergências críticas identificadas no plano F5.plan.005.md que comprometem a confiabilidade do webhook Evo GO. Sem essas correções, o webhook pode falhar em ambientes de produção multi-tenant.

## Story Statement

**As a** system,
**I want to** eliminate critical divergences between Evolution GO API documentation and our implementation,
**so that** webhook handlers work reliably with the real API behavior and can survive production incidents.

---

## Scope

**IN:**
- DIV-12: Corrigir lookup de tenant via campo `instance` (nome vs UUID)
- DIV-3: Adicionar fallbacks robustos para campos `/instance/qr` (PascalCase vs camelCase)
- DIV-4: Adicionar fallbacks robustos para campos `/instance/status` (PascalCase vs camelCase)
- Implementar logs diagnósticos estruturados em todos os endpoints Evo GO
- Validar empiricamente qual formato a API real retorna
- Database migration para suportar nova estratégia de lookup

- Mudança no contrato de nenhum endpoint público
- Validação HMAC webhook (já existe em F5.plan.004)
- Outros tipos de divergência (DIV-2, DIV-5, DIV-6, DIV-7, DIV-8, DIV-11) — separate stories

## Dev Notes

### F5.plan.005 — Resumo das Divergências

**DIV-12 (CRÍTICA — Maior Risco):**
- **Problema**: Lookup busca UUID (`evolution_instance_id`), mas webhook envia nome (`instance: "kanban-instance-4b54f906"`)
- **Impacto**: Tenant nunca encontrado, webhook cai em fallback `?tenantId=` (funciona agora mas não escalável)
- **Solução**: Adicionar coluna `evolution_instance_name`, popular com `'kanban-instance-' || substring(id::text, 1, 8)`, buscar por essa coluna
- **Fonte**: F5.plan.005.md — DIV-12

**DIV-4 (CRÍTICA):**
- **Problema**: Documentação diz `/instance/status` retorna `connected`, `loggedIn` (camelCase), mas código prioriza `Connected`, `LoggedIn` (PascalCase)
- **Risco**: Se API real retornar camelCase apenas, código não fallback corretamente
- **Solução**: Adicionar fallback explícito para ambos; adicionar log estruturado da resposta bruta
- **Fonte**: F5.plan.005.md — DIV-4

**DIV-3 (CRÍTICA):**
- **Problema**: Documentação diz `/instance/qr` retorna `qrcode`, `code` (lowercase), mas código tenta `Qrcode`, `Code` (PascalCase)
- **Risco**: Mesmo que DIV-4 — se API real divergir, logs ajudam diagnosticar
- **Solução**: Adicionar fallback explícito; adicionar log estruturado
- **Fonte**: F5.plan.005.md — DIV-3

### Implementação Detalhada

**Task 0: Database Migration**

Arquivo: `supabase/migrations/20260422000000_add_evo_go_instance_name_lookup.sql`

```sql
-- Add column to tenants table
ALTER TABLE public.tenants 
ADD COLUMN IF NOT EXISTS evolution_instance_name TEXT UNIQUE;

-- Populate for existing tenants
UPDATE public.tenants 
SET evolution_instance_name = 'kanban-instance-' || substring(id::text, 1, 8)
WHERE evolution_instance_name IS NULL;

-- Add constraint (after population)
ALTER TABLE public.tenants
ADD CONSTRAINT tenants_evolution_instance_name_unique 
UNIQUE (evolution_instance_name);

-- Index for fast lookup
CREATE INDEX IF NOT EXISTS idx_tenants_evolution_instance_name 
ON public.tenants(evolution_instance_name);
```

**Task 1: Update Webhook Handler (app/api/webhooks/evo-go/route.ts)**

Change tenant lookup from:
```typescript
.eq('evolution_instance_id', instance)  // busca UUID, recebe nome
```

To:
```typescript
.eq('evolution_instance_name', instance)  // busca nome da instância
```

Add logging:
```typescript
console.log('[WebhookEvoGo] Processing event', {
  event: payload.event,
  instance: payload.instance,
  timestamp: new Date().toISOString(),
  hasData: !!payload.data,
});
```

**Task 2: Enhance evo-go-client.ts — Status Endpoint**

In `getEvoGoStatus()`:
- Try `statusData.Connected` OR `statusData.connected` (current code already does this with `??`)
- But make it explicit with logging:

```typescript
const statusResponse = await fetch(`${EVOGO_API_BASE}/instance/status`, {
  headers: { "apikey": instanceToken },
});

const json = await statusResponse.json();
console.log('[EvoGo] /instance/status raw response', {
  timestamp: new Date().toISOString(),
  endpoint: '/instance/status',
  status: statusResponse.status,
  dataKeys: Object.keys(json.data || {}),
  responseBody: JSON.stringify(json.data).substring(0, 500),
});

const statusData = json.data || {};
const connected = (statusData.Connected ?? statusData.connected ?? false);
const loggedIn = (statusData.LoggedIn ?? statusData.logged_in ?? false);
```

**Task 3: Enhance evo-go-client.ts — QR Endpoint**

In `getEvoGoQRCode()`:
- Same pattern as Task 2: log raw response, test both formats

```typescript
const qrResponse = await fetch(`${EVOGO_API_BASE}/instance/qr`, {
  headers: { "apikey": instanceToken },
});

const json = await qrResponse.json();
console.log('[EvoGo] /instance/qr raw response', {
  timestamp: new Date().toISOString(),
  endpoint: '/instance/qr',
  status: qrResponse.status,
  dataKeys: Object.keys(json.data || {}),
  hasQrcode: 'qrcode' in (json.data || {}),
  hasQrcodeCapital: 'Qrcode' in (json.data || {}),
  responseBody: JSON.stringify(json.data).substring(0, 500),
});

const qrData = json.data || {};
const qrCode = (qrData.Qrcode || qrData.qrcode || qrData.qr_code || '');
const code = (qrData.Code || qrData.code || '');
```
