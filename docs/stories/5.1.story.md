# Story 5.1: Webhook Contact Auto-Registration

> [!IMPORTANT]
> Esta story implementa a base reativa do sistema: a capacidade de reconhecer um novo contato assim que ele envia uma mensagem.

## Executor Assignment

executor: "@dev"
quality_gate: "@architect"
quality_gate_tools: ["vitest", "lint"]

## Story Statement

**As a** system,
**I want to** automatically register new contacts from incoming webhooks,
**so that** I can track participants and associate messages correctly.

---

## Scope

**IN:**
- Auto-registration of new contacts via MESSAGES_UPSERT webhook event
- Tenant mapping via Evolution GO instance lookup with fallback to URL parameter
- Phone number normalization to E.164 format
- Multi-device suffix stripping (`:1`, `:2`, etc.)
- WhatsApp group detection (`@g.us` suffix)
- Smart sync: `wa_name` always updated, `name` preserved on updates (manual edits protected)
- Database upsert with idempotency guarantees
- UI group indicator badge on conversation card

**OUT:**
- Conversation creation (handled by separate story)
- Message ingestion workflow (separate story)
- Contact profile editing UI (separate story)
- Batch contact sync (separate story)

## Acceptance Criteria

1. [ ] Webhook endpoint `MESSAGES_UPSERT` extracts `pushName` and `remoteJid` from payload.
2. [ ] Phone number is parsed from JID and normalized to E.164 (adding `+`).
3. [ ] System checks if contact exists for `tenant_id` + `wa_phone`.
4. [ ] If not exists, a new `contacts` record is created.
5. [ ] `pushName` is stored in `wa_name` (always updated).
6. [ ] `name` is initialized with `pushName` on first registration but NOT overwritten on subsequent updates (preserves manual edits).
7. [ ] Phone number strips multi-device suffixes (e.g., `:1`, `:2`) before normalization.
8. [ ] Groups (`@g.us`) are identified and registered with `is_group = true`.
9. [ ] System resolves `tenant_id` via `payload.instance` lookup in tenants table (database lookup) with `?tenantId=` URL param as fallback.
10. [ ] Idempotency is ensured using database constraints and safe upsert logic (ON CONFLICT via RPC function).

---

## Risks & Mitigations

| Risk | Severity | Mitigation |
|------|----------|-----------|
| Tenant lookup fails (instance not found in DB) | HIGH | Fallback to `?tenantId=` URL param with warning log. If both absent, log error and return safely (webhook always 200). |
| High volume webhook upserts stress database | MEDIUM | RPC function with prepared indexes on (phone, tenant_id). Connection pooling via Supabase. |
| Name field overwritten on contact update (AC 6) | HIGH | ON CONFLICT logic explicitly preserves `name` field. Only `wa_name` and `is_group` are updated. |
| Group vs. individual contact confusion | MEDIUM | `is_group` column + UI badge (`👥 Grupo`) clearly differentiates. AC 8 validates detection. |
| Phone normalization edge cases | MEDIUM | Comprehensive contract tests in `parseJid` and `normalizePhone` with all table cases. |
| Webhook payload validation missing | MEDIUM | Extract and validate required fields (`instance`, `remoteJid`, `pushName`). Log malformed payloads. |

### Payload Structure (CRÍTICO 4)

O webhook do Evolution GO envia mensagens com esta estrutura:

```typescript
interface EvoGoMessagePayload {
  event:    string;    // "MESSAGE"
  instance: string;    // nome da instância (ex: "kanban-instance-a1b2c3d4") — usado para tenant lookup
  data: {
    key: {
      remoteJid: string;  // JID do remetente (ex: "5511999999999@s.whatsapp.net")
      fromMe:    boolean;
      id:        string;  // ID único da mensagem
    };
    pushName?:        string;  // Nome de exibição do contato
    messageTimestamp: string;
    message:          Record<string, unknown>;
  };
}
```

**Exemplo real:**
```json
{
  "event": "MESSAGE",
  "instance": "kanban-instance-a1b2c3d4",
  "data": {
    "key": {
      "remoteJid": "5511999999999@s.whatsapp.net",
      "fromMe": false,
      "id": "3EB0C5A277F7F9B6C599"
    },
    "pushName": "João Silva",
    "messageTimestamp": "1699999999",
    "message": { "conversation": "Olá!" }
  }
}
```

**Exemplo de JID de grupo:** `"remoteJid": "120363XXXXXXXXXX@g.us"`

### Tenant Mapping (CRÍTICO 1)

**Lógica de resolução:**
```
1. Ler payload.instance  →  ex: "kanban-instance-a1b2c3d4"
2. SELECT tenant_id FROM tenants WHERE evolution_instance_id = ?
3. Se encontrado → seguir normalmente
4. Se NÃO encontrado → LOG WARNING + tentar ?tenantId= como emergência
5. Se ambos ausentes → log error + return (webhook sempre 200)
```

**Nota:** A instância é gerada por `getOrCreateInstance()` em `lib/api/evo-go-client.ts:325`:
```typescript
const instanceName = `kanban-instance-${tenantId.substring(0, 8)}`;
```

O lookup vai sempre encontrar o tenant em condições normais. O fallback é um safety net para inconsistências de dados.

### Webhook Utils (CRÍTICO 3 + CRÍTICO 5)

Criar `lib/api/webhook-utils.ts` com as seguintes funções e contratos:

**Interface de retorno de `extractContactInfo`:**
```typescript
interface ContactInfo {
  waPhone:   string;   // E.164 normalizado
  name:      string;   // pushName ou waPhone como fallback
  waName:    string;   // pushName original
  isGroup:   boolean;  // true se remoteJid termina em "@g.us"
  remoteJid: string;   // original sem modificação
}
```

**`parseJid(jid: string): { phone: string; isGroup: boolean } | null`**

| Input | Output |
|-------|--------|
| `"5511987654321:1@s.whatsapp.net"` | `{ phone: "5511987654321", isGroup: false }` |
| `"5511987654321@s.whatsapp.net"` | `{ phone: "5511987654321", isGroup: false }` |
| `"120363XXXXXXXXXX@g.us"` | `{ phone: "120363XXXXXXXXXX", isGroup: true }` |
| `""` | `null` |
| `null` / `undefined` | `null` |

**Regras:**
1. Remover sufixo multi-device (tudo após `:` antes de `@`)
2. Remover domínio (tudo após `@`)
3. Se termina em `@g.us` → `isGroup = true`

**`normalizePhone(phone: string): string | null`**

| Input | Output |
|-------|--------|
| `"5511987654321"` | `"+5511987654321"` |
| `"+5511987654321"` | `"+5511987654321"` |
| `""` | `null` |
| `null` / `undefined` | `null` |

**Regras:**
1. Se começa com `+` → retornar como está
2. Senão → adicionar `+` como prefixo
3. String vazia ou null → retornar `null`

**Escopo:** JIDs do WhatsApp sempre incluem código de país. Este normalizador recebe
exclusivamente números extraídos de JIDs via `parseJid`.

### Database & Upsert Logic (CRÍTICO 2 + MÉDIO 6)

Tabela `contacts` precisa de duas colunas novas:
- `wa_name TEXT` — último pushName (sempre atualizado)
- `is_group BOOLEAN DEFAULT false` — true se grupo WhatsApp

**Função SQL de upsert parcial (Task 0):**
```sql
CREATE OR REPLACE FUNCTION upsert_contact(
  p_tenant_id uuid,
  p_phone text,
  p_name text,
  p_wa_name text,
  p_is_group boolean
) RETURNS uuid AS $$
DECLARE v_id uuid;
BEGIN
  INSERT INTO contacts (tenant_id, phone, name, wa_name, is_group, updated_at)
  VALUES (p_tenant_id, p_phone, p_name, p_wa_name, p_is_group, NOW())
  ON CONFLICT (phone, tenant_id)
  DO UPDATE SET 
    wa_name = EXCLUDED.wa_name,
    is_group = EXCLUDED.is_group,
    updated_at = NOW()
    -- name NOT updated: preserva edições manuais
  RETURNING id INTO v_id;
  RETURN v_id;
END;
$$ LANGUAGE plpgsql SECURITY DEFINER;
```

**TypeScript call (Task 2):**
```typescript
const { data: contactId, error } = await supabase
  .rpc('upsert_contact', {
    p_tenant_id: tenantId,
    p_phone:     contactInfo.waPhone,
    p_name:      contactInfo.name,       // INSERT apenas
    p_wa_name:   contactInfo.waName,     // sempre atualizado
    p_is_group:  contactInfo.isGroup,
  });
```

### Error Handling (MÉDIO 7)

**Padrão para Task 2:**
```typescript
if (upsertError) {
  console.error('[Webhook] MESSAGES_UPSERT contact upsert failed', {
    tenantId,
    waPhone: contactInfo.waPhone,
    error: upsertError.message,
    timestamp: new Date().toISOString(),
  });
  return; // não lançar; webhook sempre 200
}
```

Falha no upsert → log + return. Nunca criar conversa sem `contact_id`.
