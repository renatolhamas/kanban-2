## Story Statement

**As a** system,  
**I want to** complete the conversation routing with re-open support, unread count tracking, and group message routing,  
**so that** all edge cases are handled correctly and attendants always see accurate unread indicators.

---

## Scope

**IN:**
- Re-open archived/closed conversations when contact sends a new message
- Move re-opened conversation to Main kanban's first column
- Increment `unread_count` by 1 for every incoming MESSAGE event (idempotent via `evolution_message_id`)
- Route group messages to the correct conversation

**OUT:**
- Basic conversation creation (already done in Story 5.2)
- Contact registration logic (Story 5.1)
- UI changes (separate scope)
- Database migrations (schema already exists)

## Dev Notes

### Contexto de Implementação

A função `handleMessagesUpsert` em [app/api/webhooks/evo-go/route.ts](../../app/api/webhooks/evo-go/route.ts) (linhas 376-432) já faz:

```typescript
// Verifica se conversa existe
const { data: existingConv } = await supabase
  .from('conversations')
  .select('id')
  .eq('tenant_id', tenantId)
  .eq('contact_id', contact.id)
  .eq('kanban_id', kanban.id)
  .maybeSingle();

if (existingConv) {
  // Apenas atualiza timestamps — FALTA: re-open se archived, unread_count++
  update(last_message_at, evolution_message_id)
} else {
  // Cria nova — FALTA: verificar group routing
  insert(...)
}
```

**O que precisa mudar:**

**AC 1 — Re-open:**
```
Incluir status na query: .select('id, status')
Se existingConv.status === 'archived': adicionar kanban_id, column_id, status='active' no UPDATE
```

**AC 2 — Unread count:**
```
Antes do UPDATE/INSERT: verificar se evolution_message_id já existe em conversations
Se não existe (novo): unread_count = unread_count + 1
```

**AC 3 — Group routing:**
```
Se contact.is_group === true: buscar conversa pelo is_group ao invés de criar individual
```
