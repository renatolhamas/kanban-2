# Story 5.2: Webhook Conversation Auto-Creation

> [!IMPORTANT]
> Esta story garante que os novos contatos sejam automaticamente roteados para o Kanban Principal, tornando-os visíveis na Home Page e reabrindo conversas arquivadas.

## Executor Assignment

executor: "@dev"
quality_gate: "@architect"
quality_gate_tools: ["vitest", "lint"]

## Story Statement

**As a** system,
**I want to** automatically create or re-open a conversation for new contacts,
**so that** they appear on the kanban board without manual intervention.

---

## Acceptance Criteria

1. [ ] System finds the "Main" kanban for the tenant (`is_main: true`).
2. [ ] System finds the first column of the Main kanban (`order_position` ASC).
3. [ ] If no active conversation exists for the contact in ANY kanban, a new record is created in the Main kanban's first column.
4. [ ] If a conversation exists but is `archived` or `closed`, it is set to `active` and moved back to the first column of the Main kanban.
5. [ ] If a conversation is already `active` (in any kanban/column), it remains where it is but is updated.
6. [ ] `unread_count` is incremented by 1 for every new message received.
7. [ ] `last_message_at` and `evolution_message_id` are updated.
8. [ ] Group messages are routed to the same conversation as the individual contact if applicable.

### Routing Logic
1. **Lookup Main Kanban:** Find `is_main: true` for the tenant.
2. **Lookup First Column:** Get the column with the lowest `order_position` in the Main Kanban.
3. **Existing Conversation Check:**
    - If `active`: Stay in current `kanban_id`/`column_id`.
    - If `archived`/`closed`: Move to Main Kanban -> First Column, set status to `active`.
    - If `null`: Create in Main Kanban -> First Column.
4. **Counters:** Always increment `unread_count`.

### Database Reference
- Tables: `kanbans`, `columns`, `conversations`
- New Column: `conversations.unread_count` (int, default 0).
- [Source: architecture/3-database-schema-complete-reference.md#conversations]

### Webhook Event Reference
**Evo GO Webhook Specification:**
- Event name: `MESSAGE` (not `MESSAGES_UPSERT`)
- Payload structure:
```json
{
  "event": "MESSAGE",
  "instance": "instance-name",
  "data": {
    "key": {
      "remoteJid": "5511999999999@s.whatsapp.net",
      "fromMe": false,
      "id": "3EB0C5A277F7F9B6C599"
    },
    "message": {
      "conversation": "Mensagem de texto"
    },
    "messageTimestamp": "1699999999",
    "pushName": "João Silva"
  }
}
```
- [Source: docs/evogo/github-evo-go/docs/wiki/recursos-avancados/events-system.md]
