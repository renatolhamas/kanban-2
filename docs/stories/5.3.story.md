# Story 5.3: Webhook → Persist Incoming Messages

## Executor Assignment

executor: "@dev"
quality_gate: "@architect"
quality_gate_tools: ["vitest", "lint"]

## Story Statement

**As a** system,  
**I want to** persist each incoming WhatsApp message into the `messages` table when a MESSAGE webhook event is received,  
**so that** the chat history is available for the Chat Modal (Story 6.1) and future features.

---

## Scope

**IN:**
- INSERT one record into `messages` for every incoming MESSAGE webhook event
- Extract content from `data.message.conversation` (text messages)
- Extract `sender_jid` from `data.key.remoteJid`
- Set `sender_type = 'contact'` (incoming messages are always from the contact)
- Use `evolution_message_id` for idempotency (`ON CONFLICT DO NOTHING`)
- Retrieve `conversation_id` from the conversation ensured in the same request

**OUT:**
- Media messages (image, video, audio, document) — content extraction only, media_url/media_type stay null for now
- Outgoing messages (`fromMe=true`) — not persisted in this story
- `unread_count` increment — tracked in Story 5.2.1
- UI changes — separate scope (Story 6.1)
- New database migrations — schema already has all required columns

## Acceptance Criteria

1. [x] Para cada evento MESSAGE recebido com `data.key.fromMe = false`: INSERT na tabela `messages` com `conversation_id`, `sender_type='contact'`, `content`, `evolution_message_id`, `sender_jid`
2. [x] Idempotência: se `evolution_message_id` já existe na tabela `messages`, o INSERT é ignorado silenciosamente (`ON CONFLICT DO NOTHING`) — nenhum erro, nenhuma duplicata
3. [x] Se `data.message.conversation` for nulo ou vazio: INSERT com `content=''` (string vazia) — não bloquear o fluxo
4. [x] Erros de INSERT em `messages` são logados mas **não** retornam erro para o webhook (webhook sempre responde 200)
5. [x] Mensagens com `fromMe=true` são ignoradas — não inseridas na tabela

---

## Risks

| Risk | Probability | Impact | Mitigation |
|------|-------------|--------|-----------|
| `data.message.conversation` nulo em mensagens de mídia | High | content NOT NULL viola constraint | Usar `data.message.conversation ?? ''` — string vazia é aceita |
| `evolution_message_id` nulo em alguns eventos | Low | idempotência falha, possível duplicata | Se nulo, ainda inserir — sem idempotência nesse caso (aceitar risco) |
| conversation_id não disponível se routing falhou | Low | FK violation | Só inserir se conversation_id existir — verificar antes do INSERT |

## Dev Notes

### Contexto de Implementação

A função `handleMessagesUpsert` em [app/api/webhooks/evo-go/route.ts](../../app/api/webhooks/evo-go/route.ts) já garante contact e conversation antes do bloco atual de log (linha 434). O INSERT em `messages` deve ser adicionado **imediatamente após** o bloco CREATE/UPDATE da conversa (após linha 432, antes do `console.log` de sucesso na linha 434).

### Extração de dados do payload MESSAGE

```typescript
// Payload disponível dentro de handleMessagesUpsert
const messageId = data.key?.id;           // evolution_message_id
const fromMe = data.key?.fromMe ?? false;  // true = outgoing, false = incoming
const senderJid = data.key?.remoteJid;    // ex: "5511999999999@s.whatsapp.net"
const content = data.message?.conversation ?? ''; // texto da mensagem
```

### Lógica de INSERT

```typescript
// Executar apenas para mensagens incoming
if (!fromMe && conversationId) {
  const { error: msgError } = await supabase
    .from('messages')
    .insert({
      conversation_id: conversationId,
      sender_type: 'contact',
      content: content,
      evolution_message_id: messageId ?? null,
      sender_jid: senderJid ?? null,
    })
    .select() // necessário para ON CONFLICT funcionar com upsert
    // Alternativa: usar .upsert() com onConflict: 'evolution_message_id', ignoreDuplicates: true

  if (msgError && msgError.code !== '23505') { // 23505 = unique_violation
    console.error('[Webhook] Message insert error', { ... });
  }
}
```

> **Nota:** Supabase JS não tem `ON CONFLICT DO NOTHING` nativo. Tratar `code === '23505'` (unique_violation) silenciosamente é equivalente.

### Ponto de inserção no código

```
linha 432: }  ← fim do bloco if(existingConv)/else
           ↓
           [INSERIR AQUI: INSERT em messages]
           ↓
linha 434: console.log('[Webhook] MESSAGES_UPSERT processed...')
```

### conversation_id

O `existingConv.id` ou o ID retornado pelo INSERT de conversa precisa ser capturado para uso no INSERT de messages. Ajustar o bloco existente para retornar o `id`:

- No UPDATE: `.select('id').single()` para obter `existingConv.id` (já está no `existingConv`)
- No INSERT: adicionar `.select('id').single()` para capturar o ID da conversa criada
