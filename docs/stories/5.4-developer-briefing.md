## Performance Baselines (Validated)

### Query Pattern
```sql
SELECT 
  c.id,
  c.contact_id,
  c.tenant_id,
  (SELECT content FROM messages 
   WHERE conversation_id = c.id 
   ORDER BY created_at DESC LIMIT 1) AS last_message,
  (SELECT created_at FROM messages 
   WHERE conversation_id = c.id 
   ORDER BY created_at DESC LIMIT 1) AS last_message_at,
  (SELECT sender_type FROM messages 
   WHERE conversation_id = c.id 
   ORDER BY created_at DESC LIMIT 1) AS last_sender_type
FROM conversations c
WHERE c.tenant_id = $1
ORDER BY last_message_at DESC NULLS LAST;
```

### 3. Real-Time Subscription (Supabase Realtime)

**Setup Code Pattern:**

```javascript
// Subscribe to messages table for current tenant
const subscription = supabase
  .channel('schema-db-changes')
  .on(
    'postgres_changes',
    {
      event: '*',
      schema: 'public',
      table: 'messages',
      filter: `conversation_id=eq.${conversationId}` // Optional: filter by conversation
    },
    (payload) => {
      // Update last_message preview in UI
      // RLS ensures user only sees own tenant's messages
      handleNewMessage(payload.new);
    }
  )
  .subscribe();
```

**Important Notes:**
- RLS policies automatically isolate by tenant — no additional filtering needed
- Subscribe to `messages` table, not `conversations`
- The new message event will include `conversation_id`, so you can map it back to the conversation card
- Consider debouncing if multiple messages arrive in quick succession

**Cleanup:**
```javascript
subscription.unsubscribe();
```

## Index Details (for reference)

**Index Name:** `idx_messages_conversation_created`  
**Definition:** 
```sql
CREATE INDEX idx_messages_conversation_created 
ON public.messages(conversation_id ASC, created_at DESC)
INCLUDE (sender_type, content);
```

**Why this index works:**
1. `conversation_id ASC` — filters messages to the specific conversation
2. `created_at DESC` — sorts in reverse chronological order
3. `INCLUDE (sender_type, content)` — adds these columns for index-only scans (pure index lookups, no table access)

**When it's used:**
- Correlated subquery pattern: `WHERE conversation_id = c.id ORDER BY created_at DESC LIMIT 1`
- The index is accessed backward (DESC order already in index)
- Result fetched in 1-2 index scans instead of full table scan

## Rollback Plan (Emergency Only)

If the index causes unexpected issues:

```sql
DROP INDEX public.idx_messages_conversation_created;
```

**Fallback Query (without index — slower but works):**
```sql
WITH last_messages AS (
  SELECT 
    conversation_id,
    content,
    created_at,
    sender_type,
    ROW_NUMBER() OVER (PARTITION BY conversation_id ORDER BY created_at DESC) AS rn
  FROM messages
)
SELECT 
  c.id,
  c.contact_id,
  c.contact_name,
  lm.content AS last_message,
  lm.created_at AS last_message_at,
  lm.sender_type AS last_sender_type
FROM conversations c
LEFT JOIN last_messages lm ON lm.conversation_id = c.id AND lm.rn = 1
WHERE c.tenant_id = $1
ORDER BY lm.created_at DESC NULLS LAST;
```
