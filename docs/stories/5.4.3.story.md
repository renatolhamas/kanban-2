## 🚨 CRITICAL CONTEXT — LOGIC CORRECTION

**⚠️ ATENÇÃO:**
O webhook atual está fragmentando conversas porque filtra pelo `kanban_id` na hora de decidir se cria uma nova conversa ou vincula a uma existente. Como o `kanban_id` é um atributo da conversa (que pode mudar ou ser específico de um quadro), o critério de vinculação deve ser apenas: "Este contato tem uma conversa **ATIVA** neste tenant?".

Referência: [F5.plan.013.md](../plans/fase.5/F5.plan.013.md)

---

## 📋 Story Summary

Correct the message-to-conversation binding logic in the Evolution GO webhook.
1.  **Modify Search:** Change the query that checks for an existing conversation to filter by `tenant_id`, `contact_id`, and `status = 'active'`.
2.  **Remove Kanban Filter:** Stop using `kanban_id` as a search criterion for binding incoming messages.

This ensures that multiple messages from the same contact are correctly grouped into a single active conversation, rather than creating a new card for every message.

---

## 🛠️ Technical Decisions (SSOT)

1.  **Binding Criterion:** The "active conversation" is the Single Source of Truth for message binding.
2.  **Status over Kanban:** We prioritize the `status` field to ensure we are talking to the current open thread with the customer.
3.  **Idempotency:** The contact upsert remains the first step to ensure the contact exists before binding.
4.  **Database Constraint:** A UNIQUE partial index MUST be created to ensure only one 'active' conversation exists per contact/tenant. (Confirmed: does NOT exist currently).

## ✅ Acceptance Criteria

### AC1: Correct Webhook Search Logic
```
GIVEN an incoming message from a contact via webhook
WHEN the system checks for an existing conversation
THEN it must filter by 'tenant_id', 'contact_id', AND 'status' = 'active'
AND it must NOT filter by 'kanban_id'
```

**Code Location:** `app/api/webhooks/evo-go/route.ts` (Lines 376-382)

---

### AC2: Conversation Grouping (Functional)
```
GIVEN a contact with an existing active conversation
WHEN 3 new messages arrive via webhook
THEN all 3 messages must be linked to the SAME existing conversation ID
AND no new records should be created in the 'conversations' table
```

---

### AC3: Dashboard Integrity
```
GIVEN the corrected binding logic
WHEN viewing the Kanban dashboard
THEN the contact's card should show the latest of the N messages as the preview
AND there should be only one card for that contact in the column
```

## 📝 Implementation Notes (Dev Notes)

### 🔴 CRITICAL GOTCHAS

#### **Gotcha #1: status vs status_id**
Ensure you are checking the `status` column in the `conversations` table. According to `F5.plan.013.md`, the value should be `'active'`.

#### **Gotcha #2: Multiple Active Conversations?**
The database currently does NOT have a unique constraint for active conversations per contact. This MUST be implemented to prevent race conditions in the webhook where multiple concurrent messages could trigger the creation of multiple 'active' cards for the same contact.
