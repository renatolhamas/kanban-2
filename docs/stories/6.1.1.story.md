## Story Statement

**As a** developer,
**I want to** consolidate utility files from the root `lib/` directory into `src/lib/`,
**so that** I can eliminate the non-determinism caused by TypeScript alias shadowing and reduce risk of silent failures in testing and production.

---

## Scope

**IN:**
- Identify low-risk utility files in root `lib/` that have no duplicates in `src/lib/`
- Move `lib/email.ts` to `src/lib/email.ts` (primary target)
- Move `lib/link-validation.ts` to `src/lib/link-validation.ts` (if exists)
- Move `lib/rate-limit.ts` to `src/lib/rate-limit.ts` (if exists)
- Update ALL imports across the codebase to use `@/lib/` alias
- Verify typecheck passes after all moves
- Verify no broken imports remain
- Delete migrated files from root `lib/`
- Document migration in Change Log

**OUT:**
- Moving files with duplicates (kanban.ts, utils.ts) — handled in Onda 3
- Migrating api/, auth, jwt, supabase/, middleware/ — handled in Onda 2
- Resolving conflicts between lib/ and src/lib/ versions — handled in Onda 3
- Updating vitest.config.ts aliases — may be required in Onda 2

## Dev Notes

### Auditoria Técnica (F6.PLAN.001)

A aplicação encontra-se em estado de "arquitetura de transição" com duas pastas de utilidades:
- **`lib/` (raiz)**: Contém lógica de infraestrutura com logs vitais e tratamento robusto de erros
- **`src/lib/`**: Contém helpers UI leves e padrão Shadcn UI

**Problema Principal:** O alias `@/*` no `tsconfig.json` cria não-determinismo. O comportamento da aplicação varia conforme o bundler (Webpack/Turbo) ou test runner (Vitest) resolve esses caminhos — especialmente entre Windows local e Linux (Vercel).

**Malefícios Identificados:**
- Fragilidade nos testes (Vitest falha com alias complexos)
- Inconsistência de padrões e observabilidade
- Risco de duplicação acidental
- Falsos positivos em QA (testando versão errada)
- Armadilha de auto-import (VS Code sugere `../../lib` em vez de `@/lib`)
- Drift de ambiente (Windows vs Linux)
- Possível bypass de RLS via cliente Supabase duplicado

[Source: F6.PLAN.001 — Relatório Técnico: Auditoria da Estrutura Híbrida]

### Padrões de Importação

**Alias Correto (deve-se usar):**
```typescript
import { sendEmail } from "@/lib/email";
```

**Padrão Anterior (será removido):**
```typescript
import { sendEmail } from "../../lib/email";
import { sendEmail } from "lib/email";
```

[Source: docs/architecture/8-coding-standards-conventions.md — Seção 8.3 "Imports Organization"]

Arquivos estão organizados em:
- `src/lib/` — Utilitários e helpers gerais
- `src/lib/supabase/` — Cliente Supabase (será migrado em Onda 2)

[Source: docs/architecture/2-project-structure-source-tree.md]

### Notas Importantes

- **Atomicidade:** Mover um arquivo por vez, validar com `npm run typecheck` após cada mover
- **Preservação de Histórico:** Usar `git mv` para que o Git entenda a mudança de local, não a deleção/recriação
- **Seleção de Arquivos:** Começar SOMENTE com arquivos sem duplicatas — não mexer em kanban.ts ou utils.ts
- **Próximas Ondas:** Onda 2 (Infraestrutura Crítica) cobrirá `api/`, `auth.ts`, `jwt.ts`, `supabase/`, `middleware/` (Risco Alto)
