# Story 6.2: Send Message Logic (UI → Evo GO)

## Executor Assignment

executor: "@dev"
quality_gate: "@architect"
quality_gate_tools: ["vitest", "lint", "typecheck"]

---

## Story Statement

**As an** attendant,
**I want to** click the send button in the chat modal and have my message delivered to the contact via WhatsApp,
**so that** I can have a functional two-way conversation with contacts.

---

## Scope

**IN:**
- Make the send button in `MessageInput` component fully functional.
- Capture message text from the text area and send via `POST /api/messages/send`.
- Integrate with Evolution GO API for actual message delivery.
- Save the sent message to the database (messages table).
- Display success/error feedback to user (toast notifications).
- Show loading state while message is being sent.
- Clear the input field after successful send.
- Handle network errors and API failures gracefully.
- Support text messages only (media upload deferred to Story 6.3).

**OUT:**
- Message status tracking (Sent/Delivered/Read) — Story 6.3.
- Media/file upload functionality — Story 6.3.
- Message editing or deletion — Future stories.
- Automatic message retries — Future implementation.

## Acceptance Criteria

1. The send button in `MessageInput` is disabled when the input is empty and enabled when there is text.
2. Pressing `Enter` captures the message and sends it. `Shift+Enter` is used for adding a new line without sending.
3. An API request is made based on the schema and endpoint verified in the [Local Swagger](https://evogo.renatop.com.br/swagger/index.html).
4. On successful send:
   - The message appears in the `MessageList` immediately (Optimistic UI with 50% opacity during transit).
   - The textarea is cleared and the button returns to disabled state.
   - A success toast notification (Shadcn) displays: "Mensagem enviada!".
   - The message opacity returns to 100% once confirmed.
5. On API error:
   - A Shadcn toast notification displays: "Erro ao enviar mensagem. Tente novamente."
   - The message remains in the textarea for user correction.
   - The optimistic message is removed from the list or marked with an error icon.
6. While the message is being sent, the button shows a loading state and remains disabled.
7. Messages sent by the user appear on the right side of the chat (Emerald 500 bubble).
8. The message includes a timestamp (HH:mm format).
9. Validation: Empty messages or whitespace-only messages are blocked (button remains disabled).
10. The implementation follows coding standards and uses the official instance Swagger as the primary reference.

### Architecture References

**API Integration (Swagger-First):**
- **Primary Source of Truth:** [Interactive Swagger - Instance Renatop](https://evogo.renatop.com.br/swagger/index.html)
- **Action:** Verify exact endpoint (e.g., `/message/sendText`) and payload schema in Swagger before implementation.
- **Middleware Protection:** All API routes require JWT authentication via `@/lib/middleware/auth`.
- **Tenant Isolation:** Messages are auto-scoped to current tenant via JWT claim.

**Database Context:**
- **Table:** `messages` (conversation_id, text, sender, service, created_at, status)
- **Service Filter:** All queries MUST include `.eq('service', 'ttcx')` for multi-tenant safety.
- **Sender:** Message sender is determined from JWT `sub` claim (current user ID).

**Error Handling & Validation:**
- **Validation:** Disable send button if input is empty.
- **Keyboard:** `Enter` to send, `Shift+Enter` for new line.
- **Feedback:** Use `shadcn toast` for all user notifications.
- **Network errors:** Display "Não conseguimos enviar a mensagem. Tente novamente." toast.
- **API errors (500, timeout):** Display "Erro no servidor. Tente novamente mais tarde." toast.

### Pre-Flight Technical Decisions (Added by Dex)

Following the pre-flight planning phase, the following technical directions were established:

- **Backend Architecture:** `app/api/messages/send/route.ts` will be created to handle message persistence in the database BEFORE calling the Evolution GO API.
- **Service Layer:** A dedicated service `src/lib/services/evolution.ts` will be implemented to encapsulate Evolution GO API logic.
- **Instance Management:** `evolution_instance_id` will be retrieved via database query using the tenant context from the session.
- **UI Feedback:** Shadcn UI Toast component will be installed and used for all status notifications.
- **Date Formatting:** Native `Intl.DateTimeFormat` will be used for HH:mm timestamps.
- **Optimistic UI:** Messages will track state using a `status` field (`'sending' | 'sent' | 'error'`) to provide granular visual feedback.
- **Interaction:** `Enter` will trigger send, while `Shift + Enter` will handle new lines. Input will NOT be cleared on failure to allow retries.

**Frontend Patterns:**
- **State Management:** Use ChatContext to manage sending state (`isSending: boolean`).
- **Loading UI:** Button disabled + spinner icon during send.
- **Optimistic UI:** Add message to `MessageList` immediately with **50% opacity** while `isSending` is true. Update to 100% opacity on success.

### Coding Standards to Follow

- **TypeScript:** Strict mode, no `any` types, explicit return types on all functions.
- **Imports:** External packages first, then `@/*` absolute imports, then relative imports.
- **Error Handling:** Wrapped in try-catch in API routes; use Error Boundary on frontend.
- **Component Naming:** PascalCase (`.tsx`); utility functions in camelCase (`.ts`).
- **API Route Pattern:** Middleware → logic → response. See example in architecture/5.
- **Validation:** Check empty messages before API call (UI-level validation).
