# Story 6.3: Message Delivery Validation (Status Tracking)

## Executor Assignment

executor: "@dev"
quality_gate: "@architect"

---

## Story Statement

**As an** attendant,
**I want to** see the delivery status of messages I send (Sent/Delivered/Read),
**so that** I know whether my message reached the contact and was read.

---

## Scope

**IN:**
- Track message delivery status across states: Enviado (Sent), Entregue (Delivered), Lido (Read) using a hybrid Webhook + Server-side Polling approach.
- Update message status in the database based on Evolution GO API responses (Webhooks/ACK).
- Display status indicators in the chat UI with visual cues (checkmark icons, styling variations).
- Handle real-time updates via Webhooks with a Server-side Background Polling fallback.
- Preserve status history for audit trails using the `status_updated_at` timestamp.
- Display error status when delivery fails with user-friendly messaging.

**OUT:**
- Message editing or deletion (deferred to future story).
- Automatic message retries (deferred to future story).
- Delivery confirmations via SMS (deferred to future story).
- Read receipts with timestamp details (future enhancement).

## Acceptance Criteria

1. The `messages.status` column in the database stores one of: `'sending' | 'sent' | 'delivered' | 'read' | 'error'`.
2. A `status_updated_at` column (timestamptz) tracks the exact time of the last status change.
3. When a message is first sent, its status is set to `'sent'` immediately (optimistic UI from 6.2 carries through).
4. When Evolution GO confirms delivery via webhook (ACK), the message status updates to `'delivered'` in the database and UI.
5. When Evolution GO confirms the message was read by the contact, the status updates to `'read'`.
6. Status indicators display in the `MessageList` component:
   - `'sent'`: Single checkmark (✓) in gray.
   - `'delivered'`: Double checkmark (✓✓) in gray.
   - `'read'`: Double checkmark (✓✓) in Emerald 500 (colored).
   - `'error'`: Warning icon (⚠️) in red.
6. If delivery fails, the status is set to `'error'` and the user sees a toast notification: "Falha na entrega. Tente novamente."
7. Status updates are reflected in the chat UI within 2 seconds of webhook confirmation.
8. Server-side Polling Fallback: If no webhook is received within 20 seconds, a background job queries the Evolution API to sync the status.
9. All status transitions update the `status_updated_at` column.

### Architecture References

**Delivery Status States:**
According to Evolution GO API, message states progress as:
1. **Sending** → Optimistic UI (50% opacity, from Story 6.2)
2. **Sent** → Message confirmed in Evolution queue
3. **Delivered** → Message delivered to WhatsApp (webhook confirmation)
4. **Read** → Contact has read the message (webhook confirmation)

**API Integration:**
- **Primary Source of Truth:** [Interactive Swagger - Instance Renatop](https://evogo.renatop.com.br/swagger/index.html)
- **Webhook Endpoint:** `POST /api/webhooks/evo-go` (update to handle `MESSAGES.UPDATE` / `ACK` events).
- **Polling Strategy:** Server-side background job triggered 20s after message send if status is still `'sent'`.
- **Status Query Endpoint:** Check Evolution API for message status retrieval.

**Database Context:**
- **Table:** `messages`
- **New Column:** `status_updated_at` (TIMESTAMP WITH TIME ZONE)
- **Service Filter:** All queries MUST include `.eq('service', 'ttcx')`.
- **Status Column:** Enum constraint: `'sending' | 'sent' | 'delivered' | 'read' | 'error'`

**Error Handling & Status Updates:**
- **Network timeout:** Set status to `'error'` after 3 retries or 30-second timeout.
- **Webhook failures:** Fallback to server-side polling (max 3 attempts).
- **UI Feedback:** Use shadcn toast for delivery failures only.
- **Timestamp Tracking:** Always update `status_updated_at`.

### Pre-Flight Architectural Decisions (Dex)

As established during the Pre-Flight phase, the following technical strategies will be implemented:

- **[Dex Decision] Webhook Event Handling**: The system will specifically listen for the **`MESSAGES.UPDATE`** event from Evolution GO. The logic will extract the Evolution ID and map the status to our internal enum.
- **[Dex Decision] Polling Fallback**: The background polling (AC 8) will be implemented using a **Supabase Edge Function** triggered by a Database Webhook (on `messages` insert). This function will wait 20s and verify if the status is still 'sent' before querying the Evo GO API.
- **[Dex Decision] Real-time Communication**: We will leverage **Supabase Realtime (Table Subscriptions)**. The frontend will observe changes to the `status` field in the `messages` table and update the UI accordingly.
- **[Dex Decision] Database Integrity**: The `status_updated_at` field will be managed by a **Postgres Trigger** (`handle_message_status_update`) that fires whenever the `status` column is modified.
- **[Dex Decision] Component Modularity**: A dedicated molecule component **`MessageStatus.tsx`** will be created to render the checkmark icons, keeping `MessageList.tsx` clean and maintainable.
- **[Dex Decision] Error Mapping**: API errors will be differentiated; permanent failures (e.g., invalid-jid) will mark the message as `'error'` immediately, while network issues will trigger the polling fallback.
