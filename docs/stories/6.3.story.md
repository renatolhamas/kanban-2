# Story 6.3: Message Delivery Validation (Status Tracking)

## Executor Assignment

executor: "@dev"
quality_gate: "@architect"
quality_gate_tools: ["vitest", "lint", "typecheck"]

---

## Story Statement

**As an** attendant,
**I want to** see the delivery status of messages I send (Sent/Delivered/Read),
**so that** I know whether my message reached the contact and was read.

---

## Scope

**IN:**
- Track message delivery status across three states: Enviado (Sent), Entregue (Delivered), Lido (Read).
- Update message status in the database based on Evolution GO API responses.
- Display status indicators in the chat UI with visual cues (checkmark icons, styling variations).
- Handle status polling or real-time updates (webhook from Evolution GO when available).
- Preserve status history for audit trails (optional: store state transition timestamps).
- Display error status when delivery fails with user-friendly messaging.

**OUT:**
- Message editing or deletion (deferred to future story).
- Automatic message retries (deferred to future story).
- Delivery confirmations via SMS (deferred to future story).
- Read receipts with timestamp details (future enhancement).

## Acceptance Criteria

1. The `messages.status` column in the database stores one of: `'sending' | 'sent' | 'delivered' | 'read' | 'error'`.
2. When a message is first sent, its status is set to `'sent'` immediately (optimistic UI from 6.2 carries through).
3. When Evolution GO confirms delivery via webhook or API response, the message status updates to `'delivered'` in the database and UI.
4. When Evolution GO confirms the message was read by the contact, the status updates to `'read'`.
5. Status indicators display in the `MessageList` component:
   - `'sent'`: Single checkmark (✓) in gray.
   - `'delivered'`: Double checkmark (✓✓) in gray.
   - `'read'`: Double checkmark (✓✓) in Emerald 500 (colored).
   - `'error'`: Warning icon (⚠️) in red.
6. If delivery fails, the status is set to `'error'` and the user sees a toast notification: "Falha na entrega. Tente novamente."
7. Status updates are reflected in the chat UI within 2 seconds of webhook/API confirmation (or polling interval).
8. All status transitions are logged in the database for audit purposes (timestamp included).

### Architecture References

**Delivery Status States:**
According to Evolution GO API, message states progress as:
1. **Sending** → Optimistic UI (50% opacity, from Story 6.2)
2. **Sent** → Message confirmed in Evolution queue
3. **Delivered** → Message delivered to WhatsApp (webhook confirmation)
4. **Read** → Contact has read the message (webhook confirmation)

**API Integration:**
- **Primary Source of Truth:** [Interactive Swagger - Instance Renatop](https://evogo.renatop.com.br/swagger/index.html)
- **Webhook Endpoint:** `POST /api/webhooks/messages` (from Story 3.3)
- **Polling Alternative:** Query Evolution API periodically (if webhooks unreliable)
- **Status Query Endpoint:** Check Evolution API for message status retrieval

**Database Context:**
- **Table:** `messages` (id, conversation_id, text, sender, service, created_at, status, updated_at)
- **Service Filter:** All queries MUST include `.eq('service', 'ttcx')` for multi-tenant safety.
- **Status Column:** Type `text` with enum constraint: `'sending' | 'sent' | 'delivered' | 'read' | 'error'`

**Error Handling & Status Updates:**
- **Network timeout:** Set status to `'error'` after 3 retries or 30-second timeout.
- **Webhook failures:** Fallback to polling every 10 seconds (max 5 attempts).
- **UI Feedback:** Use shadcn toast for delivery failures only (success is silent).
- **Timestamp Tracking:** Store `status_updated_at` for audit trail.
