# Story 6.4: Message History Pagination (Load on Demand)

## Executor Assignment

executor: "@dev"

## Story Statement

**As a** user,
**I want to** load message history on demand (scrolling up in chat),
**so that** I can view previous conversations without loading all messages at once.

---

## Scope

**IN:**
- Implement pagination for message history com blocos de **50 mensagens iniciais** e **50 por requisição**.
- [Dex - Ajuste Pre-flight] Implementar **Infinite Scroll** invisível disparado no topo da lista (IntersectionObserver), substituindo botão manual para melhor fluidez de UX.
- Update `MessageList` component to support pagination state management.
- [Dex - Ajuste Pre-flight] Consultar tabela de mensagens usando **Cursor-based pagination (Data)** (ex: `.lt('created_at', cursor)`) em vez de `OFFSET`, prevenindo shift/duplicação em tempo real.
- Maintain correct message ordering (oldest to newest).
- [Dex - Ajuste Pre-flight] Preservar a posição do scroll nativamente utilizando `useLayoutEffect` e cálculos de `scrollHeight` antes/depois da injeção no DOM.
- Handle empty history gracefully (no more messages to load).

**OUT:**
- Full-text search of message history (future story).
- Export message history (future story).
- Custom date range selection (future story).
- Message filtering by type (image/text/etc.) (future story).

## Acceptance Criteria

1. When the ChatModal opens, load the **latest 50 messages** (DESC by created_at).
2. [Dex - Ajuste Pre-flight] O carregamento de mensagens anteriores deve ser disparado automaticamente via **Infinite Scroll** ao usuário se aproximar do topo do histórico de mensagens.
3. [Dex - Ajuste Pre-flight] O fetch das próximas 50 mensagens deve usar a estratégia de cursor (baseado no `created_at` da mensagem mais antiga já exibida) em vez de `OFFSET`.
4. [Dex - Ajuste Pre-flight] A posição do scroll deve ser **mantida** (sem solavancos na tela) utilizando manipulação nativa via `useLayoutEffect` ao inserir as mensagens no topo do DOM.
5. O gatilho de scroll deve ser inativado permanentemente quando a requisição retornar menos de 50 mensagens (fim do histórico do contato).
6. All queries include `.eq('service', 'ttcx')` filter [Source: architecture/9-performance-strategy.md#92-database-optimization].
7. Query performance: **< 200ms** for paginated message fetches.
8. Component supports **at least 500 messages** in memory without performance degradation.

### Database Pagination

[Source: architecture/9-performance-strategy.md#92-database-optimization]

**Query Pattern:**
```typescript
// [Dex - Ajuste Pre-flight] Fetch paginated messages (Cursor-based)
let query = supabase
  .from('messages')
  .select('*')
  .eq('conversation_id', conversationId)
  .eq('service', 'ttcx')
  .order('created_at', { ascending: false })
  .limit(50);

// Use o created_at da mensagem mais antiga carregada como cursor
if (oldestMessageDate) {
  query = query.lt('created_at', oldestMessageDate);
}
const { data: messages, error } = await query;
```

**Performance:**
- [Dex - Ajuste Pre-flight] Utilize **Cursor-based pagination** (comparando o carimbo de tempo `created_at`) no lugar de OFFSET para evitar _index shifting_ (duplicações/saltos causados por novas inserções concorrentes via webhooks/real-time).
- Index on `(conversation_id, service, created_at)` should exist (added in Story 5.3)
- Target: **< 200ms latency** for queries

### Implementation Risks (Post-Preflight)

- **IntersectionObserver "Bouncing":** O elemento observador no topo da lista pode disparar múltiplos eventos contínuos enquanto a API está buscando dados, resultando em requisições concorrentes. **Mitigação:** Travar o disparo se `isLoadingMore` ou `!hasMore` forem verdadeiros.
- **Flakiness do useLayoutEffect:** Calcular o `scrollHeight` antes e depois pode ser traiçoeiro dependendo do momento exato de renderização do React, especialmente se as mensagens contiverem imagens sem altura fixa. **Mitigação:** Capturar o state pré-render usando `useLayoutEffect` perfeitamente alinhado à atualização do DOM, antes do browser realizar o _paint_.
- **Empate de Cursor (Timestamp Collision):** Na paginação baseada em data (`created_at`), se várias mensagens tiverem **exatamente** o mesmo milissegundo de criação (ex: migração em massa) e cruzarem a margem do cursor de limite (50 itens), podemos acabar perdendo mensagens empatadas. **Mitigação:** Monitorar se isso ocorre; se ocorrer, a query de paginação precisará ser aprimorada para usar `(created_at, id)` como cursor duplo em iterações futuras.
