## Story Statement

**As an** attendant,
**I want to** see real-time updates to messages and conversations without refreshing the page,
**so that** I can respond instantly and never miss an incoming message or conversation change.

---

## Scope

**IN:**
- Implement WebSocket subscriptions using Supabase Realtime for live message updates (incoming and outgoing).
- Update `MessageList` component to reflect new messages instantly.
- Update `ConversationCard` components to reflect conversation status changes (column moves, assignment changes).
- Update `KanbanBoard` to reflect real-time column status and conversation reordering.
- Maintain connection state and graceful reconnection on network failures.
- Handle subscription cleanup on component unmount.
- Display connection status indicator in the UI (connected/reconnecting/offline).

**OUT:**
- Message editing or deletion (deferred to future story).
- Typing indicators (deferred to future story).
- User presence awareness (deferred to future story).
- Message delivery receipts via Evolution webhooks (handled by Story 6.3).

## Risks

1. **High-Frequency Event Flooding** — Supabase Realtime pode disparar 15+ INSERT events/segundo durante broadcasts. Without debouncing, React re-renders causam UI jank (frame drops) e memory pressure.
   - **Mitigation:** Implement `useDebounce` hook (200ms interval) em ChatContext para batch updates (AC #10). Test com DevTools Performance tab (target 60fps = 16ms per frame).

2. **WebSocket Connection Drops** — Network interruptions (WiFi loss, mobile handoff, proxy timeout) podem desconectar channel. Without proper reconnection logic, app fica "frozen" aguardando updates.
   - **Mitigation:** Implement exponential backoff retry (AC #5: max 5 retries, 30s timeout). Display connection status indicator (AC #4: 🟢/🟡/🔴).

3. **Memory Leaks on Unmount** — Se Realtime subscriptions não são cleanup'd corretamente, listeners permanecem em memória após component unmount, causando gradual memory growth.
   - **Mitigation:** useEffect cleanup handlers (AC #6). Test com DevTools Memory Profiler during subscription lifecycle.

4. **Cross-Tab Synchronization Latency** — Múltiplos browser tabs recebem UPDATE events simultaneamente. Sem batching/debouncing, UX pode ser inconsistente entre tabs (one tab updates before others).
   - **Mitigation:** Shared Realtime channel model (Story 6.3 pattern reused). Explicit multi-tab sync test (AC #3 validation, Subtask 3.4).

5. **Supabase Quota/Rate Limits** — High-frequency Realtime events podem exceder Supabase subscription quota limits se projeto scale rápido (10k+ concurrent subscriptions).
   - **Mitigation:** Monitor Supabase dashboard usage. Optional polling fallback (Story 6.8) como backup se Realtime quota exhausted.

## Acceptance Criteria

1. **Message List Subscriptions**: When a new message arrives via webhook or is sent by the current user, the `MessageList` component updates instantly without requiring a page refresh.

2. **Conversation Status Updates**: When a conversation is moved to a different column (kanban status change), all subscribed clients see the update in real-time.

3. **Multiple Active Conversations**: Users with multiple browser tabs/windows open see updates to all conversations simultaneously via shared subscriptions.

4. **Connection State Display**: A connection status indicator shows user the current state:
   - 🟢 **Connected** — Real-time active, listening for updates.
   - 🟡 **Reconnecting** — Connection lost, attempting to reconnect.
   - 🔴 **Offline** — No connection; fallback to 15s polling and manual refresh button.

5. **Graceful Reconnection**: If the WebSocket connection drops, the system automatically attempts to reconnect (exponential backoff, max 5 retries over 30 seconds).

6. **Subscription Cleanup**: When user closes the chat modal or navigates away, all Realtime subscriptions are properly unsubscribed to prevent memory leaks.

7. **Message Status Updates**: Real-time updates to message status fields (`status`, `status_updated_at`) from Story 6.3 are reflected in the UI within 1 second of database change.

8. **Kanban Board Sync**: Real-time updates to conversation column assignments and reordering are reflected in the `KanbanBoard` component without full page reload.

9. **Error Handling (Partial)**: If Realtime subscription fails (permissions, quota, etc.), system logs error to console/debugger for troubleshooting. Graceful fallback (polling + manual refresh button) deferred to Story 6.8 (Loading + error states refinement).
   - **[Pax @po — Recommendation #3]** Simplified AC #9 scope: error logging only; fallback polling moved to Story 6.8 per MVP focus and explicit scope separation.

10. **Performance**: Real-time updates do not cause UI blocking or jank (jank = > 16ms render time). System handles high-frequency updates (> 10 messages/second) gracefully.
    - **[Pax @po — Recommendation #7]** Reworded AC #10 to focus on outcome (no jank) rather than technical implementation (debounce details moved to Dev Notes).

### Performance Optimization — Debouncing Strategy

**Problem:** Supabase Realtime can fire INSERT events at high frequency (e.g., 15+ messages/sec during broadcast). React re-renders on every state update → UI jank (frame drops).

**Solution:** Debounce message state updates at ChatContext/useMessages level.

**Implementation Guidance:**
- **Custom Hook [Dev @dev — TBD Resolution]:** Implement `src/hooks/useDebounce.ts` natively using `useEffect` and `setTimeout` instead of adding an external dependency.
- **Pattern:** 
  ```typescript
  const [messages, setMessages] = useState([]);
  const debouncedMessages = useDebounce(messages, 200); // 200ms
  
  // Subscribe to Realtime
  channel.on('INSERT', (payload) => {
    setMessages(prev => [...prev, payload.new]); // Triggers debounce
  });
  ```
- **Interval:** 200ms (balances responsiveness + batching)

**Test AC #10:** Open 2 chat windows. Send 15+ messages rapidly. Verify target window displays all messages without frame drops (use DevTools Performance tab: target 60fps = 16ms per frame).
- **[Pax @po — Recommendation #7]** Added debouncing strategy to Dev Notes (separated from AC which focuses on outcome)

### Coding Standards to Follow

- **TypeScript:** Strict mode, no `any` types, explicit return types on hook returns.
- **Imports:** External packages first, then `@/*` absolute imports, then relative imports.
- **React Hooks:** Use `useEffect` for subscription setup/cleanup; `useCallback` for handlers to prevent re-renders.
- **Error Boundaries:** Wrap Realtime logic in try-catch; log to console for debugging.
- **State Updates:** Ensure state updates are immutable (spreads, concat, filter).
- **Connection Status:** Store in Context or custom hook (e.g., `useRealtimeStatus()`).
- **Testing:** Mock Supabase Realtime channel API in Vitest; test subscription lifecycle.
