# Story 6.6.1: Multi-Kanban Column Selection — Global Selector in Chat Modal

executor: "@dev"

## Story Statement

**As an** attendant,
**I want to** move a conversation to any column in any available kanban directly from the ChatModal,
**so that** I can transfer conversations between different service flows (Main, Suporte, Arquivos, etc.) without leaving the chat interface.

---

- **Phase 1: Backend & Hooks** - Create `useKanbanStructure` hook to fetch all kanbans and their columns
- **Phase 2: API Enhancement** - Update `PUT /api/conversations/update-column` to handle kanban_id
- **Phase 3: Component Refactoring** - Refactor `KanbanTransferDropdown.tsx` with optgroup rendering
- **Real-time Sync** - Reflect changes instantly via WebSocket subscriptions

**OUT:**
- Kanban/Column management from chat (deferred to Story 7.3)
- Bulk transfer operations (deferred to future)

## Acceptance Criteria

### 1. Hook `useKanbanStructure` Implemented
- Located at `src/hooks/useKanbanStructure.ts`
- Fetches kanbans and columns from Supabase with order_position ordering
- Returns structured data with proper TypeScript types
- **UI Feedback:** Includes loading state (Disabled with Spinner) and robust error handling with Retry logic within the dropdown (Adjusted by @dev)

### 2. API Endpoint Enhanced
- Accepts `conversation_id` and `column_id`
- **Backend Resolution:** Back-end receives `column_id`, consults the `columns` table to discover the correct `kanban_id`, ensuring data integrity (Adjusted by @dev)
- Updates BOTH `column_id` AND `kanban_id` in conversation record in a single transaction (Adjusted by @dev)
- Validates: conversation, column, kanban, tenant access
- Returns success response with the **Full Conversation Object** updated, ensuring local state synchronization (Adjusted by @dev)
- Proper error responses: 400, 403, 404

### 3. UI Component Refactored
- Uses `useKanbanStructure()` hook with global tenant-wide fetch for better performance (Adjusted by @dev)
- Renders hierarchical structure: native `<optgroup>` for each kanban to ensure WCAG AA compliance (Adjusted by @dev)
- Option labels: "Kanban Name - Column Name"
- Current selection shown as selected
- Loading/error states maintained
- Retry logic functional

### 4. Real-time Synchronization
- **Optimistic UI:** Conversation card moves on board immediately after selection, before API confirmation (Adjusted by @dev)
- Conversation moves to new kanban and appears on board via Realtime events
- Other users/browsers see changes without refresh
- Conversation disappears from original kanban
- No duplicates across kanbans

### 5. Cross-Kanban Integrity
- Conversation exists in ONLY ONE kanban
- Moving removes from previous kanban
- Changes persisted: refresh shows correct kanban
- Updates are atomic (both fields or neither)

## Technical Context

### Data Models
[Source: Epic 1, Story 1.1]

**kanbans:** id, tenant_id, name, order_position, created_at, updated_at
**columns:** id, kanban_id (FK), name, order_position, created_at, updated_at
**conversations:** id, tenant_id, kanban_id (FK), column_id (FK), contact_id, created_at, updated_at

### Hook Query Pattern
```typescript
const { data } = await supabase
  .from('kanbans')
  .select('id, name, order_position, columns(id, name, order_position)')
  .order('order_position', { ascending: true })
  .order('order_position', { foreignTable: 'columns', ascending: true });
```

### API Validation Sequence (Adjusted by @dev)
1. Verify conversation exists and belongs to tenant
2. Verify `column_id` exists, belongs to tenant, and fetch its `kanban_id`
3. Verify `kanban_id` belongs to tenant
4. Update both fields in single transaction to ensure atomicity

### Component Pattern
```tsx
const kanbans = useKanbanStructure();
<select>
  {kanbans.map(kanban => (
    <optgroup key={kanban.id} label={kanban.name}>
      {kanban.columns.map(col => (
        <option key={col.id} value={col.id}>
          {kanban.name} - {col.name}
        </option>
      ))}
    </optgroup>
  ))}
</select>
```
