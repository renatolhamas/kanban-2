# Story 6.6: Kanban/Column Selector in Chat — Change Contact Status Within Chat

## Executor Assignment

executor: "@dev"
quality_gate: "@architect"
quality_gate_tools: ["vitest", "lint", "typecheck"]

## Story Statement

**As an** attendant,
**I want to** change a conversation's kanban column directly from the ChatModal,
**so that** I can update conversation status without switching to the kanban board view.

---

## Scope

**IN:**
- Implement a dropdown selector in `ChatModal` displaying available kanban columns
- **[PO Clarification - 2026-04-28]** Dropdown pattern: Smart multi-level selection supporting both current kanban columns and cross-kanban transfers (see Dev Notes - UX Pattern Recommendation)
- Column display format: `"Kanban Name - Column Name"` (e.g., "Main - Novo", "Main - Em Progresso")
- Update conversation's kanban column via PUT `/api/conversations/update-column`
- Display loading state (spinner) during API request
- Disable dropdown while request is in progress
- Display error toast on failure with retry option
- Real-time WebSocket sync: reflect column changes instantly without page refresh
- Validate that only columns from the current kanban are available
- **[PO Clarification - 2026-04-28]** Support moving conversation to different kanban (if user has access) OR limit to current kanban only per phase 1 (expansion to multi-kanban drag-drop deferred to Story 4.3)

**OUT:**
- Multi-kanban switching (deferred to future story)
- Reordering conversations within a column via drag-and-drop in chat (deferred to Story 4.3)
- Column creation/deletion from chat (deferred to Story 7.3)
- Bulk operations (change multiple conversations at once)

## Acceptance Criteria

1. **ChatModal exibe dropdown de colunas no Header [Atlas Recommendation]**
   - Dropdown localizado no Header do Modal (ao lado do nome do contato) para visibilidade imediata de status.
   - Dropdown lista todas as colunas do kanban atual da conversa.
   - Colunas exibidas no formato: `"Nome do Kanban - Nome da Coluna"`.
   - Coluna atual é exibida como selecionada por padrão.
   - Dropdown acessível via teclado (Arrow Up/Down, Enter).

2. **Selecionar coluna atualiza a conversa**
   - Ao clicar em nova coluna, envia PUT para `/api/conversations/update-column` com `{ conversation_id, column_id }`
   - Conversa move para coluna selecionada no kanban
   - `ConversationCard` move visualmente no `KanbanBoard` (ou reordena lista)
   - Seleção anterior é deselect automaticamente

3. **Loading + Error States (Server-First Strategy) [Atlas Recommendation]**
   - UI utiliza estratégia Server-First: exibe spinner e desabilita seletor até confirmação do backend para evitar inconsistências visuais.
   - Dropdown é desabilitado (disabled state) durante a requisição.
   - Erro exibe toast com mensagem legível: "Failed to move conversation. Retry?".
   - Toast inclui botão de retry que reexecuta a requisição.

4. **Real-time Sync via WebSocket**
   - Se outro usuário/navegador muda a coluna da conversa, modal atualiza instantaneamente sem refresh
   - Status reflete alterações via Supabase Realtime (construído em Story 6.5)
   - Conversas que saem da coluna atual desaparecem da lista visível

5. **Validação de Dados e Segurança [Atlas Recommendation]**
   - Backend valida rigorosamente se a `column_id` pertence ao `kanban_id` da conversa e ao `tenant_id` do usuário logado.
   - Apenas colunas do kanban atual são exibidas no dropdown (filtro via query).
   - Sistema impede mudança para coluna inexistente ou inválida.
   - Se coluna for deletada enquanto modal está aberto, dropdown reflete mudança via real-time.

### UX Pattern Recommendation [PO Analysis - 2026-04-28]

**Context Analysis:**
Analyzed patterns in similar systems (Trello, Asana, Monday.com, Jira) to recommend optimal UX for kanban/column selection in chat context.

**Pattern Recommendation: Smart Multi-Level Dropdown (Opção C)**

```
Move to:
├─ [CURRENT KANBAN - Main]          ← Group current kanban
│  ├─ ✓ Novo (current column)
│  ├─ Em Progresso
│  └─ Resolvido
├─ ─────────────────────
├─ [OTHER KANBANS]                  ← Expandable group for other kanbans
│  ├─ Backup - Novo
│  ├─ Backup - Em Progresso
│  └─ Arquivos - Concluído
```

**Rationale:**
- ✅ Visual separation between current kanban (primary action) and other kanbans (secondary)
- ✅ Single dropdown — avoids cramped two-dropdown layout in modal
- ✅ Scalable — handles N kanbans without UI collapse
- ✅ Familiar pattern — similar to Trello/Asana behavior
- ✅ Reduces user error — clearly shows "where am I going"

**Implementation Guidance:**
1. **Phase 1 (Story 6.6):** Implement for current kanban columns only
   - Dropdown shows only `[CURRENT KANBAN - Main]` group
   - Simpler implementation (MEDIUM complexity maintained)
   - Status: Ready for @dev

2. **Phase 2 (Story 4.3 - Drag-and-drop):** Add cross-kanban transfers via visual drag-drop on board
   - Story 4.3 handles movement when user is viewing kanban board
   - Story 6.6 handles quick-change when user is in chat
   - Both are complementary, not competing

**Complementary Operations:**
- **Story 6.6 (Chat):** Quick-change via dropdown (stateless, quick)
- **Story 4.3 (Board):** Visual movement via drag-drop (spatial, intuitive)

**[DEX RECOMMENDATION]** Implement the dropdown using a **Grouped List** pattern from Phase 1, even if only showing the current kanban, to align with the Smart Pattern.
**[DEX RECOMMENDATION]** Ensure the API endpoint `/api/conversations/update-column` is **Generic** to serve both the Chat Dropdown and the future Kanban Board drag-drop (Story 4.3).

**Note:** Multi-kanban transfer via dropdown in chat can be implemented later as expansion if users request it. Current design leaves room for growth without overcomplicating Phase 1.

### Architectural Pre-Flight Decisions [Dex - 2026-04-28]

#### Decision 1: Context Expansion
- **What:** Expand `ChatContext` state from a simple `activeConversationId: string` to a full `activeConversation: Conversation | null`.
- **Why:** To eliminate redundant data fetching in Header components (like the new Dropdown) and ensure UI consistency. The `ChatModal` needs immediate access to `column_id` and contact details without waiting for sub-component mounts.
- **Impact:** Simplifies integration for Stories 6.3/6.4 (Contact Details) and 6.6 (Status Change). Memory impact is negligible (single object storage).

#### Decision 2: Generic API Endpoint
- **What:** Create a centralized `PUT /api/conversations/update-column` endpoint instead of a chat-specific one.
- **Why:** The action of changing a conversation's column is functionally identical whether triggered from a Chat Dropdown (Story 6.6) or a Kanban Board Drag-drop (Story 4.3). A generic endpoint centralizes RBAC validation and integrity checks.
- **Scope:** Primarily Story 6.6 (current) and Story 4.3 (future). Prevents technical debt by avoiding duplicated logic for the same DB operation.
